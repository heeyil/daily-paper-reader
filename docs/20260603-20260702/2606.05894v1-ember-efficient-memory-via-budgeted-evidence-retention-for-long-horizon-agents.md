## 摘要
长时程智能体可以存档大量历史记录，但未来的答案检索仍会带来回读和上下文成本。当保留的记忆缺失与答案相关的证据时，系统必须返回到原始历史的大部分内容。我们研究预算化证据存续问题：在查询未知之前，应保留哪些源证据，以便在固定的保留源证据令牌预算下保持其可恢复和可用？我们将此场景实例化为预算化查询前留存，其中记忆在摄取时写入，随后在无需访问完整原始流的情况下读取。我们引入EMBER，一种学习型留存策略，构建紧凑、基于源的证据状态。EMBER存储证据胶囊：逐字的源摘录，配以检索键和更新元数据，同时保留接地性和读取时访问。查询后结果反馈训练写入器在摄取-检索-答案链中保留证据。在LongMemEval-RR（我们基于LongMemEval的保留证据协议）上，EMBER-14B在8192令牌保留证据比较点达到0.3017 F1，而最强的非EMBER预算化基线为0.1765。在保留源证据预算下，EMBER提升了F1、留存召回和读取召回，表明长时程记忆取决于在预算内保留证据，而非重新读取更大的历史记录。

## Abstract
Long-horizon agents can archive large histories, but future answers still incur retrieval, rereading, and context costs. When retained memory misses answer-relevant evidence, the system must return to larger portions of the raw history. We study budgeted evidence survival: before the query is known, which source evidence should be retained so that it remains recoverable and usable under a fixed retained source-evidence token budget? We instantiate this setting as Budgeted Pre-Query Retention, where memory is written during ingestion and later read without access to the full raw stream. We introduce EMBER, a learned retention policy that constructs a compact, source-backed evidence state. EMBER stores evidence capsules: verbatim source excerpts paired with retrieval keys and update metadata, preserving both grounding and read-time access. Post-query outcome feedback trains the writer to preserve evidence across the ingestion-retrieval-answer chain. On LongMemEval-RR, our LongMemEval-derived retained-evidence protocol, EMBER-14B reaches 0.3017 F1 at the 8192-token retained-evidence comparison point, compared with 0.1765 for the strongest non-EMBER budgeted baseline. Across retained source-evidence budgets, EMBER improves F1, Retain-Recall, and Read-Recall, indicating that long-horizon memory depends on retaining evidence within the budget rather than rereading larger histories.

## 1. 核心问题与整体含义（研究动机和背景）
- **问题**：长时程智能体（如对话助手、代码代理）需要处理大量历史信息（会话、文件、工具调用），但未来查询未知时，传统方法要么保留全部历史（消耗大量检索和上下文 token），要么使用通用的压缩或摘要，但可能丢失关键证据。当保留的记忆错过答案相关证据时，系统必须重读或检索更大范围的原始历史，导致高昂的推理成本。
- **核心动机**：在固定的保留源证据 token 预算 \(B_{\text{ret}}\) 下，**在查询未知之前**，如何智能地选择并保留那些未来查询最可能需要的源证据，使得在查询到达后能从中高效恢复并生成准确答案。
- **整体含义**：长时程记忆不应只是被动地存档或重读，而需要在写入阶段进行有远见的抉择。论文将这一问题形式化为 **预算化证据存续** 并实例化为 **预算化查询前留存** 协议。该方法使记忆质量变得可审计：可量化哪些证据在写入时保留、哪些在读取时丢失。

## 2. 方法论
### 核心思想
- **EMBER** 是一个学习型保留策略，构建**紧凑、基于源文本的证据状态**。它存储**证据胶囊**：逐字的源摘录（source excerpts）配以检索键（retrieval keys）和更新元数据。这样做既保留了与原始源文本的接地性，又使得在查询到达后可以通过检索键方便地找到。
- 写入阶段（未知查询）通过**答案性探查**（answerability probes）决定哪些源片段应保留、以何种形式保留；读取阶段（已知查询）从保留的证据胶囊中检索并选择证据用于回答。
- 训练目标使用**答案门控的证据链奖励**（Answer-Gated Evidence-Chain Reward）：最终答案质量作为门控信号，只有能支持正确答案的证据保留行为才获得正向奖励；同时引入预算惩罚，鼓励在预算内保留高价值证据。

### 关键技术细节
1. **证据胶囊结构**：
   - `title`, `entities`（实体锚点）, `retrieval_keys_surface`（表面检索键）, `retrieval_keys_intent`（意图检索键）, `focused_source`（源摘录）, `update_mode`（插入/合并/覆盖/跳过）。
   - 存储时只计数源摘录 token，检索元数据单独序列化不占用答案承载预算。

2. **写入流程**（算法 1 及附录算法 2）：
   - 在线摄取流式数据，维护活跃上下文和外部记忆状态。
   - 对于每个到达的数据块，写入器通过 schema 约束输出胶囊提议；预算会计层根据 \(B_{\text{ret}}\) 决定是否接纳、合并或拒绝。
   - 写入器在训练时通过 GRPO/PPO 更新，奖励来源于整个摄取→检索→答案链。

3. **读取流程**：
   - 查询到来后，生成 1-3 个短检索查询（query），从保留的证据胶囊中检索 top-k 候选。
   - 选择器从候选中选择最佳证据，答案生成器基于该证据回答。

4. **训练**：
   - 三阶段课程：Stage I（RULER-HotpotQA 受控多跳）、Stage II（MuSiQue + 2WikiMultiHopQA 多会话）、Stage III（更新、时间范围、放弃案例）。
   - 每种 episode 采样预算 \(B_{\text{ret}} \in \{512, 1024, 2048, 4096, 8192\}\)，使策略学会在不同预算下分配资源。
   - 奖励由最终答案质量 \(Q_i\) 门控辅助项（证据覆盖、排名分数、选择纯度、写入效用），并减去超出预算的惩罚。

## 6. 主要结论与发现
- 长时程记忆的效果取决于**预算内保留关键证据的能力**，而非简单扩大历史规模。EMBER 通过答案性探查和检索键，使保留的证据在未来查询时更容易被找回和使用。
- 在 8192 token 预算下，EMBER 的 F1 是最佳非 EMBER 基线的 1.7 倍；在更严格的 512 token 预算下，EMBERT-7B 的 F1（0.2413）甚至超过了所有基线在 8192 时表现。
- 证据流失分解：黄金证据主要损失发生在写入阶段（Retain-Recall 未饱和）和读取选择阶段，检索本身并非瓶颈。
- Oracle 保留显示当前上限约为 0.45 F1，表明仍有较大提升空间（尤其是读取阶段的选择偏差和响应聚合）。

## 7. 优点
- **问题形式化清晰**：首次将长时程记忆预算化为预算化查询前留存，并定义了分离写入/读取失败的指标（Retain-Recall、Read-Recall），使失败分析透明。

## 8. 不足与局限
- **基准覆盖面有限**：仅使用 LongMemEval-RR 和 RULER-HotpotQA 等受控/合成基准，未在真实部署的持久助手或代码代理上测试。用户真实历史的隐私、缓存、偏好漂移等情况未考虑。
- **证据标签粒度依赖**：Retain-Recall 需要 gold 证据标签，LongMemEval 的标签可能不完全准确；对于无显式标签的会话，使用启发式（标注会话中所有 turns），可能引入噪声。
- **检索和阅读器固定**：论文固定使用 BGE-small 检索器和 GPT-4o 阅读器，未探索不同检索/阅读组合下的泛化性。EMBER 的写入器训练依赖于这些固定组件，换用更强/更弱的检索可能改变最优策略。
- **训练与评估预算一致性**：训练时预算从 512~8192 采样，而主要评估固定 8192。实验证明了在不同预算下的优势，但未测试极低（如 128）或极高（如 16K）预算。
- **种子方差**：仅报告了 7B 模型的三次种子（F1 标准差 0.0049），14B 模型仅单次种子，可能影响统计稳健性。
- **记忆状态规模**：当前方法假设元数据不计入预算，但在极端大容量场景下元数据可能成为瓶颈。附录表 10 显示 EMBER-14B 元数据/源证据比约 18.5%，尚可接受，但未来部署需考虑。
- **未解决读取聚合失败**：尽管 Gold 证据被检索到，阅读器仍可能因推理错误或聚合不当而给出错误答案（如表 24 显示 22.7% 答案缺少选中的 gold 证据）。这是阅读器层面的限制，非 EMBER 本身能解决。
