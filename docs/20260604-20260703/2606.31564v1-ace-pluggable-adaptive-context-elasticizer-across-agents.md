## 摘要
智能体任务复杂性的增加导致轨迹长度快速增长，这给基于固定上下文窗口的大语言模型（LLM）的智能体带来了重大挑战。现有的上下文管理技术如截断和摘要存在固有的不灵活性和不可逆性：一旦信息被丢弃或压缩，即使在后来的决策步骤中变得至关重要，也无法恢复。为了解决这些限制，我们提出了自适应上下文弹性化器（ACE），这是一个即插即用的模块，在每个决策步骤中弹性地将历史步骤信息编排到智能体的上下文中。ACE维护一个无损消息维护层，存储每个历史步骤的原始消息和压缩抽象，同时上下文编排层根据当前任务状态在每个决策步骤自适应地为每个步骤分配弹性类型：原始、抽象或丢弃。这种可逆设计确保主LLM始终接收紧凑但信息丰富的上下文。我们将ACE适配到四种不同的智能体框架中，包括ReAct、DeepAgent、WebThinker和MiroFlow，无需训练或架构修改。实验表明，ACE始终优于截断和摘要基线，并在所有四种智能体框架中带来一致的性能提升。

## Abstract
The increasing complexity of agentic tasks has led to rapidly growing trajectory lengths, which poses significant challenges for large language model (LLM) based agents with fixed context windows. Existing context management techniques, such as truncation and summarization, suffer from inherent inflexibility and irreversibility: once information is discarded or compressed, it cannot be recovered even when it becomes critically relevant in later decision steps. To address these limitations, we propose the Adaptive Context Elasticizer (ACE), a plug-and-play module that elastically orchestrates historical step information into the agent's context at each decision step. ACE maintains a lossless message maintenance layer that stores both raw messages and compressed abstractions for each historical step, while a context orchestration layer adaptively assigns each step an elastic type as raw, abstract, or drop, at every decision step based on the current task state. This reversible design ensures that the main LLM always receives a compact yet information-rich context. We adapt ACE to four diverse agent frameworks, including ReAct, DeepAgent, WebThinker, and MiroFlow, without training or architectural modifications. Experiments show that ACE consistently outperforms truncation and summarization baselines, and brings consistent performance gains across all four agent frameworks.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：基于大语言模型（LLM）的智能体在复杂任务中会产生越来越长的轨迹（trajectory），每个历史步骤的信息被逐字追加到上下文，导致：
  - 信息密度下降
  - 注意力分散
  - 推理质量退化
- **现有方法缺陷**：
  - **截断（Truncation）**：丢弃最远历史，不可逆，可能丢失后续仍需要的关键信息。
  - **摘要（Summarization）**：将旧历史压缩为摘要，一经生成无法恢复细节，且摘要质量依赖压缩模型。
  - **其他方法**（阈值触发、递归压缩、智能体控制压缩）同样存在**不灵活**（无法根据任务状态动态调整）和**不可逆**（一旦丢弃或压缩，原始信息永久丢失）的共性缺陷。
- **整体含义**：需要一种可逆、弹性、自适应且无损的上下文管理机制，使得每个决策步骤主LLM接收的信息密度最大化，同时能够根据需要随时恢复之前被压缩或丢弃的内容。

## 2. 方法论：核心思想与关键技术细节

- **核心思想**：将**实际接收的上下文**与**原始消息**解耦，通过两层架构实现无损存储和弹性编排。

- **关键组件**：
  1. **消息维护层（Message Maintenance Layer）**：
     - 无损存储每个历史步骤的**原始消息**（`h_raw_i`）和**压缩抽象**（`h_abs_i`）。
     - 一旦记录，永不修改，保证完整性。
     - 压缩抽象由辅助LLM生成，保留关键事实（数字、URL、日期、ID、工具模式等）并明确失败信号。
  2. **上下文编排层（Context Orchestration Layer）**：
     - 在每个决策步骤`t`，根据当前任务状态动态为每个历史步骤`i`分配三种弹性类型之一：`Raw`（原始）、`Abs`（抽象）、`Drop`（丢弃）。
     - 实际上下文`C_t`由三部分拼接：任务描述`T`（始终保留）+ 历史步骤1到`t-2`按弹性类型选择 + 最新步骤`t-1`始终保留为**原始消息**（保证最新状态完整）。
     - 编排函数`ϕ`根据决策`d_t^i`输出对应形式的内容。
  3. **弹性化器（Elasticizer E）**：
     - 实例化为一个LLM（与主LLM相同），输入为任务描述`T`和所有历史步骤的压缩抽象`{h_abs_i}`。
     - 输出每个历史步骤的决策`d_t^i ∈ {Raw, Abs, Drop}`。
     - 输入长度受限于抽象总长度，避免长上下文问题。

- **流程示例**：
  - 每一轮智能体完成推理、工具调用、观察获取后，新的原始消息和对应的抽象存入消息维护层。
  - 下一轮开始前，弹性化器基于任务+历史抽象预测决策，编排层根据决策构造实际上下文，传递给主LLM进行新的推理。

- **可逆性与弹性**：
  - 由于消息维护层不变，同一历史步骤在不同决策步骤可以被分配不同的类型（例如：某步在步骤10被`Drop`，在步骤15被恢复为`Raw`）。
  - 所有决策都是**当前步最优**，且不会永久丢失信息。

- **即插即用集成**：
  - ACE作为外部包装器，包裹智能体的推理循环，无需修改原有动作空间、消息流或训练过程。
  - 适配了四种不同架构的智能体框架：ReAct、DeepAgent、WebThinker、MiroFlow。
