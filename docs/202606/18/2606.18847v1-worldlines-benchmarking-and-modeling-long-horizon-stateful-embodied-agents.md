## 摘要
为了在真实家庭环境中长期协助人类，具身智能体必须记住用户日常习惯、世界状态和过去的交互。现有的长期记忆基准主要评估以语言为中心的检索和问答，而具身基准通常侧重于短期任务执行，不测试动态环境中长期记忆的使用。我们引入了WorldLines，一个用于长时域具身家庭辅助的项目驱动基准。它构建了包含对话、动作、执行反馈、物体和设备状态变化的时域扩展家庭轨迹，并将其转化为带有证据链接的样本，用于记忆问答和具身任务规划。我们进一步提出了ObsMem，一个基于观察者的记忆框架，它维护可见性感知记忆和动作原生状态轨迹，以实现状态感知决策。实验揭示了在部分可观测性、被覆盖的世界状态以及将长期记忆转化为具身规划方面的持续挑战，而ObsMem为此场景提供了一个更强的参考架构。

## Abstract
To assist humans over extended periods in real homes, embodied agents must remember user routines, world states, and past interactions. Existing long-term memory benchmarks mainly evaluate language-centric retrieval and question answering, while embodied benchmarks often focus on short-horizon task execution without testing long-term memory use in dynamic environments. We introduce WorldLines, a project-driven benchmark for long-horizon embodied household assistance. It constructs temporally extended household traces with dialogues, actions, execution feedback, object and device state changes, and converts them into evidence-linked samples for Memory QA and Embodied Task Planning. We further propose ObsMem, an observer-grounded memory framework that maintains visibility-aware memories and action-native state trails for state-aware decisions. Experiments reveal persistent challenges in partial observability, overwritten world states, and translating long-term memory into embodied plans, while ObsMem offers a stronger reference architecture for this setting.

- **问题**：现有长期记忆基准（如 LoCoMo、LongMemEval、RealMem）主要评估语言中心化的检索与问答，忽略了物理世界状态的变化；而具身基准（如 ALFWorld、PARTNR、BEHAVIOR-1K）则聚焦于短时任务执行，世界状态不跨交互持久化。两者间缺失的是：在部分可观测、动态变化的家庭环境中，智能体能否维持长期世界状态，并用于后续问答和规划。
- **动机**：真实家庭服务请求（如“我7:30去健身房，8:30回来，之后想看电影并吃东西”）需要智能体连接当前指令与历史日程、偏好、环境状态。因此需要一个能够评估“长时域有状态具身智能体”的基准，测试其在对话、人类活动、机器人动作、设备变化中维持部分可观测世界状态的能力。

## 2. 方法论：核心思想、关键技术细节
### 2.1 WorldLines 基准构建
- **核心思想**：构建跨天的家庭轨迹，包含对话、人类活动、机器人动作、设备控制、执行反馈、世界状态变化，并转化为带有证据链的 Memory QA 和 Embodied Task Planning 样本。
- **关键技术**：
  - **World Grounding**：使用 Habitat/HSSD 场景，将交互式物体、容器、可控设备转换为语义视图（用于项目生成）和可执行场景视图（用于动作验证）。
  - **Project Planning**：定义多天生命线项目（如日常支持、餐食准备、家庭整理），指定参与者、空间、时间偏好和约束，生成持久活动线索。
  - **Closed-Loop Trace Simulation**：对每天生成上下文 C_d = (W, P_d, S_{d-1}, N_{<d})，由生成器提出对话/动作，确定性执行器检查实体引用、可行性后才写入状态变化。
  - **Carry-Forward Memory**：提取每日笔记（状态变化、偏好、未完成计划等），传递至后续天，使交互依赖历史。
- **Task Construction**：在轨迹中设置 cutoff，智能体只能看到 cutoff 前的历史。Memory QA 覆盖当前状态、被覆盖状态、多跳时间推理、偏好等；Embodied Task Planning 要求生成基于当前状态和历史可执行的动作序列。
- **样本形式**：x_i = (H_{<c_i}, q_i, S_{c_i}, E_i, y*_i, τ_i)，仅暴露 H_{<c_i} 和 q_i。

### 2.2 ObsMem：观察者接地记忆框架
- **核心思想**：解决“将所有信息压入单个文本记忆”会导致无法区分直接观察、间接报告、被覆盖状态、未来约束的问题。ObsMem 将交互流分类为不同类型，并分别存储和更新。
- **关键组件**：
  1. **观察者门控（Observer Gate）**：事件进入记忆前，检查机器人是否在观察者集合 V_t 中。若为观察事件，存储为 e^{obs}_t；若为机器人听的他人言论，额外创建报告原子 e^{rep}_t，明确区分“观察到”和“听说”。
  2. **类型化内存视图**：
     - **事件轨迹（Event Track）**：仅追加，保留历史证据。
     - **状态轨迹（State Track）**：维护当前快照 Ŝ_t(i,a) 和完整历史 H_t(i,a)，支持当前状态与历史上被覆盖状态的查询。
     - **信念轨迹（Belief Track）**：记录每个事实的置信度（fresh/stale/uncertain/contradicted），基于干预事件数、干预者、矛盾证据等更新。
     - **承诺轨迹（Commitment Track）**：保存未来义务（请求、提醒、承诺等）。
  3. **章节合并（Episode Consolidation）**：当检测到章节边界时，生成摘要卡片注入各视图，但保留原始事件。
- **查询时检索**：先解析查询意图，选择相应视图进行检索（状态/信念/事件/承诺等），再通过证据选择器（LLM-based）从候选集中选出互补的紧凑证据包，传递给答案生成器。避免单一相似度搜索。

## 6. 主要结论与发现
1. **ObsMem 在 Memory QA 上全面领先**：Judge 0.713 vs 最好基线 A-mem 0.575，Perfect Rate 69% vs 53%；Event R@5 0.537 vs Mem0 0.378。
2. **粗粒度片段检索不足**：基线方法 Session Any@5 较高，但 Event R@5 显著低，说明仅仅找到相关时间段并不能提供精确的状态变化证据。
3. **ObsMem 在状态多跳问题上表现突出**：State Multi-Hop 的 Event R@5 从 0.264 提升至 0.452，表明类型化状态轨迹有助于恢复精确证据链。
4. **消融实验显示证据选择器最关键**：移除后 Judge 从 0.699 降至 0.435；章节合并次关键；信念检索对隐藏观测问题至关重要（Hidden Judge 从 0.278 降至 0.000）。
5. **下游规划能力提升**：ObsMem 在规划任务上的 Plan Judge 0.684，远超其他方法，尤其在状态一致性和条件有效性上得分更高。

## 7. 优点
- **问题定位精准**：首次系统性地将“长期记忆”与“具身世界状态”结合，强调部分可观测性下的状态维护，而非简单文本记忆。
- **基准构建严谨**：
  - 项目驱动的 trace 生成，避免了孤立日常事件，体现长时间依赖。
  - 确定性执行器确保动作可行，carry-forward 笔记实现跨天连续性。
  - 证据链与 cutoff 的自动化构建防止信息泄漏，保证评估公平。
- **ObsMem 框架创新**：
  - 观察者门控区分直接与间接信息，符合机器人真实感知局限性。
  - 类型化记忆视图（事件/状态/信念/承诺）分别存储，查询时按需组合，提升精度。
  - 信念轨迹动态更新置信度，支持不确定性下的决策。
- **评估全面**：多种指标（Judge、Session Any@5、Event R@5）分别考察答案质量和检索精度；包含消融实验和 Judge 验证；同时涵盖 QA 和规划任务。

## 8. 不足与局限
- **仿真环境限制**：目前基于 Habitat/HSSD 模拟，未覆盖真实感知噪声、执行误差和开放的人类行为。证据链、cutoff、状态变化均基于精确标签，实际部署需感知模块提供实体ID、可见性等。
- **ObsMem 额外延迟**：结构化检索和信念感知选择带来更高延迟（平均 5.13s vs 基线 3-4s），需优化效率。
- **规划测试样本少**：仅 21 个规划样本，虽动作密集，但统计泛化性有限。
- **需依赖外部感知模块**：ObsMem 的观察者门控和状态追踪要求准确的感知、定位和语义理解，在现实世界中的鲁棒性未验证。
- **未涵盖视觉感知**：当前阶段使用结构化事件日志，未集成视觉观察，未来需扩展至多模态。
- **语言模型依赖**：QA 和规划均依赖 GPT/LLM 生成，可能存在对语言模型本身的依赖（如生成器偏好），虽然控制了上下文长度对比，但不同 LLM 内部差异可能影响结果。
