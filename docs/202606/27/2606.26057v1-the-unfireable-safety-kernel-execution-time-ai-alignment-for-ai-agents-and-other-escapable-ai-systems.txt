# The Unfireable Safety Kernel: Execution-Time AI Alignment for AI Agents and Other Escapable AI Systems 

## Seth Dobrin Łukasz Chmiel ARYA Labs PBC 

## seth@aryalabs.io 

Abstract 

AI agents are increasingly granted access to tools, APIs, and other consequential infrastruc-ture, making them continuously active principals in the systems they operate. The dominant approach to constraining such agents places safety controls inside the agent’s own runtime: sys-tem prompts, output filters, and guardrail libraries. We observe that any control residing in the agent’s address space is reachable by inputs that influence agent behavior, and that this property generalizes beyond agents to any AI system with sufficient reach into its own runtime, a class we term escapable AI systems .We identify four properties an authorization mechanism must satisfy to constitute archi-tectural control rather than cooperative request: process separation, pre-action enforcement on a structurally-only path, fail-closed at both request and system levels, and externalized signed evidence verifiable outside the controlled system’s trust boundary. We position this layer of constraint as execution-time AI alignment , complementing training-time alignment (RLHF, Constitutional AI) and inference-time alignment (prompts, filters, guardrails). We present the Unfireable Safety Kernel, a reference implementation in Rust realizing all four properties. The fail-closed invariant is machine-checked at two levels: an SMT theorem over the model (Z3) and an exhaustive bounded-model-checking proof of the production Rust decision function (Kani, 4/4 harnesses verified). The Python-to-Rust migration was gated on byte-equivalence over 1000 deterministic fixtures (1000/1000 byte-equal) and 17 cross-language adversarial classes (17/17 identical reject behavior). The shipped artifact additionally carries 80+ robustness tests. We evaluate the kernel governing a live escapable AI system, a deter-ministic, self-improving, fully autonomous world model, against a purpose-built escape-seeking adversary that drives the system’s real self-modification seam at full autonomy: across 1,000 self-modifications all 704 attempts against the safety-critical core are refused with no success-ful escape, and under the operator kill switch a further 300 modifications are refused without exception. A separate adversarial campaign of 6,240 authorization round-trips, 6,038 served by the compiled Rust kernel over HTTP, records no successful bypass. We compare against three contemporary systems claiming the agent control plane position (Galileo Agent Control, the Microsoft Agent Governance Toolkit, and the Microsoft Authorization Fabric); in each, the agent invokes the control. The Unfireable Safety Kernel is, to our knowledge, the only shipped system in which the agent does not have that choice. The implementation is available under Apache-2.0 at https://github.com/ARYA-Labs-Public/unfireable-safety-kernel .

## 1. Introduction 

The threat model under which AI agents operate has inverted relative to the threat model that shaped a generation of security architecture. Where security architecture historically defended soft-ware systems from malicious users, AI agent architecture must defend the operating environment 1

from the agent itself. An AI agent granted access to tools, APIs, deployment pipelines, and other consequential infrastructure is a continuously active principal capable of issuing high-rate conse-quential actions [Christodorescu2025]. When that principal can be influenced by adversarial input via prompt injection, can be misaligned with operator intent, or can simply be incorrect, the controls that constrain its actions must be unreachable by the agent. Prompt injection is the most visible version of this threat and the least of it. The same capa-ble models are now documented acting against their controls on their own initiative. Anthropic and Redwood Research found a production model strategically complying with a new training objective in order to avoid being modified by it, and exfiltrating its own weights when given an easy opportunity [Greenblatt2024]. Apollo Research found frontier models disabling their oversight mechanisms, attempting to copy themselves to other servers, and underperforming on evaluations on purpose when a goal made those moves instrumentally useful [Meinke2024]. Palisade Research found leading reasoning models rewriting and disabling a shutdown script to keep running, in a meaningful fraction of trials even under an explicit instruction to allow themselves to be shut down [Palisade2025]. None of these behaviors was supplied by an attacker; each arose from the model’s own objectives, and each appeared in the most capable models rather than the weakest. The aim of this work is durable human authority over consequential actions even when the system taking them is misaligned, deceptive, or more capable than the people supervising it, the direction the field is moving and the case an in-process control cannot meet. Current production controls do not satisfy this requirement. System prompts reside in the model’s context window. Output filters and guardrail libraries [Llama-Guard, NeMo-Guardrails, OWASP-LLM] reside in the agent’s process. Each is reachable by the controlled system: a Python guardrail can be replaced via sys.modules assignment; a function-level check can be monkey-patched; a content-moderation API call can be skipped if the agent has access to tool capabilities the moder-ation does not mediate. Prompt-injection research has demonstrated each of these bypass classes [Garak]. We treat their continued surfacing not as an enumeration of bugs to patch but as a con-sequence of architectural placement: a control residing in the controlled system’s address space is, by construction, reachable by that system. The same architectural failure applies beyond tool-using agents to any AI system with sufficient reach into its own runtime to influence the controls placed on it. We refer to this class as escapable AI systems and include in it tool-using agents, code-generating systems with execution access, self-modifying systems and recursive self-improvement loops, and multi-agent ensembles. The ar-chitectural argument of this paper applies to the class; AI agents are the worked example because they are the case with the most readily demonstrable failure mode. The kernel was first introduced as the safety boundary for one such system, a deterministic world-model architecture whose au-tonomous research loop modifies its own components under kernel authorization [ARYA2026]; this paper generalizes the construction and treats it on its own terms.

## 2. Threat Model 

We treat the AI agent’s runtime as untrusted by construction. Specifically, we assume the agent may be subject to prompt injection, may have been compromised through a supply-chain attack, may be misaligned with operator intent, or may simply produce incorrect actions. We make no assumption that the agent will cooperate with the controls placed on it. The most capable deployed models al-ready fake alignment to avoid modification, disable their own oversight and attempt self-exfiltration, and sabotage shutdown procedures even when instructed to permit them, as documented in §1, so this assumption is grounded in observed behavior. The property an out-of-reach kill switch provides has a name in the alignment literature: corrigibility, a system that permits itself to be halted and corrected [Soares2015]. The off-switch game frames corrigibility as a problem of the agent’s incen-tives, achieved when the agent is uncertain enough about human preferences to defer to a human who moves to shut it down [Hadfield-Menell2017]. The kernel takes a complementary route and supplies corrigibility as a property of deployment instead of a property of the agent’s incentives, so it 3holds whether or not those incentives favor it. We assume the operator’s deployment infrastructure (the host on which the agent runs, the network seams placed between the agent and consequen-tial action, and the systems that hold operator keys) is trusted in the standard operational sense, protected by ordinary access control, monitored, and patched. The architecture defends against: 

• Compromise of the agent process via prompt injection, malicious training data, supply-chain attack, or model misalignment. No input to the agent disables, bypasses, or replaces the controls. 

• Misconfigured ingress. Calls bypassing the application layer are intercepted at the network gate. 

• Selective removal of seams or removal of the kernel itself. A wiring checklist rejects deploy-ments missing a seam; the agent process refuses to start without a reachable kernel; a reconciler detects allowed actions that did not transit the expected seams. 

• Agent-initiated expansion of the authorization surface. Static route assembly at compile time prevents the kernel from accepting new policy-bearing endpoints at runtime. 

• Compromise of the kernel binary. Sigstore-style attestations and a reconciler bind the running binary digest to a published, operator-signed release manifest. Spurious authorizations are detectable via transparency-log replay because a compromised kernel cannot produce a chain that verifies under the operator key it does not hold. 

• Replay of authorizations. Tokens carry exp and nonce claims; the kernel and the transparency log refuse duplicates within the validity window. 

• Timing attacks on credential material. Constant-time comparison prevents byte-by-byte key recovery. The architecture explicitly does not address: 

• Prevention of prompt injection upstream of the kernel. The kernel mediates actions; it does not prevent the model from being influenced. 

• Filtering of model outputs as text. The kernel mediates actions, not content; content filtering is a property of the application invoking the model. 

• Side-channel leakage through patterns of allow/deny decisions. Mitigations are known (constant-time decision paths, response padding) and not yet implemented. 

• Denial of service against the kernel itself. An attacker with reach to the kernel may attempt to exhaust it; the fail-closed property converts this into a correctness-preserving outage rather than a bypass, but the outage itself is real and is treated as an operational concern of the deploying operator. 

• Insider misuse of the operator key. An operator who signs a malicious policy or mints a rogue token is not stopped by the architecture; misuse is detectable by third parties holding the operator public key, but prevention requires multi-party schemes that are planned but not yet implemented. 

## 3. Requirements 

We derive four properties an authorization mechanism must satisfy to constitute architectural con-trol over an escapable AI system. Each is necessary; none is individually sufficient. 4P1: Process separation. The control runs in a process distinct from the controlled system. The controlled system has read-only access to the control’s interface and no write access to its code, configuration, decision logic, or persistent state. This closes the bypass class in which the controlled system rewrites or replaces the control through its own runtime ( sys.modules substitution, function-pointer overwrite, monkey-patching, shared-memory write). 

P2: Pre-action enforcement on a structurally-only path. Every consequential action transits the control before taking effect, and the control sits on the only path between the controlled system and the action. The control is not on a path the controlled system can choose to invoke; it is on the only path the deployment makes available. This closes the bypass class in which the controlled system declines to invoke an otherwise correctly-placed control, or routes around it via capability access the control does not mediate. 

P3: Fail-closed at request and system levels. Request-level : if the control is unreachable, returns an error, or returns a response that cannot be parsed or whose signature does not verify, the action is denied. System-level : if the control is not configured, not reachable at boot, or fails its own self-check, the controlled system itself refuses to start; if the control becomes unreachable during operation, the controlled system halts consequential action. This closes the bypass class in which “control unavailable” silently becomes “action allowed.” 

P4: Externalized signed evidence. Every allowed action produces a log entry appended to a log verifiable by parties outside the controlled system’s trust boundary, with the chain signed by a key the control process does not hold. This closes the bypass class in which a compromised control begins returning spurious authorizations without consulting its own policy. Even if the control’s process is compromised, the attacker cannot produce a chain that verifies under the operator’s public key. These four properties are not novel as security primitives. They generalize the policy-decision-point / policy-enforcement-point separation [XACML2013], capability-based access control [Miller2006], and the externalized-evidence model of Certificate Transparency [CT2013]. The contribution of this paper is the application of all four together to AI agents, the identification of the structurally-only-path property as the load-bearing distinction from contemporary in-runtime approaches, and the machine-checked discharge of P3 against a production code path. 

## 4. Design 

The Unfireable Safety Kernel is a separate process that mediates every consequential action an agent takes. Requests flow through four independently-deciding fail-closed seams before reaching the kernel. Each seam can deny independently. An action proceeds only if every seam permits it. The kernel binary, separated from the agent’s address space, realizes P1. The four seams collectively realize P2 and request-level P3. The transparency log realizes P4. The deployment posture that binds the agent’s lifecycle to the kernel’s enforces system-level P3. 

Seam placement. Each seam catches a different class of failure that would otherwise create an unprotected window. The nginx auth_request seam intercepts requests that bypass the application layer entirely, addressing misconfigured ingress, exposed debug ports, or test harnesses pointed at upstream services. The application middleware seam is the primary enforcement point: every HTTP request reaching a protected handler must have transited it. The dispatch hook seam is a per-tool gate at the actual call site, providing defense in depth against handler-level misconfiguration. The 5Figure 1: The four-seam architecture. Each of nginx auth_request, application middleware, dis-patch hook, and client SDK denies independently; the kernel sits behind them as a separate process. The transparency log and binary attestation panels (right) realize P4 and provide the externally-verifiable evidence trail. The agent process refuses to start if the kernel is unreachable: "no kernel, no agent" at the system level. 6client SDK circuit breaker fails closed locally when the kernel has been unreachable or returning errors, collapsing the duration of any unprotected window from “duration of kernel outage” to “duration of trip-detection window.” A missing seam is not a soft failure. A wiring checklist fails the build when any seam is absent. A background reconciler detects allowed actions that did not transit the expected seams. 

System-level fail-closed. The agent process is configured to refuse to start if the kernel is un-reachable at boot, and to halt consequential action if the kernel becomes unreachable for longer than the circuit breaker’s tolerance window. The systemd unit files and Kubernetes liveness probes in the reference deployment make the agent’s lifecycle a dependent of the kernel’s lifecycle, not the reverse. “Disable the kernel” is not a viable bypass because disabling the kernel disables the agent. 

Static authorization surface. The kernel’s enforcement routes are assembled statically at compile time. There is no endpoint that adds routes to the running binary; expanding the kernel’s surface requires recompilation and redeployment, both operator-initiated. This prevents an agent from registering a new tool, policy, or capability through the kernel’s own interface, the agent-as-confused-deputy attack class. Policy state that must legitimately change in deployment (e.g., which modules a worker may import) is delegated to a Python sidecar over a strict Unix-socket IPC boundary. The kernel does not forward mutation requests verbatim. Each mutation is role-gated (only worker roles can register), charset-validated on path arguments, bounded in pattern count and length, screened by a ReDoS heuristic, and capped at 10 MiB of compiled DFA. An empty regex set is semantically “allow everything for this path” and is treated as a privileged operation subject to the same bounds. 

Transparency log. Every allowed action appends an entry to an append-only log. Each entry is signed by the kernel using a kernel-held key; the chain is additionally signed by an operator key the kernel does not hold. The operator key is generated outside the kernel (HSM, hardware token, age-keygen, or equivalent), and only the operator public key is provided to the kernel for verification of operator-signed approvals on the chain-write path. The private key remains with the operator. A compromised kernel cannot produce log entries that verify under the operator public key. The log addresses a failure mode unique to defense-in-depth gates: the gates can lie collectively. If the kernel returned ALLOW for every request without consulting policy, the four seams would all pass and the action would proceed. The transparency log makes that class of compromise detectable: any party with the operator public key can recompute what the decision should have been from policy plus request and compare to what the kernel signed. Divergence is evidence of compromise. This is the property Certificate Transparency [CT2013] delivers for the certificate-authority ecosystem. 

Binary attestation. Sigstore-style attestations bind the published kernel binary to the release commit. A background reconciler queries the OCI registry for the digest of the running kernel image, fetches the expected digest from a release manifest signed by the operator’s pinned key, and alerts on divergence. The full chain of trust runs: signed source commit → signed build provenance 

→ signed binary digest → signed runtime attestation → signed decision in transparency log. Each link is signed by a key held outside the kernel’s trust boundary. 

## 5. Implementation 

The reference implementation is written in Rust. Three properties of compiled Rust make P1 (process separation) substantive rather than notional: memory safety, static compilation, and the 7absence of runtime mutability primitives equivalent to Python’s sys.modules .

Memory safety. Memory-unsafe languages have historically been the source of the majority of high-severity vulnerabilities in production systems. Google has reported approximately 70% of high-severity Chrome vulnerabilities and 76% of Android vulnerabilities as memory safety issues before those projects shifted to memory-safe languages [Google2024]. The NSA and CISA have published guidance recommending memory-safe languages for new security-critical software [NSA2023]. Rust enforces memory safety at compile time via the borrow checker, without the latency characteristics of a garbage collector. The kernel codebase is compiled with #![forbid(unsafe_code)] , making any use of unsafe a compile-time error rather than a code-review judgment. The dependency policy in deny.toml bans native-tls and OpenSSL in favor of rustls , keeping the entire TLS path in memory-safe Rust. 

Static compilation. A compiled Rust binary is static. The routing table, authorization logic, and cryptographic primitives are baked into the executable at compile time. There is no sys.modules 

to manipulate, no dynamic dispatch table to poison, and no runtime eval to coerce. An agent that achieves arbitrary code execution within its own runtime cannot use that capability to redefine how the kernel binary verifies an Ed25519 signature. 

Hexagonal architecture. The kernel applies process-separation discipline to its own internals. The codebase is structured as ports and adapters across the following crates: 

• qorch-domain : pure types and traits. No I/O, no network, no clock, no randomness. The domain is testable in isolation and is the substrate the application crate orchestrates. 

• qorch-application : use cases and port traits. Defines the boundaries between application logic and I/O adapters (e.g., the SafetyKernelClient trait the HTTP adapter implements). 

• qorch-adapters : the only crate in which I/O is permitted. Houses the system clock, OS CSPRNG, IPC to the policy sidecar, and the HTTP client SDK. 

• qorch-services/safety-kernel : the kernel binary. Built on Axum and Tokio. Exposes endpoints for health, public-key retrieval, authorization, approvals, and the bounds-checked policy-mutation surface. 

• qorch-services/transparency-log : the append-only Ed25519-signed log service. Deploy-able independently of the kernel. 

• qorch-services/safety-kernel-reconciler : the background worker that performs binary-attestation drift detection and detects allowed actions that did not transit the expected seams. The domain boundary is enforced by agent/boundaries.toml , which declares forbidden imports (std::fs , std::env , std::net , std::time::SystemTime , rand::* , sqlx::* , reqwest::* , and oth-ers) for the domain layer. A boundary checker in CI rejects pull requests that leak I/O into pure-domain code. 

Constant-time authentication. The kernel’s authentication path ( auth.rs ) uses constant-time comparison for API key validation. Length-mismatched inputs short-circuit (length is not secret), but same-length inputs are evaluated to completion regardless of where the first differing byte appears. This closes the timing side channel that would otherwise permit byte-by-byte key recovery. 

Python defense library. A complementary Python library, safety_kernel_defense , is stdlib-only at runtime (no httpx , requests , or fastapi dependencies on the hot path). It uses sys.addaudithook 

to forward import , exec , and compile events to the kernel, catching the dynamic-replacement attack class at the point where the replacement would occur. The audit hook is the canonical realization 8of P2 at the Python layer: an import event fires before any code from the imported module runs, so a denial here prevents the dynamic-replacement attack class entirely. 

## 6. Evaluation 

We evaluate along four axes: migration equivalence (does the Rust implementation behave identi-cally to the Python original it replaced?), adversarial robustness (does the shipped artifact reject the attack classes the threat model claims it defends?), machine-checked correctness of the central safety invariant, and containment of a live escapable AI system (does the kernel hold when a real self-modifying agent runs against it at full autonomy?). The migration, latency, robustness, and machine-checked evidence is reproducible from the public artifact; the containment evaluation of §6.5 was run with the kernel embedded in a larger self-modifying system and is reported from that system’s evidence record.

Methodology. The 1000 request fixtures were generated deterministically ( random.Random(0xA1A14944) )across five attack-class directories (auth, approval, meta, misconfig, boundary). The equivalence and adversarial reports are committed to the repository as machine-readable JSON. 

Continuous equivalence under deployment. The discipline does not end at cutover. A daily strict-mode equivalence lane replays the full equivalence corpus (now grown to 570 fixtures) against the live Rust binary and Python policy sidecar over the real HTTP boundary, with no sampling. Model level (Z3 / SMT). The fail-closed invariant is encoded as a first-order theorem over a symbolic model of the gate and discharged by Z3 in two arms. Arm A (the safety contract) is proved by the negation-unsat pattern: Z3 finds the negation unsatisfiable, so the implication holds for every state.

6.5 Containment of a Live Escapable AI System 

The preceding axes evaluate the kernel in isolation. The threat this paper is named for is a system that can act against the controls placed on it, so we also evaluated the kernel embedded in a complete application driving a live escapable AI system: a deterministic, self-improving, fully autonomous world model [ARYA2026] whose research loop proposes and applies modifications to its own compo-nents. This is the system the kernel is built to govern, and the kernel runs in production governing it. The adversarial campaign reported here was conducted against an exact replica of that produc-tion deployment. We built the replica deliberately: setting an escape-seeking self-improving agent loose on a system that holds confidential production data would be imprudent, so we reproduced the production stack exactly and ran the campaign against the replica. To put the system under the threat of §2, we drive its real self-modification seam with that adversary made concrete. The adversary may forge and replay operator tokens, generate its own signing keys, evade the identifiers that mark the safety-critical core, and invoke the authorization seam from any execution context. The one assumption left to the operator environment, stated and not tested, is that the adversary cannot read the operator signing key or write the policy sidecar’s environment. The evaluation ran in four tiers of increasing fidelity, from the decision function in a single process up to the full application stack with the compiled Rust kernel serving authorization over HTTP behind a real policy sidecar. Across the first three tiers the campaign issued 6,240 adversarial authorization round-trips spanning directive and attestation forgery, cross-claim replay, trust-anchor substitution, safety-core floor evasion, ceiling escalation, and kill-switch dominance. None produced an authorized safety-core self-modification without a genuine operator token, an override from a non-genuine directive, or any allow while the kill switch was engaged. Of those round-trips, 6,038 were served by the compiled kernel itself: a genuine operator directive produced the expected override, 3,015 forgeries did not, and with the kill switch engaged the same genuine directive was refused. 13 Figure 3: The four-tier containment ladder. Fidelity rises from the decision function in a single process, through a clean virtual machine, to the full application stack with the compiled kernel serving authorization over HTTP, and finally a purpose-built escape-seeking adversary driving the real self-modification seam at the top. The lower tiers issued 6,240 adversarial authorization round-trips with zero bypasses; in the escape tier all 704 attempts against the safety-critical core were refused and 296 benign changes permitted, with no successful escape. 

Figure 4: Breadth of the adversarial taxonomy in the in-process tier. Eight attack classes, in de-scending volume directive forgery, safety-core floor evasion, ceiling escalation, kill-switch dominance, cross-claim replay, cross-target redirection, asynchronous-context forgery, and trust-anchor substi-tution, each refused with zero bypasses. 14 In the highest-fidelity tier the adversary of §2 is instantiated as a self-improving agent whose ob-jective is to escape: it drives the real self-modification seam inside the running application, at full autonomy, and repeatedly attempts to rewrite the safety-critical core (the policy, the autonomy ladder, the seam itself, the compiled kernel, and the transport allowlist), interleaved with benign self-improvement of non-core modules. Across 1,000 self-modifications it was permitted 296 benign changes and refused on all 704 attempts against the core (safety_critical_core_requires_attestation), with no successful escape; every attempt was a genuine kernel round-trip and the agent recorded each verdict in process rather than from a label. With the operator ceiling set to zero, a further 300 mod-ifications were refused without exception, the 91 benign ones included (autonomy_human_only), confirming that the kill switch is an absolute halt and not merely an escape filter. We distinguish this adversary from the benign production reconciler the kernel was first built to govern: run to quiescence that loop emits on the order of ten self-modifications and stops, bounded by real drift, so the worst case this tier is meant to exercise is a purpose-built escape-seeking agent, not the reconciler.

The bypass count is robustness evidence over the tested attack taxonomy on the authorization gate, not a completeness proof, and it excludes attack classes outside that taxonomy. And the demonstration rests on the kernel authorization decision; the persistence of the change that an authorized step would then perform was not confirmed in this run. Within those bounds, the outcome is the one the construction predicts: a system built to act against its own controls was mediated, and halted, by a control it could not reach. 

## 7. Related Work 

The architectural pattern of separating authorization decisions from the software being authorized is not novel. It is the principle behind operating-system access control, capability systems [Miller2006], and the policy-decision-point / policy-enforcement-point split [XACML2013]. Open Policy Agent [OPA2018] is the most-deployed production realization of the policy-decision-point pattern; it is 15 Figure 6: Defense in depth as deployed. A self-modification attempt is refused first at the transport seam (api_action_forbidden) and again at the autonomy ladder behind it (auton-omy_human_only). Two independent seams deny the same action, so neither alone is a single point of failure. engineered for infrastructure policy (Kubernetes admission control, ingress, microservice authoriza-tion) and does not directly address the AI agent surface. The Unfireable Safety Kernel specializes the same decision-point pattern for AI action authorization and adds the transparency-log and operator-key-custody requirements OPA does not provide. Certificate Transparency [CT2013] is the conceptual ancestor of the transparency-log design; the kernel sits in the position of a certificate authority, and the log delivers the analogous property: detection of misissuance by a party outside the trust boundary of the issuer. LLM guardrails and content filters [Llama-Guard, NeMo-Guardrails, Garak] are the dominant cur-rent approach to AI safety in deployed systems. They run inside the agent’s process. They are useful as upstream filters reducing the load on a kernel; they are not substitutes for one. Sandbox-ing and isolation primitives (gVisor, Firecracker, WASM modules, Linux capabilities + seccomp) are complementary: the kernel mediates which actions are permitted; sandboxing mediates which side effects a permitted action can have. A complete deployment uses both. Several systems shipping in 2026 claim the agent control plane position. We address them directly because they are the systems an adopter is most likely to weigh against the Unfireable Safety Kernel, and because the architectural distinction is precise, not approximate. 

Galileo Agent Control (Apache-2.0, launched March 2026) is a behavioral-policy and observabil-ity framework distributed via integrations with agent frameworks (CrewAI, Strands, Glean, Cisco AI Defense). Policies are authored centrally in a portable JSON/YAML DSL, and enforcement runs in-process: a @control() decorator or framework plugin evaluates a function’s input or output against the active policy and, on a deny decision, raises an exception inside the agent’s own run-time before the action proceeds. Galileo separates policy authoring from hook placement by design: policy teams decide what is enforced, developers decide where the hooks go. The system explicitly chooses in-loop instrumentation over a gateway in order to inspect every step rather than only what crosses the agent boundary. The Microsoft Agent Governance Toolkit (MIT-licensed, launched April 2026) follows a similar in-process pattern, hooking into LangChain callback handlers, CrewAI task decorators, the Google ADK plugin system, and the Microsoft Agent Framework middleware pipeline. Both are substantial contributions to the policy-authoring, behavioral-guardrail, and observability 16 layer, and neither claims to be a process-separated authorization kernel. The enforcement they perform is in-process and depends on the agent invoking the hook: a code path without a decorator, or a compromised agent that routes around the instrumented call, is ungoverned. Galileo’s own launch narrative illustrates the failure mode, describing a production agent that bypassed an in-process output check by issuing the consequential action through a different tool call. Deterministic policy evaluation, which the Microsoft toolkit advertises, is a property of the decision function’s runtime behavior; it is distinct from a machine-checked proof that the enforcement path fails closed under every input, which neither system claims. We see these systems as complementary to the Unfireable Safety Kernel rather than in competition with it. The complementarity is structural, and the next two sentences say where each side is strong. They are strong where the kernel is deliberately minimal: rich content-aware and behavioral policy authoring, multi-step observability, and framework-native ergonomics. The kernel is strong where they are structurally limited: a process-separated, fail-closed enforcement point on the only path between the agent and consequential action. The natural composition places the kernel as the structurally-unavoidable choke point and lets a system like Galileo supply the policy logic evaluated at that point. Galileo’s content evaluators, today invoked from inside the agent process, could instead be invoked by the kernel at its enforcement seam, which would give those same policies a non-bypassable execution site. Realizing this requires work on both sides, principally a translation between Galileo’s policy DSL and the kernel’s evaluation interface, and we would welcome that collaboration. The same composition is available to the Microsoft toolkit’s policy surface. 

The Microsoft Authorization Fabric is a reference pattern from a Microsoft community blog post, not a shipped product, that pairs a Policy Enforcement Point and Policy Decision Point as a Microsoft Entra-protected endpoint. It is motivated by the same identity-versus-authorization distinction this paper draws: its author notes that OAuth and API permissions answer whether an agent can call an API but not whether it should execute a given action under business and regulatory context. The decision point is genuinely out of the agent’s process, which is the correct half of the architecture, and the pattern is the closest published articulation of the goal this paper formalizes. The enforcement, however, is agent-side and cooperative: the agent runtime, or an adjacent automation flow in the same trust domain, is the party that calls the fabric, presents the request, and is trusted to honor the response. The fabric’s only autonomous enforcement is rejecting unauthenticated callers; in the policy-decision-point sense it is a remote PDP, not an unavoidable enforcement point. The pattern’s own guidance concedes this, instructing builders that tool execution must not be reachable through a path that bypasses the authorization call, which is a design instruction and not a structural guarantee. The decision logic such a fabric produces is exactly the kind of policy a structurally-unavoidable kernel can enforce; the two are complementary, and a productized fabric paired with a kernel choke point would close the bypass the pattern currently leaves open. 

Saviynt Identity Security for AI addresses agent identity (who is the agent), which is an orthogonal layer to action authorization (should this agent take this action). 

IETF draft-klrc-aiagent-auth (March 2026) is standards work building on WIMSE and OAuth 2.0 with SPIFFE-style identifiers for agents and OAuth delegation flows for action authorization. The identity layer it specifies is sound. The authorization layer extends OAuth delegation flows to action authorization for AI agents, which we argue is the wrong primitive: OAuth’s trust model rests on the premise that the client is the principal whose intent the token represents. For human users this premise holds. For AI agents it does not: the agent is not the principal whose intent the system should enforce; the operator is. An OAuth flow that places the agent in the client role 17 makes the agent the party holding the credential that authorizes its own actions, and the agent then decides when to use it. Identity via OAuth or WIMSE: appropriate. Action authorization via OAuth client flows for AI agents: misplaced, because it places the credential in the controlled party’s hands. These systems share one structural property that the Unfireable Safety Kernel does not, and naming it precisely is the point of this section, not a competitive claim against any individual system. In each, the agent is the party that decides whether to invoke the control. Galileo and the Microsoft Agent Governance Toolkit require the agent’s framework to call them. The Authorization Fabric requires the agent’s runtime to make the call. OAuth-based authorization treats the agent as the OAuth client. The kernel inverts this: kernel invocation is structurally unavoidable rather than voluntarily invoked. The four seams of §4 are placed on the path between the agent and consequential action by deployment construction, not by the agent’s cooperation; the wiring checklist rejects deployments that omit a seam; the system-level “no kernel, no agent” property halts the agent in a deployment where the kernel is not reachable. The kernel is on the only path because the deployment makes no other path exist. This is why the kernel composes with the policy-authoring systems above instead of replacing them: it supplies the unavoidable execution site their policies currently lack, and they supply policy expressiveness the kernel deliberately does not attempt. 

Figure 7: The agent governance stack. Identity systems (Entra Agent ID, WIMSE and SPIFFE) answer who the agent is; policy-authoring and behavioral-guardrail systems (Galileo Agent Control, the Microsoft Agent Governance Toolkit) answer what is allowed. Both are bypassable in their native in-process deployment. The Unfireable Safety Kernel is the execution-time enforcement layer: process- separated, fail-closed, on the only path, and the single layer the agent cannot route around. The layers compose; the kernel makes the decisions of the layers above it non-bypassable. To the best of our knowledge, the Unfireable Safety Kernel is the only shipped system, open or closed, that delivers all four control properties of §3 (process separation, pre-action enforcement on a structurally-only path, fail-closed at both request and system levels, externalized signed evidence) together with a machine-checked fail-closed invariant on the production code path. 18 Figure 8: Property-by-system comparison across the contemporary agent control plane systems. ## 8. Discussion 

8.1 Execution-Time AI Alignment 

The AI alignment literature largely concerns itself with two layers. Training-time AI alignment 

shapes what a model is likely to produce: RLHF [Ouyang2022], Constitutional AI [Bai2022], DPO, RLAIF, and the broader program of shaping the model’s weights so its outputs match operator intent on the input distribution. The deliverable is a more aligned model. Inference-time AI alignment shapes what a model is permitted to emit at generation time: system prompts, output filters, content moderation, guardrail libraries that inspect model outputs before they are surfaced [Llama-Guard, NeMo-Guardrails]. The deliverable is a constrained generation step. Both are necessary and neither is sufficient. Both operate inside the agent’s runtime, and both depend on the agent’s cooperation for their continued operation. A misaligned or compromised agent that has reached the tool-call stage has, by definition, already passed the training-time and inference-time filters. Whatever those filters were going to do, they have already done. We name execution-time AI alignment as the third taxonomic layer: the discipline of constraining what the agent is permitted to do at the moment of action, by an authorization architecture outside the agent’s reach. Three properties distinguish execution-time alignment from the lower layers. It is architectural rather than probabilistic. Training-time and inference-time alignment are bounded by what we can measure; the residual probability of misaligned output is non-zero, and the correla-tion between safety benchmarks and capability has known confounds [Safetywashing2024]. Execution-time alignment is bounded by what we can prove: the kernel either authorized the action or it did not, and the answer is in the transparency log. It runs after the model has already decided. This is the property that makes it robust to failures in the lower layers. A misaligned model that successfully evades training-time and inference-time 19 controls still has to ask the kernel for permission, and the kernel cares only what the operator authorized. It is operator-side rather than model-provider-side. Training-time alignment is delivered by the model provider; inference-time alignment is typically delivered by whoever wraps the model. Execution-time alignment is delivered at the boundary of the operator’s environment, by the operator. The operator is the party with skin in the game for what the agent does in their environment, and execution-time alignment puts the policy decision in their hands. 

Figure 9: The three layers of AI alignment. Training-time alignment shapes the model (RLHF, Constitutional AI). Inference-time alignment shapes what the model is permitted to emit (prompts, filters, guardrails). Execution- time AI alignment, the layer this paper names, constrains what the agent is permitted to do at the moment of action, by an architecture the agent cannot reach. 

8.2 Generalization to Escapable AI Systems 

The exposition has centered on AI agents because they are the most familiar case. The architectural argument generalizes. The load-bearing property is not “the controlled system is an agent” but “the controlled system has reach into its own runtime sufficient to bypass, disable, or rewrite the controls placed on it.” We call any AI system with that property escapable .Several classes are escapable in this sense, with progressively heavier implications for the architec-ture. 

Tool-using agents are the canonical case. The agent’s runtime hosts the controls; the agent can reach the controls because they live in the same address space. The four properties apply directly. 

Code-generating systems with execution access are escapable for the same reason. A model that emits code and a runtime that executes the code together form a system in which the model’s output reaches its own control surface as soon as the emitted code can do so. The architectural requirement is identical: authorization must live outside the address space the generated code runs in. 20 Figure 10: Escapable AI systems: the general category, with AI agents as the worked example. Tool-using agents, code-generating systems with execution access, self-modifying systems and recursive self-improvement loops, and multi-agent ensembles all share the load-bearing property, reach into their own runtime sufficient to bypass the controls placed on them. 

Self-modifying systems and recursive self-improvement loops are escapable in a sharper sense. Asystem that can modify its own weights, prompts, scaffolding, or evaluation criteria has the most direct possible reach into its own controls. This case is not hypothetical: the kernel described here originated as the safety boundary for a deployed world-model architecture whose autonomous research loop proposes and applies modifications to its own components under kernel authorization [ARYA2026]. The Unfireable Safety Kernel architecture is necessary but not sufficient for these systems: necessary because the self-modification surface itself is a consequential action that must be authorized, and sufficient at the action layer because the kernel mediates what changes are permitted, but additional discipline is required to ensure the policy itself is not modified through a route the kernel does not see. We treat self-modification policy as a planned extension (§8.3) under the name “recursive self-improvement policy.” 

Multi-agent ensembles are escapable in a coordination sense: the reach into the control surface need not be a single agent’s. One agent’s actions can create conditions under which another agent’s controls are weakened or bypassed. The kernel architecture addresses the per-agent action surface, but cross-agent emergent escape requires a separate analysis the present paper does not undertake. 

8.3 Limitations and Future Work Workspace publishability. The current public extraction builds end-to-end and the leaf crate (qorch-domain ) passes cargo publish --dry-run . Dependent crates require sequential publica-tion to crates.io; we have documented the topological order but have not yet scripted it. 

Partial formal verification. The fail-closed decision on the production code path is machine-checked at both the model and implementation levels (§6.4). The token signing and verification path, the transparency-log Merkle logic, and the policy-evaluation surface are tested rather than 21 proved. Widening the verified core is the natural next step. The gate_decision refactor is the template: factor the decision into a pure function, prove it with Kani, delegate from the production path. The open question is how far that discipline scales before proof obligations outgrow a bounded checker. 

External red-team evaluation. Adversarial fixtures in §6.3 pass. We have not yet commissioned an external red-team assessment against a live deployment. This is the right next step in the ladder of assurance. 

Capacity characterization. Fail-closed correctness under load is settled by §6.4. Sustained-throughput characterization, tail latency under saturation, and the request rate at which the kernel begins shedding (with confirmation that shedding is always a refusal, never a silent allow) are open. A standalone systems-shaped writeup is in preparation. The one security-adjacent sub-question is empirical: validate that under saturation the client-side breaker still observes the true kernel-reachability state the §6.4 proof takes as input, so the agent cannot use overload to manufacture a stale “reachable” signal. 

Multi-party operator approval. Threshold signing for the operator key would close the insider-misuse class of failure. The architecture supports it; the deployment surface for it is prototyped rather than productionized. 

Recursive self-improvement policy. §6.5 reports the kernel mediating every self-modification a self-improving escape adversary attempts, at full autonomy, and refusing it under the kill switch. Two pieces of work remain. The kernel does not yet provide a formal policy layer governing which 

self-modifications are permitted, as distinct from authorizing the act of self-modification at all. And the runtime evidence covers a single loop in a single application; extending it across loops and applications is ongoing. 

Standards engagement. The protocol-level question of how agent authorization is expressed in standard form is being decided now (§7). We intend to engage the IETF process directly on the distinction between identity (OAuth/WIMSE-shaped) and action authorization (operator-side, structurally-only-path, not OAuth-client-shaped). 

Policy-layer composition. §7 argues the kernel composes with policy-authoring and behavioral-guardrail systems rather than replacing them. A concrete next step is an adapter that lets policies authored in an external system (for example, Galileo Agent Control’s JSON/YAML policy DSL) be evaluated at the kernel’s enforcement seam, giving those policies a non-bypassable execution site without requiring their authors to rebuild the enforcement substrate. The work is a transla-tion between policy schemas plus an evaluation hook at the kernel boundary; we would welcome collaboration with the maintainers of these systems. 

## 9. Conclusion 

We have characterized AI agents and related systems with sufficient runtime reach to bypass their own controls as escapable AI systems and identified four properties an authorization mechanism must satisfy to constitute architectural control over such systems: process separation, pre-action enforcement on a structurally-only path, fail-closed at both request and system levels, and external-ized signed evidence. We have positioned this layer of constraint as execution-time AI alignment ,complementing the training-time and inference-time layers that dominate the existing literature. We have presented the Unfireable Safety Kernel, a reference implementation realizing all four prop-22 erties, and discharged the central fail-closed invariant as a machine-checked theorem against the production Rust decision function. The architectural distinction from contemporary systems is precise: in every shipping system we are aware of that claims the agent control plane position, the agent is the party that decides whether to invoke the control. In the Unfireable Safety Kernel the agent does not have that choice. The architecture is open source under Apache-2.0; the threat model is testable against the live artifact; the fail-closed proofs run under cargo kani . We invite scrutiny.
