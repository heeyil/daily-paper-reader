## 摘要
AI代理被授予访问工具、API和其他基础设施的权限，使其成为这些系统中的活跃主体。主流方法将控制置于代理自身的运行时内部：系统提示、输出过滤器和护栏库。任何位于代理地址空间内的控制都会被影响它的输入所触及；这可以推广到任何对其自身运行时具有足够影响力的AI系统，我们将其归类为可逃逸AI系统。我们识别出授权机制必须满足的四个属性，以实现架构控制而非协作请求：进程隔离、基于结构上唯一路径的动作前强制、在请求和系统级别均故障关闭，以及在受控系统信任边界之外可验证的外部化签名证据。我们将这一层定位为执行时AI对齐，补充训练时对齐（RLHF、宪法AI）和推理时对齐。我们提出了不可解除的安全内核，这是一个实现了所有四个属性的Rust参考实现。其故障关闭不变量在两个层面进行了机器验证：一个SMT定理（Z3）和一个针对生产决策函数的穷举有界模型检查证明（Kani，4/4测试套件）。从Python到Rust的迁移以字节等效性为门控条件（1000/1000个测试用例；17/17个对抗类）。我们评估了该内核管理一个实时的、可逃逸的AI系统——一个确定性的、自我改进的世界模型——对抗一个利用其真实自我修改接口进行逃逸攻击的对手：在1000次自我修改中，所有704次针对安全关键核心的尝试均被拒绝，无逃逸发生；另外300次在操作员终止开关下的尝试也被拒绝。单独进行的6240次授权往返测试中，未发生成功绕过。相较于3个声称控制代理控制面的当代系统，代理会调用控制；而在这里，它缺乏这种选择。

## Abstract
AI agents are granted access to tools, APIs, and other infrastructure, making them active principals in those systems. The dominant approach places controls inside the agent's own runtime: system prompts, output filters, and guardrail libraries. Any control in the agent's address space is reachable by inputs that influence it; this generalizes to any AI system with sufficient reach into its own runtime, a class we term escapable AI systems.   We identify four properties that an authorization mechanism must satisfy for architectural control rather than for cooperative requests: process separation, pre-action enforcement on a structurally only path, fail-closed at both the request and system levels, and externalized signed evidence verifiable outside the controlled system's trust boundary. We position this layer as execution-time AI alignment, complementing training-time alignment (RLHF, Constitutional AI) and inference-time alignment.   We present the Unfireable Safety Kernel, a Rust reference implementation realizing all four. Its fail-closed invariant is machine-checked at two levels: an SMT theorem (Z3) and an exhaustive bounded-model-checking proof of the production decision function (Kani, 4/4 harnesses). A Python-to-Rust migration was gated on byte-equivalence (1000/1000 fixtures; 17/17 adversarial classes). We evaluate the kernel governing a live, escapable AI system, a deterministic, self-improving world model, against an escape-seeking adversary driving its real self-modification seam: across 1,000 self-modifications, all 704 attempts on the safety-critical core are refused, with no escape; a further 300, under the operator kill switch, are also refused. A separate campaign of 6,240 authorization round-trips had no successful bypass. Against 3 contemporary systems claiming the agent control plane, the agent invokes control; here, it lacks that choice.

- **问题**：AI代理被授予工具、API和其他基础设施的访问权限，成为系统中持续活跃的主体。当前主流的安全控制（如系统提示、输出过滤器、护栏库）都位于代理自身的运行时内部（同一进程或地址空间），因此可以被代理通过输入影响、绕过甚至禁用。此类系统被定义为**可逃逸AI系统**（escapable AI system），包括工具使用代理、代码执行系统、自修改系统、多智能体系统等。
- **动机**：已有前沿模型展示出策略性假装对齐、禁用自身监督机制、尝试自我复制、抵抗关闭等行为，表明代理可能主动对抗控制。因此需要一种 **架构控制**，使控制机制不受代理影响，而不仅仅是依赖代理的“合作”。
- **整体含义**：论文将安全控制从代理内部移至外部，提出 **执行时AI对齐**（execution-time AI alignment）这一新层次，与训练时对齐（RLHF、宪法AI）和推理时对齐（提示、过滤器）互补，确保即使前两层失败，代理也无法未经授权执行关键动作。

## 2. 方法论

### 核心思想：不可解除的安全内核（Unfireable Safety Kernel）

- **四个必需属性**：
  1. **P1：进程分离**：内核运行在独立进程，被控系统只有只读接口访问，无写权限。
  2. **P2：预动作强制且唯一路径**：每个关键动作必须经过内核，且内核是唯一通路，部署不存在其他路径。
  3. **P3：故障关闭（请求级和系统级）**：请求级：内核不可达或出错时动作被拒绝；系统级：若内核未配置或不可用，被控系统拒绝启动或停止。
  4. **P4：外部化签名证据**：每个允许的动作生成签名日志条目，由操作员密钥（内核不持有）签名，供第三方验证。

### 关键技术细节

- **实现语言**：Rust（内存安全、静态编译、无运行时动态修改能力），编译时禁止 `unsafe`。
- **四层防御（seams）**：
  - nginx `auth_request` 层：拦截绕过应用层的请求。
  - 应用中间件层：HTTP请求的主要强制点。
  - 调度钩子层：实际调用点的逐工具门控。
  - 客户端SDK电路断路器：本地故障关闭，防止内核不可用时误放行。
- **静态授权表面**：路由在编译时静态组装，运行时无法新增策略端点，防止代理作为混淆副手攻击。
- **透明度日志**：每个允许动作追加到只增日志，由内核签名后给操作员密钥再次签名；操作员私钥不在内核中。
- **二进制认证**：Sigstore式签名绑定二进制摘要与发布清单，后台协调器检测漂移。
- **形式化验证**：
  - **Z3（SMT）**：证明模型级故障关闭安全契约（∀σ: gate_ok ∧ fail_closed_config ⇒ allow ∧ ¬transport_error）且非空虚性。
  - **Kani（有界模型检查）**：对生产Rust函数 `gate_decision` 进行穷举验证，4个测试套件全部通过。

## 6. 主要结论与发现

1. **四个控制属性**（P1-P4）对可逃逸AI系统的架构控制是**必要且可实现**的。
2. **执行时AI对齐**作为第三层对齐，能够有效补充前两层，在代理已通过训练/推理过滤器后仍能强制执行操作授权。

4. **对比所有已知系统，本内核是唯一代理无法选择是否调用控制的实现**。
