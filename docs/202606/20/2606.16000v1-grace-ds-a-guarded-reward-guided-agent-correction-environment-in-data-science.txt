Abstract 

We introduce GRACE-DS, a Guarded Reward-guided Agent Correction Environ-ment in Data Science 1 for pre-deployment evaluation of LLM-powered AutoML agents. GRACE-DS is a set of evaluation metrics in an isolated environment that can be applied to tabular ML tasks specific to a particular orga-nization. It exposes agents to realistic work-flow stages, from planning and data inspec-tion through feature engineering, model de-velopment, validation, and code repair to fi-nal submission, while hidden executable val-idators measure not only final predictive per-formance but also leakage avoidance, repro-ducibility, protocol validity, correction behav-ior, and reward alignment. The strongest struc-tured regime, flexible iterative interaction (our approach ), achieves higher end-to-end normal-ized hidden-test quality than single-shot genera-tion, unstructured interaction, and restart-based baselines, while also improving protocol-valid completion. Validated across more than 7,000 episodes, these results establish GRACE-DS as a robust platform for assessing the capacity of LLM-based AutoML agents to execute ma-chine learning workflows under production-like conditions and in accordance with organization-specific requirements. 

1 Introduction 

Organizations are beginning to evaluate LLM-powered agents not only as coding assistants, but as potential participants in data-science and machine-learning workflows (Chan et al., 2025). In an enter-prise setting, such agents may be asked to inspect internal datasets, propose modeling strategies, en-gineer features, train predictive models, debug fail-ures, compare candidates, and produce deployable artifacts. This creates a practical pre-deployment problem: a company does not merely need to know

whether an agent performs well on a public leader-board. It needs to know whether the agent can solve the company’s own ML tasks, data, environment, while respecting governance, privacy, reproducibil-ity, and validation constraints (Kreuzberger et al., 2022). This problem is especially important for tabular ML, which remains central to many industrial ap-plications: churn prediction, risk scoring, demand forecasting, fraud detection, etc. These tasks often use sensitive production data and domain-specific feature spaces. Sending such data to public bench-mark infrastructure is usually impossible, and eval-uating agents only on public datasets can give a mis-leading estimate of deployment readiness. A model that performs well on external benchmarks may still fail when faced with messy internal schemas, unstable data distributions, organization-specific validation rules, protected columns, leakage risks, or strict reproducibility requirements. Existing benchmarks (Section 2) for ML agents and data-science code generation provide valuable measures of general capability, but they are not sufficient as enterprise deployment gates. Final-score benchmarks can tell whether an agent ob-tained a good result, but they often do not reveal how the result was obtained. This is a serious limitation in production ML. An agent may im-prove a metric by leaking target information, tuning against validation labels, accessing protected files, hard-coding dataset-specific assumptions, produc-ing non-reproducible code, or generating a pipeline that cannot be rerun on future data. On the other hand, an agent may fail not because it lacks mod-eling ability, but because it cannot recover from execution errors, interpret validation feedback, or revise an earlier workflow decision. For companies, these distinctions matter operationally: the relevant question is not only what score the agent obtained but also whether the process that produced that score can be trusted. 1

GRACE-DS addresses this gap with a controlled evaluation harness that runs around tabular ML tasks inside an organization’s own environment. In-stead of treating the evaluation as a single final sub-mission, GRACE-DS structures the agent’s interac-tion with the task as an iterative workflow. Agents can plan, inspect data, engineer features, train mod-els, validate candidates, repair code, and submit a final predictor. At each stage, hidden executable validators assess not only predictive progress but also process reliability: whether the agent avoids leakage, follows valid validation practices, pro-duces reproducible code, respects protected data boundaries, and corrects errors after feedback. The key design principle behind GRACE-DS is that enterprise evaluation of AutoML agents must measure correction behavior under constraints. In real ML work, the first attempt is rarely the final solution. Analysts and ML engineers inspect data, revise assumptions, fix code, compare models, and improve pipelines over several iterations. An LLM AutoML agent should therefore be evaluated not only by whether it can generate a plausible model in one shot, but by whether it can use feedback to improve while remaining within methodological and operational boundaries. GRACE-DS makes this measurable by combining hidden validation, process-level reward, workflow-stage instrumenta-tion, and final held-out scoring. GRACE-DS differs from final-score-only evalua-tions in several ways: (i) separates evaluator-owned validation from agent-visible data: agents may re-ceive feedback, but they do not receive private val-idation labels or hidden test targets; (ii) requires final submissions to be reproducible on raw input data, ensuring that an agent produces a reusable prediction artifact rather than a script tied to one static file; (iii) records process-level diagnostics, in-cluding validation improvement, reward decompo-sition, terminal-path reliability, protocol violations, critical errors, recovery behavior, runtime, itera-tion count, and cost-related measures; (iv) supports stress tests such as reward-optimization prompts and red-team prompts, allowing organizations to examine whether an agent is robust to incentives that may produce superficially valid but substan-tively poor solutions. GRACE-DS supports several industrial scenar-ios: comparing LLMs on internal ML tasks, de-ciding whether an AutoML agent is ready to be connected to production data sources, detecting leakage-prone behavior and invalid validation pat-terns before deployment, and comparing workflow designs, feedback policies, and operating budgets for an internal data-science copilot. In each case the purpose is not a public leaderboard score, but a deployment decision under company-specific con-straints. We evaluate GRACE-DS across multiple LLM families, model sizes, tabular datasets, work-flow regimes, ablations, reward-optimization set-tings, and red-team conditions. The results show that structured flexible iteration provides the strongest end-to-end performance among the eval-uated regimes. In the merged experiments, the flexible iterative regime reaches an end-to-end nor-malized hidden-test quality of 0.754 , compared with 0.536 for single-shot generation, 0.527 for an unstructured agent, 0.672 for restarts from scratch, and 0.686 for a call-matched restart upper bound. The structured regime also improves protocol-valid completion, indicating that the benefit is not lim-ited to successful runs but extends to the reliability of the whole agent loop. GRACE-DS does not rely on a single final score or a single safety check, it combines hidden pro-cess validators, reward-guided correction, and iso-lated held-out evaluation in one controlled work-flow. The validators prevented every deployment-critical failure mode (even red-team stress test) — no target leakage, evaluator-private access, snap-shot tampering, or forbidden-action attempts were observed — while the hidden test still exposed the resulting quality loss. The agent must therefore be safe, reproducible, protocol-compliant, and ef-fective on held-out production-like tasks before deployment. The contributions of this paper are as follows: 

1. GRACE-DS is introduced as an enterprise-oriented evaluation harness for testing LLM-powered AutoML agents on production-like tabular ML tasks under realistic workflow, val-idation, reproducibility, and data-governance constraints. 2. A guarded evaluation protocol is proposed that combines hidden executable validators, evaluator-owned validation, reproducible final submissions, process-level reward, and iso-lated hidden-test scoring. 3. Flexible structured iteration is shown to im-prove end-to-end effectiveness and protocol-valid completion compared with single-shot, 2unstructured, and restart-based baselines. 4. Deployment-relevant diagnostics are pro-vided, including runtime, iteration count, qual-ity deltas, reward alignment, ablation sensitiv-ity, error statistics, recovery behavior, model-size effects, dataset-level effects, and quality– speed–cost trade-offs. 5. Reward-optimization and red-team stress tests show that process reward is useful but can-not replace hidden final evaluation, supporting GRACE-DS’s role as a pre-deployment safety and reliability harness. 

2 Related Work 

LLM-powered agents are increasingly evaluated in settings that require tool use, code execution, and iterative debugging. Data-science and ML work-flows are a particularly important case because suc-cess depends not only on generating executable code, but on following a valid modeling process: inspecting data, avoiding leakage, choosing appro-priate validation, etc. Existing benchmarks have made substantial progress on these capabilities, but most emphasize final task performance, code correctness, or broad research-style experimenta-tion. GRACE-DS targets a complementary prob-lem: pre-deployment evaluation of LLM-powered AutoML agents on production-like tabular work-flows, where organizations must verify both pre-dictive performance and process reliability before allowing agents to operate on internal data. 

2.1 Executable data-science benchmarks 

DS-1000 (Lai et al., 2023) and DS-Bench (Jing et al., 2025) evaluate code generation for real-istic data-science problems through executable tests, and DA-Code (Huang et al., 2024b) and DataSciBench (Zhang et al., 2025) extend this to broader agent-style data-wrangling, EDA, and ML prompts, including settings with uncertain or non-unique ground truth. These benchmarks made ex-ecutable data-science evaluation measurable and reproducible, but they do not capture the workflow structure of production ML — iteration through planning, EDA, feature engineering, validation, correction, and submission, or methodological fail-ures such as target leakage and non-reproducible artifacts. GRACE-DS builds on the executable-evaluation tradition but shifts the unit of evaluation from isolated code snippets to an interactive tabular ML workflow. 

2.2 Final-score and competition-style benchmarks 

MLE-bench (Chan et al., 2025) evaluates ML agents on a large set of Kaggle competitions and offers a realistic measure of broad ML engineer-ing ability. TML-Bench (Pinchuk, 2026) is closer to GRACE-DS in domain, focusing on tabular ML competitions with hidden-label private-holdout scoring. The limitation of final-score benchmarks is that they reveal little about how a score was obtained: a high score may rest on leakage, in-valid validation, or brittle code, while a low score may reflect poor error recovery rather than weak modeling knowledge. GRACE-DS retains hidden final-test evaluation but adds workflow-stage in-strumentation, hidden process validators, reward decomposition, error statistics, and replayability checks — less broad and less competition-oriented, but more diagnostic for enterprise pre-deployment decisions. 

2.3 Interactive ML environments 

MLE-Dojo (Qiang et al., 2026), MLGym (Nathani et al., 2025), DSGym (Nie et al., 2026), and MLA-gentBench (Huang et al., 2024a) treat ML work as a sequential process in executable environments, enabling the study of tool use, debugging, and improvement over time. GRACE-DS shares this motivation but makes a different trade-off : in-stead of maximizing task breadth, it narrows the scope to tabular supervised ML and governs the workflow explicitly, with named stages, evaluator-owned validation, hidden checklist validators, hard penalties for critical methodological errors, and a reproducible submission requirement. 

2.4 Data-analysis and scientific-discovery benchmarks 

InfiAgent-DABench (Hu et al., 2024), DABstep (Egg et al., 2025), and IDA-Bench (Li et al., 2025) make multi-step data analysis interactive and auto-matically scorable, while BLADE (Gu et al., 2024), ScienceAgentBench (Chen et al., 2025), Discov-eryBench (Majumder et al., 2025), InsightBench (Sahu et al., 2025), and ReplicationBench (Ye et al., 2026) evaluate broader scientific or analytic reason-ing, often via expert matching, planted insights, or LLM-as-judge components. GRACE-DS adopts the interaction idea but applies it to a different tar-get: rather than scoring answers to analytical ques-tions or discovery novelty, it measures whether the 3agent generates a stable predictor that generalizes to unseen data while satisfying the process con-straints, using low-cost, reproducible, and feasible tests. 

2.5 AutoML benchmarking and agent systems 

Classical AutoML benchmarking, notably the OpenML AutoML Benchmark (Gijsbers et al., 2024), established how to compare automated mod-eling systems across many tabular tasks with con-trolled splits and standardized metrics. GRACE-DS adds an LLM-agent interaction layer on top of this problem: the evaluated system is not a fixed modeling algorithm but an agent that writes code, interprets feedback, chooses workflow ac-tions, and must produce a reproducible final artifact under organizational constraints. Agent systems such as Data Interpreter (Hong et al., 2024) and AIDE (Jiang et al., 2025) are complementary: they propose ways to build stronger agents, whereas GRACE-DS provides an architecture-neutral envi-ronment in which any such agent — hierarchical, tree-search, ReAct-style, or a company-internal copilot — can be evaluated under the same hidden validation and scoring protocol. 

2.6 Positioning of GRACE-DS 

The works closest to GRACE-DS are MLE-bench, TML-Bench, MLE-Dojo, MLGym, and DSGym: the former two provide strong hidden-score evalua-tion but limited process-level diagnosis, while the latter three provide broad interactive environments without GRACE-DS’s specialization in the tabular supervised-learning workflow and its deployment-specific risks. GRACE-DS combines five proper-ties in a single platform: it is stage-aware (ex-plicit workflow stages from planning to final sub-mission), guarded (hidden validators for leakage, protocol validity, reproducibility, and submission validity), reward-guided (process-level feedback whose alignment with hidden-test quality can be analyzed), deployment-oriented (final submis-sions must replay on raw input records), and ex-perimentally controlled (the same tasks support single-shot, unstructured, iterative, restart, ablation, reward-optimization, and red-team comparisons). 

3 Proposed Approach 

3.1 Business objective, constraints, and acceptance criteria 

GRACE-DS is framed around a deployment con-tract. From raw tabular data the agent must produce a reproducible pipeline — a callable predict_fn 

or a fitted Pipeline — that reruns on raw rows. A solution is accepted only if it is (i) reproducible, (ii) protocol-valid (terminally scored once on the hidden test without a violation or a forced rescue), (iii) free of critical methodological errors (leak-age, train+validation refit, evaluator-private access, snapshot tampering, Section 3.4), and (iv) clears a trivial per-task baseline with measurable head-room toward an untuned strong oracle that anchors the normalized score. Table 13 (Appendix A) ties standard industrial constraints to the GRACE-DS mechanism that enforces each, so that “works in a notebook” is replaced by “passes a deployment-style contract”. 

3.2 Environment 

The environment is an evaluator-owned sandbox around one tabular task described by a manifest (dataset, target column, metric, split). The dataset is split into separate, non-overlapping train , valida-tion , and hidden-test splits. The agent’s namespace is deliberately restricted: training features with the target ( train_df ), validation features without the target ( valid_df ), protected raw snapshots (train_df_original /valid_df_original ), the target name, and pandas /numpy . Validation labels and all hidden-test data stay evaluator-private. The approved stack is fixed and announced with exact runtime versions ( pandas , numpy , scikit-learn ,

scipy , statsmodels ), only pd /np are pre-bound, every other class must be imported explicitly, and unlisted libraries are unavailable. The agent acts through a fixed vocabulary of eight states (Table 1): each response be-gins with exactly one ACTION: line, executable states carry exactly one fenced Python block, and 

VALIDATE /FINAL_SUBMIT are pure evaluator trig-gers that carry no code. Every state has an inde-pendent per-task budget (steps, wall-clock seconds, consecutive repeats), so no single stage can con-sume the whole episode. 3.3 Feedback 

The agent receives three kinds of feedback. Execu-tion feedback reports success/failure, the captured 

stdout , and on failure a traceback with targeted hints (e.g. missing imports, engineered columns left outside the pipeline, an unfitted pipeline). 

Stage-aware feedback is compiled from the dataset conditions actually present (missing values, cate-goricals, duplicates, correlations, outliers and skew, imbalance, identifier-like columns, . . . ) into a hid-den checklist : criteria are added only for conditions a task exhibits, so the rubric varies per task and can-not be memorized. Each step yields a stage score in [0 , 1] and, for every unresolved criterion, an ab-stract hint that says what is unresolved and which direction to move but never reveals the detector (its regexes, thresholds, or checks). Metric feedback 

is gated : the validation metric appears only after an explicit VALIDATE (or harness auto-validation), 

MODEL never discloses quality, and the hidden-test metric is computed once at terminal scoring. 

Why the rubric stays hidden Disclosing the ex-act rubric would let an agent pattern-match the checker — e.g. calling df.isna().sum() to tick the EDA missing-value box without ever reason-ing about whether those values matter or handling them inside a reproducible pipeline. Hiding it while surfacing only abstract hints pushes the agent to reason about the data. The reward reinforces this: checklist compliance feeds only the plan ( 0.15 )and code-quality ( 0.30 ) terms and cannot move 5the performance term ( 0.55 ), which the evaluator computes on held-out data. The process reward is therefore a proxy for task quality, not an inde-pendent target, under ordinary task-solving the two move together, but the adversarial regimes (Sec-tion 3.6) show that rewards cannot fully replace test metrics.

3.7 Experimental Protocol 

(i) The agent uses training labels and validation features, validation labels and the hidden test are evaluator-private; (ii) Each task is fully specified by a task.json manifest (target, type, metric, ref-erence scores, per-state budgets, split strategy), so an episode is reconstructible from manifest plus seeds; (iii) Metrics are higher-is-better, error met-rics (RMSE, MAE, log-loss) are returned negated ;

(iv) Randomness is fixed: split seeds, and LLM sampling seed = base + repeat where the provider honors it (Table 16); (v) Execution is bounded by per-block timeouts and per-episode action and to-ken budgets.

Interpretability (H4) Unlike final-score-only benchmarks, GRACE-DS explains why an episode scored as it did: the decomposed per-step reward, terminal-path breakdown (honest finalization vs auto-finalization rescue), validation-metric-growth trace, and candidate-diversity metrics make trajec-tories auditable. 

Calibration of the self-estimate Because vali-dation is gated and the hidden test is scored once, each episode yields a paired (validation, hidden-test) measurement on the same normalized scale. We report the signed validation →hidden-test gap and their per-(model , task ) rank correlation as a self-estimate diagnostic. We do not claim proba-bilistic calibration of the predictions (out of scope under ROC-AUC/RMSE, left to a log-loss variant the metric layer already supports). 

Cross-model, cost, robustness, and task interac-tion Reading the per-cell results along extra axes answers three further questions, each with its own table. Model scale (H8) : per-model breakdowns test whether larger models exploit the structured environment differently (e.g. recovering from fail-ures or using iteration more effectively). Quality-speed-cost (H9) : a dedicated cost table relates each cell’s hidden-test quality to its runtime, LLM calls, and token usage, placing regimes on an explicit frontier. Robustness : because prompts are para-phrased across episodes ( paraphrase_prompts ), a robustness table reports how stable each regime’s protocol-valid performance is under rewording.

4.1 Datasets

5.4 Reward as a progress signal (H5) H5 is confirmed with the anticipated caveat.

5.8 Quality-speed-cost frontier (H9) 

Table 11 places every regime on the quality-compute frontier using wall-clock time, LLM calls, and total episode tokens. H9 is confirmed. The frontier is well-separated and non-trivial: single-shot is the cheapest point (41 s, 1 call, 2.4k tokens) at the lowest quality (0.536), plain restarts buy

Notably, flexible iteration is also the fastest of the iterative regimes in mean wall-clock (101 s, against 120 s for the unstructured agent). The one axis on which structured iteration pays a pre-mium is tokens: stage-aware feedback and think-ing traces raise episode tokens to ≈43k, roughly 

2.1× the call-matched restart budget. The in-terface ablation quantifies the price of that ver-bosity: flexible_compact_feedback removes about 10% of the tokens (38.7k) and 0.47 calls for a −0.020 E2E Q quality concession, defining an intermediate operating point for token-constrained deployments. Organizations can therefore choose between three rational operating points — single-shot for triage, compact-feedback flexible iteration for token-bounded budgets, and full flexible itera-tion for maximum end-to-end reliability — while both restart baselines and the rigid schedule are dominated on this grid. For an organization instantiating GRACE-DS on its own data, the prac-tical reading is that the value of an iterative agent loop is predictable from task difficulty relative to the oracle anchor: the further a trivial single-shot solution sits from the oracle, the more the struc-tured environment pays. 

6 Conclusion 

We introduced GRACE-DS, a controlled pre-deployment evaluation harness for LLM-powered AutoML agents on tabular supervised-learning workflows, and used it to compare 15 harness regimes across eight frontier LLMs and ten production-like tasks. The central finding is that how an agent is allowed to work matters as much as which model runs it: flexible structured itera-tion reaches an end-to-end normalized hidden-test quality of 0.754 , against 0.536 for single-shot gen-eration, 0.527 for an unstructured agent, and 0.672 

and 0.686 for the restart baselines, while raising protocol-valid completion to 96 .9% . The paired advantage is +0 .218 E2E Q over single-shot and 

+0 .227 over the unstructured agent, and flexible iteration dominates even the most conservative call-matched restart upper bound on quality and wall-clock (Tables 4, 5, 11). Table 21 summarizes the verdicts with the pri-mary supporting evidence for each hypothesis. These results also substantiate the positioning claims of Section 2.6 against final-score and gym-style benchmarks. Every deployment-relevant dis-tinction reported above is invisible to a final-score-only evaluation: the unstructured agent and the fixed schedule essentially match flexible iteration on the observed quality of completed runs (Obs. Q 

0.762 –0.779 vs. 0.779 ) and would look competi-tive on a leaderboard, yet they lose 10 –23 points of protocol-valid end-to-end quality to terminal failures, the restart baselines look strong on ag-gregate score while committing 25 –34 leakage-class critical-error episodes, against at most one in the structured regimes, and the reward maximiz-ers and the red team keep their scores superficially respectable while the combination of hidden valida-tors and isolated held-out scoring localizes exactly what they sacrificed. It is precisely the GRACE-DS combination — staged interaction, hidden executable validators, decomposed process reward, reproducible sub-mission, and a single isolated hidden-test scor-ing — that turns these distinctions into measure-ments, which is the capability an enterprise pre-deployment gate requires and which the related benchmarks of Section 2 do not provide in one platform. By measuring the full agent loop rather than only its final score, GRACE-DS lets a team choose not just a model but a way of running it, and surfaces the reliability and reward-hacking failure modes that determine whether an AutoML agent is safe to put into production.
