## 摘要
人工智能智能体的性能不仅是一个建模问题，从根本上讲也是一个系统问题。模型的高级能力是通过智能体夹具（agent harness）实现的。因此，模型假设与夹具行为之间的差距很容易阻碍模型全部能力转化为智能体性能。我们将此形式化为“意图-执行”差距：模型意图与夹具执行之间的不匹配，反之亦然。我们认为，最小化这种意图-执行差距与夹具设计的其他方面（如工具和执行循环）同样重要。为了说明这种夹具-模型对齐的影响，我们开发了一个简单且可定制的夹具，称为“Simple Strands Agent”（SSA）。SSA旨在找出在不同模型系列（如Claude、Gemini、GPT、Grok、Qwen）中泛化的常见模式，以及少量模型特定的偏好。我们做出了两项贡献：（i）我们在流行的智能体基准测试（SWE-Pro、SWE-Verified和Terminal-Bench-2）上复现或改进了不同模型提供商系列报告的pass@1性能；（ii）基于对SSA生成的138k条轨迹的分析，我们超越了在前沿模型中往往相对均衡的pass@1数值。通过将智能体轨迹表示为代码状态空间，我们观察到问题求解行为中的模型级别差异。更细粒度的指标，如编辑频率、测试活动和阶段转换，揭示了单个模型在自主问题求解的不同阶段如何分配精力。

## Abstract
AI agent performance is not just a modeling problem, it is fundamentally a systems problem. The advanced capabilities of models are realized through agent harnesses. Therefore, a gap between model assumptions and harness behavior can easily prevent the model's full capabilities from translating into agent performance. We formalize this as the `intent-execution' gap: the mismatch between what the model intends and what the harness executes, and vice versa. We argue that minimizing this intent-execution gap is as important as other aspects of harness design such as tools and execution loops. To illustrate the impact of this harness-model alignment, we develop a simple and customizable harness called `Simple Strands Agent' (SSA). SSA aims to find the bulk of common patterns which generalize across different model families (such as Claude, Gemini, GPT, Grok, Qwen), as well as a small number of model-specific preferences. We make two contributions: (i) we reproduce or improve on the pass@1 performance reported by diverse model-provider families on popular agentic benchmarks (SWE-Pro, SWE-Verified and Terminal-Bench-2), and (ii) building on an analysis of 138k trajectories generated by SSA, we look beyond the pass@1 numbers which tend to be relatively even across frontier models. By representing agent trajectories in code state-spaces, we observe model-level differences in problem-solving behavior. Finer-grained metrics such as edit frequency, testing activity, and phase-transitions reveal how individual models allocate effort across different stages of autonomous problem solving.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- 论文指出，现代编码智能体的性能不仅取决于底层大语言模型（LLM）的原始能力，还严重依赖于**代理夹具（harness）**——即协调工具调用、管理反馈循环的软件层。如果模型的行为假设与夹具的实际执行之间存在不匹配，模型的全部能力就无法有效转化为智能体性能。
- 作者将这种不匹配形式化为**“意图-执行差距”（intent-execution gap）**：这是一个双向问题——一方面，模型意图执行某个操作，但夹具可能曲解或错误执行；另一方面，环境执行的结果可能没有被完整、真实地反馈给模型。
- 现有工作往往针对特定模型、特定基准进行定制化调优（如专有提示、特殊工具模式），但这种调优是脆弱的，随着模型更新可能失效。因此，论文的核心动机是：**寻找跨模型家族、跨基准的通用接口原则，而非仅仅追求模型特定的“小技巧”**，从而从根本上减少意图-执行差距。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：设计一个简单但可定制的代理夹具——**Simple Strands Agent (SSA)**。SSA 的目标是捕捉不同模型家族（Claude、GPT、Gemini、Grok、Qwen）之间的通用交互模式，同时允许少数模型特定的偏好适配，从而在最小化意图-执行差距的前提下，释放模型的真实能力。
- **关键技术细节**：
  - **反馈丰富的工具边界**：
    - 结构化执行反馈：所有 shell 调用返回命令、状态、退出码、输出和错误文本。长输出在中间截断但保留首尾，关键测试失败信息不丢失。
    - 安全的编辑语义：所有编辑以相邻代码行为锚点；拒绝部分行匹配或重复匹配，并返回精确的反馈帮助模型修正；成功编辑后返回统一差异（unified diff），减少模型验证其修改是否生效的需要。
  - **平衡推理与工具调用**：鼓励模型进行“**足够推理后立即验证**”。对于 Claude 变体，使用定量指导（如“工具调用次数应超过100次”）打破长推理链；对于 Gemini、Grok 等家族，使用柔性指令（如“尽可能多地使用工具”）。没有统一的“金提示”，而是根据模型倾向调整。
  - **对齐夹具与模型的工具使用偏好**：例如 GPT 模型偏好 `apply_patch` 风格的代码更新；Grok 模型难以处理行号前缀，因此禁用行号显示；不同模型对工具接口（ monolithic vs. 原子操作）有不同适应性。夹具应尊重这些由训练形成的偏好，而非强行统一。
- **轨迹质量分析框架**：提出 **“解决方案距离”（solution distance）** 度量，将智能体仓库状态映射到文本级代码状态空间 $X$，并定义一个召回式离散度 $D(t)$，度量当前代码状态与最近已知正确解决方案之间的距离。通过忠实回放所有编辑事件，可以追踪 $D(t)$ 随时间的变化，从而量化回溯（$\Delta D > 0$）等行为模式。

- **充分性**：
  - 每个模型在每个基准上运行 5 次，计算置信区间，统计可靠性较好。
  - 除了最终 pass@1 指标，还进行了详尽的细粒度 **轨迹分析**：编辑/测试比例、阶段（探索/定位/实现/验证）变迁、回溯（∆D > 0）概率、工具错误率等 9 大类指标，并跨模型、跨基准呈现（附录中提供了大量图表）。
  - 进行了 **消融实验**：包括 Git 历史泄漏对比（Base vs. Sanitized 容器）、推理提示强度对比、命令批处理影响等。
  - **客观性**：使用统一的 SSA 夹具，仅通过狭窄的家族适配器调整；评估协议严格（禁用互联网、独立评估容器等）。但需要注意，实验全部由同一家研究机构（AWS AI Labs）完成，可能存在软硬件环境偏差，但论文已开源代码和配置以提高可复现性。

### 6. 论文的主要结论与发现

1. **意图-执行差距是可显著缩小的**：通过 SSA 框架，所有测试模型的 pass@1 均达到或超越其官方发布数字，证明了当前许多代理性能损失源于夹具摩擦而非模型本身能力限制。
2. **不同模型家族的行为差异大于 pass@1 差异**：当多个模型 pass@1 相近时（例如 Opus 4.6 和 GPT-5.4 在 SWE-Verified 上均为 ≈80%），它们的轨迹风格（编辑/测试时间分配、阶段安排、回溯率）存在显著且稳定的差异。例如，Opus 4.6 早期大量进行测试，GPT-5.4 则逐渐增加测试活动。
3. **回溯与解决率解耦**：Gemini 3.1 Pro 具有最高的回溯率（14.3%），但 pass@1 同样最高（80.8%）；GPT-5.4 回溯率极低（1.2%），但 pass@1 也接近最高。回溯本身不意味着性能差，但反映了不同的探索风格。
4. **SWE-Bench-Pro 公共容器存在 Git 历史泄漏**：公开展示了未来提交信息，导致 pass@1 虚高。例如 Opus 4.6 在去污染后下降 6.87%，GPT-5.4 下降 3.32%。一些模型在明确指示查看 Git 历史后性能提升更大（Qwen3-Coder Next 提升 21.12%）。
5. **基础设施设置（如时间预算、批处理）对模型性能有非均匀影响**：Terminal-Bench-2 上放松时间/内存限制可将 pass@1 提高 5-10 个百分点；命令批处理对 Claude 模型无效（因释放更多推理），但对 Gemini 和 Grok 有帮助。

### 7. 优点

- **概念贡献**：正式定义了“意图-执行差距”，并系统论证了代理夹具设计对模型能力释放的关键作用。
- **实用贡献**：开发并开源了 SSA 框架，提供了可复现的配置和评估工具，社区可直接使用。
- **分析深度**：超越仅报告 pass@1，提出了“解决方案距离”等轨迹级度量，揭示了模型之间在问题解决策略上的稳定差异，为未来模型-夹具协同设计提供了新视角。
- **评估完整性**：覆盖 21 个模型、3 个基准、138k 条轨迹，并进行了多种消融和敏感性分析（Git 泄漏、推理提示、批处理、约束条件等），具有很强的说服力。
- **诚实性**：主动披露并量化基准容器中的 Git 历史泄漏问题，推荐使用去污染版本进行公平比较，体现了对基准评估完整性（evaluation integrity）的重视。

### 8. 不足与局限

- **模型覆盖范围**：虽然覆盖了 5 个家族，但未包括 DeepSeek、Mistral、Llama 等同样重要的开放模型，以及 MiniMax、Amazon Nova 等新兴模型。结论的泛化性可能仍有待验证。
- **开放权重模型性能差距大**：在 Terminal-Bench-2 上，Qwen 等开放模型与闭源模型差距巨大（如 Qwen3-Coder 30B 仅 17.07% vs. Opus 4.6 83.14%），这可能部分源于开放模型本身能力，但也可能与 SSA 的适配不足有关，论文未深入诊断。
- **解决方案距离度量的局限性**：基于文本重叠（recall-style），而不是语义等价性。一个正确但文字风格不同的修复可能被评分为距离远。虽然使用自锚点（self-anchor）和多个参考模式缓解，但根本局限仍在。
- **少量评估环境缺陷**：SWE-Bench-Verified 中约有 ≤0.41% 的实例因环境问题永远无法通过（如外部 URL 失效、设置不一致），这降低了绝对上限。
- **基础设施干扰**：Terminal-Bench-2 的约束设置下，后端的延迟、依赖下载时间等非智能体因素会消耗预算，可能混淆真实的模型性能。
- **单次运行方差**：虽然进行了 5 次重复，但有些行为指标（如回溯率）可能存在较高的随机性，尤其在小模型或困难实例上。
