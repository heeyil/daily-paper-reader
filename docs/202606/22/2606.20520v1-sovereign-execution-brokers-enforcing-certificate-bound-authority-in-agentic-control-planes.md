## 摘要
自主代理越来越多地连接到云、部署和数据控制工作流，但生产变更权限不应驻留在非确定性推理过程中。现有的访问控制机制对身份进行授权，而保证层对提议的操作进行认证；两者单独都无法在变更时刻提供认证权限的强制性执行点。本文引入了主权执行代理（SEB），这是一种针对证书绑定代理基础设施的运行时强制边界。SEB 消耗由主权保证边界（SAB）颁发的证书，验证请求的变更是否与认证的执行合同匹配，检查有效窗口、策略时期、撤销时期和实时状态漂移，生成作用域执行身份，调用基础设施 API，并记录签名的决策和结果记录。通过分离提议、准入和执行，SEB 将认证权限转化为短时、可撤销、可审计的运行时能力，前提是生产变更 API 拒绝非代理身份。我们介绍了 SEB 执行模型、证书和重放验证谓词、作用域身份语义、绕过防止部署模式、故障行为以及具体的原型实现。我们在 AWS 和 Kubernetes 集群上评估了该原型，测量了延迟开销、撤销传播、漂移检测以及在故障注入下的安全性。

## Abstract
Autonomous agents are increasingly connected to cloud, deployment, and data-control workflows, but production mutation authority should not reside inside non-deterministic reasoning processes. Existing access-control mechanisms authorize identities, while assurance layers certify proposed actions; neither alone provides a mandatory enforcement point for certified authority at the moment of mutation. This paper introduces the Sovereign Execution Broker (SEB), a runtime enforcement boundary for certificate-bound agentic infrastructure. SEB consumes certificates issued by the Sovereign Assurance Boundary (SAB), verifies that the requested mutation matches the certified execution contract, checks validity windows, policy epochs, revocation epochs, and live-state drift, mints scoped execution identity, invokes infrastructure APIs, and records signed decision and outcome records. By separating proposal, admission, and execution, SEB turns certified authority into a short-lived, revocable, auditable runtime capability, provided that production mutation APIs reject non-broker identities. We present the SEB execution model, certificate and replay-verification predicates, scoped identity semantics, bypass-prevention deployment patterns, failure behavior, and a concrete prototype implementation. We evaluate the prototype on AWS and Kubernetes clusters, measuring latency overheads, revocation propagation, drift detection, and security under fault injection.

## 1. 核心问题与整体含义
- **研究动机**：自主代理（如LLM驱动的Agent）越来越多地直接操作云基础设施，但非确定性推理过程（幻觉、提示注入）可能导致危险变更。现有访问控制机制（如IAM）只验证身份权限，而安全保证层（如SAB）只认证提议，两者都未在**执行时刻**提供强制验证点，导致“提议可绕过、凭据可滥用、状态可漂移”的重大安全漏洞。
- **整体含义**：需要一种运行时强制边界，将“谁可以做”与“当前操作是否仍安全”区分开来，使认证权限成为**短期、可撤销、可审计的运行时能力**。论文提出**Sovereign Execution Broker (SEB)**，作为证书绑定代理控制平面的强制执行节点。

## 2. 方法论
- **核心思想**：分离提议（Proposal）、准入（Admission）和执行（Execution）。SEB是唯一持有生产变更凭据的实体，所有自主变更必须通过SEB验证SAB颁发的证书（Ω）后，才动态铸造限作用凭据并执行。
- **关键技术细节**：
  - **形式化执行接口**：`Execute(Ω, req, St, Platform) → D | O`，其中Ω为证书，req为具体请求，St为当前基础设施状态，Platform描述目标平台能力。
  - **验证管道**（图2所示）：依次检查签名 (Φsig)、合约匹配 (Φmatch)、有效期 (Φtime)、策略时期 (Φpolicy)、撤销时期 (Φrev)、状态漂移 (Φdrift)、防重放 (Φreplay) 以及平台可作用性 (Scopeable)。仅当所有检查通过才铸造身份并调用API。
  - **限作用身份**：从身份提供者（AWS STS或K8s TokenRequest）获取**临时、短时、作用范围严格限制**的凭据，仅允许证书中指定的操作、资源、参数。对于IAM无法表达的参数级约束，通过Broker代理或准入webhook强制执行。
  - **防绕过部署模式**：目标平台必须拒绝非Broker主体的变更请求（如AWS SCP、K8s RBAC+webhook），确保Broker是唯一自主变更路径。
  - **状态漂移与撤销**：Broker实时查询当前状态并与证书记录的准入状态比较，若距离超过容忍度则拒绝；使用全局撤销时期计数器，若当前时期大于证书内时期则拒绝。
  - **防重放**：每个证书具有唯一nonce，Broker通过数据库原子预留确保nonce只使用一次。

## 6. 主要结论与发现
- SEB增加了可接受的延迟：Kubernetes broker-only约28.2ms（p50），AWS约136.9ms（p50）。
- 撤销传播延迟最大5.2秒（均值2.6秒），由5秒轮询间隔主导。
- 漂移检测成功识别所有模拟状态变更（严格阈值下拒绝率100%）。
- 吞吐量：K8s TokenRequest可线性扩展至820 req/s，AWS STS受限于云API节流（峰值240 req/s）。
- 安全测试中所有攻击向量（未认证、重放、漂移、参数违规等）均被100%拒绝。
- 崩溃恢复中：API调用前失败均可安全停止，调用后的未决事务可通过幂等令牌正确解析，未出现重复变更。
- **核心贡献**：SEB将证书绑定权威转化为实际可执行的运行时强制，只要生产API拒绝非Broker身份，即可实现零常驻凭据+完整中介+动态约束验证。

- **验证充分**：进行了多维度性能和安全测试，消融实验揭示瓶颈来源（漂移查询和凭据铸造占主导）。

## 8. 不足与局限
- **信任依赖**：依赖云IAM/RBAC的正确性、SAB签名密钥安全、Broker运行环境安全。若这些被攻破，安全性失效。
- **性能敏感性**：漂移查询需要实时调用云API，成为主要延迟源（占AWS工作负载62.3%）。对于微秒级控制循环不适用。
- **冷启动开销**：每次变更都需铸造新凭据，无法复用，导致高延迟。
- **单点故障**：Broker是所有自主变更的必经路径，其宕机会阻塞所有Agent操作，需要高可用部署增加复杂度。
- **状态观察陈旧**：云API是最终一致性模型，漂移检查可能读到过时状态，存在TOCTOU窗口。
- **跨云/多账户复杂性**：统一信任根、全局撤销时期、一致Proxy部署在多云环境非常困难。
- **未覆盖场景**：语义安全性仍由SAB负责，若SAB被攻破（例如接受危险提议），SEB无法拦截。实验未在真实生产多租户或混合云环境中进行，可能遗漏现实问题。
