## 摘要
AI智能体越来越多地通过框架在生产环境中运行，即大语言模型周围的软件，其中包括一个强制执行安全和有效性策略的引擎，例如“在提交之前运行测试”。执行这些策略需要弥合语义鸿沟：策略意图以未充分指定的自然语言表达，而执行必须作用于具体的系统操作，例如运行哪个测试。许多策略还定义了事件顺序或数据流操作。然而现有方法存在不足。工具调用护栏会遗漏绕过工具层的系统操作，而操作系统沙箱控制的是资源访问而非操作，返回的模糊错误会迷惑智能体。我们的关键洞察是策略上下文位于最接近任务的智能体内，而执行必须在操作系统层面进行以覆盖所有执行路径。我们引入了ActPlane，一个策略引擎，允许智能体声明策略并在操作系统内核中通过语义反馈和隔离强制执行。ActPlane使用一个简单的信息流控制（IFC）DSL来支持跨事件策略。我们使用eBPF实现了ActPlane，并在来自实证研究、编程任务基准和安全基准的策略上进行了评估。ActPlane提高了策略合规性，包括在工具调用拦截无法观察到的间接执行路径上，开销为1.9%-8.4%。ActPlane位于https://github.com/eunomia-bpf/ActPlane

## Abstract
AI agents increasingly run in production through harnesses, the software around the LLM, including an engine that enforces safety and effectiveness policies, e.g., 'run tests before committing.' Enforcing these policies requires bridging a semantic gap: policy intent is expressed in underspecified natural language, while enforcement must act on concrete system actions, e.g., which test to run. Many policies also define event ordering or data flow actions. Yet existing approaches fall short. Tool-call guardrails miss system actions that bypass the tool layer, while OS sandboxes control resource access instead of actions, returning opaque errors that confuse the agent. Our key insight is that policy context lives within the agent closest to the task, while enforcement must happen at the OS to cover all execution paths. We introduce ActPlane, a policy engine that lets agents declare policies and enforces them in the OS kernel with semantic feedback and isolation. ActPlane uses a simple information-flow control (IFC) DSL to support cross-event policies. We implement ActPlane with eBPF and evaluate it on policies from the empirical study, coding-task benchmarks, and safety benchmarks. ActPlane improves policy compliance, including on indirect execution paths that tool-call interception cannot observe, with 1.9%-8.4% overhead. ActPlane is at https://github.com/eunomia-bpf/ActPlane

#### 1. 论文的核心问题与整体含义（研究动机和背景）

*   **核心问题**：AI智能体在运行时需要执行一系列安全与有效性策略（如“提交前必须运行测试”），但存在一个根本的**语义鸿沟**：策略意图通常以模糊的自然语言表达，而执行必须作用于具体的系统操作（如“运行哪个测试命令”）。现有方法要么无法覆盖所有执行路径（工具调用拦截会遗漏子进程、脚本等间接路径），要么过于底层（OS沙箱控制资源访问而非操作，并返回模糊错误）。
*   **研究背景**：AI智能体日益复杂，其行为可能因规划错误、提示注入攻击或工具/脚本副作用而偏离预期。因此，需要一个确定性的策略引擎来强制保证合规性，而不仅仅是依赖LLM的概率性遵守。
*   **研究动机**：为了弥合意图与执行之间的鸿沟，需要一个策略引擎，既能利用最接近任务的智能体所拥有的上下文（如仓库结构、当前任务），又能在操作系统底层强制执行，以确保覆盖所有执行路径。

#### 2. 论文提出的方法论：核心思想、关键技术细节

*   **核心思想**：**由智能体编写策略，在操作系统内核强制执行**。将策略上下文（最接近任务的智能体）与执行点（OS内核，覆盖所有路径）分离。智能体编写一个高级的、可编译的DSL策略，该策略被编译为eBPF程序，在内核中确定性地执行和检查。
*   **关键技术细节**：
    1.  **可编程策略DSL**：提供一种简单但表达能力强的领域特定语言（DSL），让智能体可以声明策略。策略模型采用**源到目标约束**，包含源（进程、文件、网络端点）、目标操作（exec, open, write, connect等）、效果（`block`/`kill` 禁止，`notify` 提示）、可选的时序门和原因。
    2.  **信息流控制（IFC）引擎**：为了支持跨事件策略（如“运行测试后提交”），ActPlane在OS内核层实现了一个轻量级的IFC引擎。
        *   它通过**标签**跟踪信息流，为每个进程、文件和网络端点附加标签。
        *   标签沿着数据流边（fork, exec, read, write, connect）单调地传播。
        *   引擎维护状态机，当事件发生时，检查主体和客体上的标签是否满足规则条件，从而在不重建历史日志的情况下判断策略。
    3.  **eBPF内核执行**：DSL规则被编译成eBPF程序，挂载到内核的关键钩子（exec, open, write, connect等）上。这实现了低开销的、确定性的策略检查，并且能覆盖子进程和shell等工具调用层无法观察到的间接路径。
    4.  **分层策略域与隔离**：引入分层域（hierarchical policy domains），组织成进程树边界。子域继承父域的所有规则，并可以增加本地规则，但**绝不能削弱继承的约束**。这确保了安全约束（来自用户、平台或父智能体）不会被智能体自身编写的规则所破坏，实现了安全的运行时适应。
    5.  **语义反馈**：当策略违反时，引擎不是返回模糊错误（如EPERM），而是返回具体的、人类的可读的语义反馈，如“提交被阻止：没有先运行测试。请先运行 `npm test`。”这帮助智能体理解违规原因并自我纠正。

*   **策略生成更可行**：使用LLM将607个自然语言策略翻译为DSL规则，成本低（约$0.028/条），且覆盖了大部分DSL特性，证明了DSL的可生成性。
*   **显著提升策略合规性**：ActPlane在决策符合率（DCR）上达到了**75.8%**，比所有基线（Prompt-filter: 48.4%, Tool-regex: 45.3%, FIDES: 48.9%）高出22-31个百分点。其优势主要在**间接路径**（脚本、隐藏执行）和**跨事件规则**上，这些是工具级方法无法处理的。

*   **提升安全性**：在OpenAgentSafety基准上，通过加载从任务描述自动生成的策略，ActPlane预防了**74%** 的默认不安全行为。

*   **方法创新性强**：提出将IFC、eBPF和策略DSL结合，由智能体在OS内核层编写和执行策略，巧妙解决了现有系统在路径覆盖和反馈方面的核心不足。

#### 8. 不足与局限

*   **适用场景有限**：系统设计目标明确，专注于OS级可观察的策略（per-event和cross-event），这仅占总策略的45%。内容级政策（需检查文件内容）和纯语义政策不在考虑范围内，存在固有局限性。
