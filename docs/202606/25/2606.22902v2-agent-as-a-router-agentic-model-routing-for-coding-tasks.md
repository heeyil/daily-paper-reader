## 摘要
现实世界中的用户通常可以访问来自不同提供商的多个大型语言模型（LLM），这些LLM往往在不同领域表现出色，但没有任何一个模型在所有领域都占据主导地位。因此，将每个任务路由到最合适的模型对于性能和成本都至关重要。现有的路由器将其视为静态的一次性分类问题。然而，我们识别出这些路由器的性能瓶颈在于信息缺失：仅通过在任务维度级别上为普通LLM路由器增加性能统计信息，就能获得15.3%的相对提升，超越了基于相同维度级别先验的启发式路由器。受此发现启发，我们提出了Agent-as-a-Router框架，该框架将路由形式化为C-A-F循环（上下文->动作->反馈->上下文）。它通过在部署过程中积累基于执行的经验来弥合信息差距。我们将该框架实例化为ACRouter，由编排器、验证器、记忆模块组成，并引入了CodeRouterBench，这是一个评估环境，包含约1万个任务实例以及来自8个前沿LLM的验证分数，支持在流式任务上进行基于遗憾值的路由器比较。实验表明，ACRouter在分布内任务上取得了最低的累积遗憾值，并泛化到分布外的智能体编程任务，证明我们的路由框架能够主动弥合信息差距。代码和基准数据集已发布于https://github.com/LanceZPF/agent-as-a-router。

## Abstract
Real-world users typically have access to multiple Large Language Models (LLMs) from different providers, and these LLMs often excel at distinct domains, yet none dominate all. Consequently, routing each task to the most suitable model becomes critical for both performance and cost. Existing routers treat this as a static, one-off classification problem. However, we identify the performance bottleneck for these routers as information deficit: simply augmenting a vanilla LLM router with performance statistics at the task-dimension level yields a 15.3% relative gain, surpassing a heuristic router built on the same dimension-level priors. Motivated by this finding, we propose Agent-as-a-Router, a framework that formalizes routing as a C-A-F loop (Context->Action->Feedback->Context). It closes the information gap by accumulating execution-grounded experience during deployment. We instantiate this framework as ACRouter, composed of an Orchestrator, a Verifier, a Memory module, and introduce CodeRouterBench, an evaluation environment comprising ~10K task instances with verified scores from 8 frontier LLMs, enabling regret-based router comparison on streaming tasks. Experiments show that ACRouter achieves the lowest cumulative regret on in-distribution tasks and generalizes to out-of-distribution agentic-programming tasks, demonstrating that our routing framework actively closes the information gap. Codes and benchmarks are released at https://github.com/LanceZPF/agent-as-a-router.

- **瓶颈诊断**：初步实验（表1）表明，仅向 LLM 路由器提供任务维度的性能统计数据（+Perf stats），即可将平均性能从 41.41% 提升至 47.74%（相对提升 15.3%），甚至超过基于相同先验的启发式方法 DimensionBest（47.50%）。这证明**信息缺失而非推理能力不足是性能瓶颈**。
- **整体目标**：提出一种能动态积累执行经验、主动弥合信息缺口的路由框架，实现任务-模型的最优匹匹配。

## 2. 提出的方法论

### 2.1 核心思想：C-A-F 循环（Context - Action - Feedback - Context）

将路由形式化为一个迭代的上下文多臂赌博机问题：
- **Context c_i**：当前任务输入、元数据（维度、难度、语言）以及从之前循环累积的 Memory 状态 H_<i。
- **Action a_i**：从模型池 M 中选择一个索引（即选择哪个 LLM 来执行任务）。
- **Feedback f_i**：执行选定模型后，Verifier 模块返回的分数（s_i）和成本（κ_i）。
- **Memorize**：将本次反馈写入 Memory 模块，更新上下文用于下一任务。

$$
c_i \xrightarrow{\text{Decide}} a_i \xrightarrow{\text{Execute}} f_i \xrightarrow{\text{Memorize}} c_{i+1}
$$

- **奖惩函数**：$r_i(a_i) = \epsilon_1 s_i(a_i) + \epsilon_2 \kappa_i(a_i)$，其中 $\epsilon_1 > 0$ 鼓励性能，$\epsilon_2 < 0$ 惩罚成本。
- **累积遗憾（Cumulative Regret）**：$CumReg_N(\pi) = \sum_{i=1}^N \delta_i = N(V^* - V(\pi))$，其中 $V^*$ 是每任务最优模型（per-task oracle）的平均奖励。

### 2.2 实例化：ACRouter 系统架构

- **Orchestrator（编排器）**：基于动态上下文进行路由决策。融合三种信号：
  - DimensionBest 先验（冻结的维度级最佳模型映射）。
  - 从 Memory 中检索出当前任务最相似的 top-10 历史邻居（通过余弦 kNN）。
  - 任务元数据（描述、难度、语言）。
  - 核心策略为微调后的 Qwen3.5-0.8B 模型，结合启发式规则加权投票。
- **Verifier（验证器）**：在沙箱（Docker）中执行模型输出，或使用 AST 解析、LLM-as-Judge 等工具，根据任务类型将多种信号加权融合为统一分数 $u_i \in [0,1]$。
- **Memory（记忆模块）**：在线向量存储（FIFO，最多 20K 条目），以任务嵌入（voyage-code-3 或 BGE-large）为键，存储历史选择、性能、成本和验证追踪。

### 2.3 可分解的路由策略分类

根据 C-A-F 循环中组件的开启/关闭，可将所有路由方法归为：
- **Single-Model**：固定使用一个模型（无 Orchestrator、Memory、Verifier）。
- **Static: Heuristic**：冻结的先验（如 DimensionBest、kNN Retrieval），无在线反馈。
- **Static: Trained Policy**：学习到的分类器（LogReg、RouteLLM、微调 Qwen3.5 等），使用静态训练数据。
- **Dynamic: Online Bandit**：在线更新参数的后验（LinUCB、LinTS），仅使用奖励信号。
- **Agent-as-a-Router（完整）**：ACRouter，同时激活所有三模块。

## 6. 主要结论与发现

1. **信息缺失是路由瓶颈**：仅增加维度的性能统计数据即可使 LLM 路由器性能相对提升 15.3%，超过基于相同先验的启发式方法。
2. **无单一模型独强**：8 个模型在 9 个编码维度上各有优势（如 GLM-5 在算法设计上领先 Opus 86%，Qwen3-Max 在测试生成上领先 111%）。
3. **ACRouter 性能最佳**：在分布内测试中 AvgPerf 达 49.98%，CumReg 205.5，优于所有对比方法；在 OOD 上 AvgPerf 62.50%，CumReg 17.0，领先 LinUCB 和 Always-Opus。
4. **静态学习器 OOD 泛化差**：LogReg、RouteLLM 等在分布内性能尚可，但 OOD 下 AvgPerf 骤降至 8.93%~21.43%，低于随机选择（31.25%）。
5. **ACRouter 成本效率优于高成本单模型**：Perf/$ 在分布内 3.79、OOD 1.18，均高于 Always-Opus（1.29/0.64）。

## 8. 不足与局限

- **成本估计简化**：无法观察供应商缓存命中率，成本基于公开 token 价格和实测 token 用量，仅作为相对参考。
