## 摘要
实际用户通常可以访问来自不同提供商的多个大型语言模型（LLM），这些LLM往往在不同领域表现出色，但没有一个能在所有领域占据主导地位。因此，将每个任务路由到最合适的模型对于性能和成本都至关重要。现有的路由器将此视为一个静态的、一次性分类问题。然而，我们识别出这些路由器的性能瓶颈在于信息不足：仅在任务维度层面用性能统计数据增强一个普通的LLM路由器，就能获得15.3%的相对提升，超越了基于相同维度层面先验的启发式路由器。受此发现启发，我们提出了“代理即路由器”框架，该框架将路由形式化为一个C-A-F循环（上下文->行动->反馈->上下文）。它通过在部署过程中积累基于执行的经验来缩小信息差距。我们将此框架实例化为ACRouter，由编排器、验证器、记忆模块组成，并引入了CodeRouterBench，一个包含约10K个任务实例的评估环境，这些实例来自8个前沿LLM的已验证评分，支持基于遗憾的流式任务路由器比较。实验表明，ACRouter在分布内任务上实现了最低的累积遗憾，并能泛化到分布外的代理编程任务，证明我们的路由框架主动缩小了信息差距。代码和基准测试已在https://github.com/LanceZPF/agent-as-a-router发布。

## Abstract
Real-world users typically have access to multiple Large Language Models (LLMs) from different providers, and these LLMs often excel at distinct domains, yet none dominate all. Consequently, routing each task to the most suitable model becomes critical for both performance and cost. Existing routers treat this as a static, one-off classification problem. However, we identify the performance bottleneck for these routers as information deficit: simply augmenting a vanilla LLM router with performance statistics at the task-dimension level yields a 15.3% relative gain, surpassing a heuristic router built on the same dimension-level priors. Motivated by this finding, we propose Agent-as-a-Router, a framework that formalizes routing as a C-A-F loop (Context->Action->Feedback->Context). It closes the information gap by accumulating execution-grounded experience during deployment. We instantiate this framework as ACRouter, composed of an Orchestrator, a Verifier, a Memory module, and introduce CodeRouterBench, an evaluation environment comprising ~10K task instances with verified scores from 8 frontier LLMs, enabling regret-based router comparison on streaming tasks. Experiments show that ACRouter achieves the lowest cumulative regret on in-distribution tasks and generalizes to out-of-distribution agentic-programming tasks, demonstrating that our routing framework actively closes the information gap. Codes and benchmarks are released at https://github.com/LanceZPF/agent-as-a-router.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想：Agent-as-a-Router 与 C-A-F 循环

- 将路由建模为**上下文-行动-反馈循环**：  
  $$c_i \xrightarrow{\text{Decide}} a_i \xrightarrow{\text{Execute}} f_i \xrightarrow{\text{Memorize}} c_{i+1}$$
  - 上下文 $c_i$：包含当前任务提示 $p_i$、元数据 $d_i$、历史记忆 $\mathcal{H}_{<i}$。
  - 行动 $a_i$：从模型池 $\mathcal{M}$ 中选择一个模型。
  - 反馈 $f_i$：包括验证器给出的分数 $\hat{s}_i$ 和货币成本 $\hat{\kappa}_i$。
  - 记忆模块将反馈合并到下一上下文中。

- 该形式化等价于**上下文多臂老虎机**（contextual multi-armed bandit），以**累积遗憾**（cumulative regret）作为自然流式评价指标。

### 2.2 关键技术细节：ACRouter 实例化

- **编排器**：基于动态上下文（DimensionBest 先验、来自记忆的 kNN 前10邻居、任务元数据）做出路由决策。核心策略是微调的 Qwen3.5-0.8B 模型与启发式规则的加权投票。
- **验证器**：在沙盒中执行模型输出，集成多种信号（AST 解析、代码执行、LLM-as-Judge 等）生成统一分数 $u_i \in [0,1]$。
- **记忆模块**：在线向量存储，以任务嵌入（voyage-code-3 / BGE-large）为键，存储所选模型、性能、成本及验证跟踪。使用余弦 kNN 检索前10邻居，FIFO 上限 20K 条目。
- **工具层**：包括候选模型池（8个前沿 LLM）、路由工具、评估工具、嵌入编码器和沙盒基础设施。

### 2.3 奖励定义与遗憾

- 每任务奖励：$r_i(a_i) = \epsilon_1 s_i(a_i) + \epsilon_2 \kappa_i(a_i)$，文中取 $\epsilon_1=1, \epsilon_2=-0.1$。
- 累计遗憾：$\text{CumReg}_N(\pi) = \sum_{i=1}^N \left( \max_j R_{ij} - R_{i,a_i} \right)$，其中 $R_{ij}$ 是模型 $j$ 在任务 $i$ 上的成本感知奖励。

## 6. 论文的主要结论与发现

1. **信息缺失是 LLM 路由器的核心瓶颈**（非推理能力）：提供任务维度性能统计即可提升 15.3%。
2. **没有单一模型在所有编码维度上占优**：如 GLM-5 在算法设计上领先，Qwen3-Max 在测试生成上领先，Kimi-K2.5 在数据科学上更具性价比。
3. **ACRouter 在分布内和分布外任务上均达到最优性能**，且成本低于始终选择最强模型（Always-Opus 4.6）。
4. **静态策略（如 LogReg、RouteLLM）在分布内性能尚可，但在分布外任务上严重退化**（甚至低于随机），而 ACRouter 通过记忆积累能够适应新任务类型。
5. **主动关闭信息差距**（通过执行反馈）是构建代理系统时通用原则。
