## 摘要
基于多模态大模型的移动GUI代理在短时任务上取得了显著进展，但在需要跨多个步骤和应用程序转换时保留中间事实的长时任务上仍不可靠。我们将这一限制归因于ReAct风格的提示方式，该方式被动地累积每一步的记录，导致提示膨胀和关键跨应用事实的稀释。为解决此问题，我们提出MemGUI-Agent，一种具有主动上下文管理的端到端长时移动GUI代理。MemGUI-Agent构建于上下文即动作（ConAct）之上，该机制将上下文管理视为与选择UI动作的策略相同的顶层动作。与被动追加历史记录不同，ConAct维护三个结构化上下文字段：折叠的动作历史、折叠的UI状态以及最近步骤记录，从而在保持上下文紧凑的同时保留关键UI事实。为了使主动上下文管理跨模型规模可学习，我们构建了MemGUI-3K，一个包含2,956条轨迹的数据集，带有完整的ConAct注释，用于监督训练和离线分析。在MemGUI-3K上训练一个8B模型，得到MemGUI-8B-SFT，这是一个在MemGUI-Bench上达到最佳开放数据8B性能，并能泛化到分布外MobileWorld基准的8B MemGUI-Agent。代码、数据和训练好的模型将在https://memgui-agent.github.io/上发布。

## Abstract
MLLM-based mobile GUI agents have made substantial progress on short-horizon tasks, yet remain unreliable on long-horizon tasks that require retaining intermediate facts across many steps and app transitions. We attribute this limitation to ReAct-style prompting, which passively accumulates per-step records, leading to prompt explosion and dilution of critical cross-app facts. To address this, we introduce MemGUI-Agent, an end-to-end long-horizon mobile GUI agent with proactive context management. MemGUI-Agent is built on Context-as-Action (ConAct), which casts context management as first-class actions emitted by the same policy that selects UI actions. Instead of passively appending history, ConAct maintains three structured context fields: folded action history, folded UI state, and recent step record, preserving critical UI facts while keeping context compact. To make proactive context management learnable across model scales, we construct MemGUI-3K, a 2,956-trajectory dataset with full ConAct annotations for supervised training and offline analysis. Training an 8B model on MemGUI-3K produces MemGUI-8B-SFT, an 8B MemGUI-Agent that achieves the best open-data 8B performance on MemGUI-Bench and generalizes to the out-of-distribution MobileWorld benchmark. Code, data, and trained models will be released at https://memgui-agent.github.io/.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **问题**：现有的多模态大模型（MLLM）驱动的移动GUI代理在短时任务上表现良好，但在长时任务（通常超过20步，需跨应用保留中间事实）中严重退化。例如，GUI-Owl-1.5-8B在AndroidWorld（平均8.4步）上达71.6%，在MobileWorld（平均27.8步）上降至38.2%，在MemGUI-Bench（平均36.2步）上仅11.7%。
- **根本原因**：主流的ReAct式提示策略被动地逐条追加历史记录，导致“提示膨胀”（prompt explosion）和关键跨应用事实（如价格、电话号码、复制文本）被稀释或丢失。
- **核心需求**：长时移动GUI控制需要紧凑的工作上下文和跨屏幕、步骤、应用转换的持久化UI事实。现有方法（外部记忆、基于规则的截断）均无法同时满足这两点。

### 2. 论文提出的方法论

- **核心思想**：提出**上下文即动作（Context-as-Action, ConAct）**，将上下文管理（历史折叠、UI记忆、自描述输出）作为与UI选择同策略的一等动作，由同一个策略在单次前向传播中联合决策。
- **关键技术细节**：
  - **结构化上下文**：由三个字段组成：折叠动作历史 \(H_t\)（压缩轨迹摘要）、折叠UI状态 \(M_t\)（持久化UI事实）、最近步骤记录 \(L_t\)（最近交互细节）。
  - **单步执行流程**：在每个步骤 \(t\)，MLLM策略 \(\pi_\theta\) 接收 \((I_t, S_t)\)，输出五部分 \(y_t = (\tau_t, \phi_t, a_t, o_t, \iota_t)\)，分别对应：推理、折叠指令（指定要压缩的步骤范围和摘要）、UI或记忆动作、UI观测描述、动作意图。

### 6. 论文的主要结论与发现

- **核心发现**：ConAct通过将上下文管理作为策略内置动作，显著缓解了长时任务中的提示膨胀和关键事实丢失。
  - 零样本MemGUI-Agent-235B在MemGUI-Bench上达到37.5% Pass@1和62.5% Pass@3，超越最好的代理框架M3A（32.8%/47.7%），并在MobileWorld上取得29.1% SR（+14.6）。
  - 微调后的MemGUI-8B-SFT在MemGUI-Bench上达23.4% Pass@1（+14.0），在MobileWorld上达17.9% SR（+8.5），为开放数据8B模型SOTA，且域外泛化良好。
- **分解洞察**：
  - 三个组件互补：UI记忆动作可持久化精确事实（Pass@1从5.0%升至17.5%）；历史折叠控制上下文增长；自描述步骤为两者提供基础。三者结合（40.0% Pass@1）远优于任一单独组件。
  - 最大的离线技能提升来自记忆触发：8B模型触发F1从19.9%跃升至48.0%，表明模型学会了在需要时将瞬态观测提升为持久记忆。
- **错误分析**：完整ConAct将总失败次数减少41%（99→58），主要降低过程幻觉（-42%）和输出幻觉（-57%），但知识不足和意图理解错误无显著改善，说明剩余瓶颈在领域知识和意图理解能力。

### 7. 优点

- **方法创新**：将上下文管理端到端化并作为一等动作嵌入策略，避免了外部记忆模块带来的系统复杂性和延迟，同时保留了关键事实的精确性。
- **数据集贡献**：MemGUI-3K是目前最长的移动GUI SFT数据集（平均28.8步），且包含细致的步骤合理性标注、记忆操作和折叠粒度信息，支持离线细粒度技能评估。
- **实验严谨性**：消融实验清晰分解了三个组件的贡献；离线分析隔离了上下文固定下的技能学习效果；域外泛化验证了方法的鲁棒性。
- **零样本有效性**：即使不进行微调，仅通过提示格式更改（ConAct）即在235B模型上取得显著提升，表明协议设计具有启发性且易于推广。
- **效率**：ConAct在150步后比ReAct节省约1.5k输入token，降低了长程推理的计算开销。

### 8. 不足与局限

- **环境覆盖**：实验仅限于Android风格移动GUI环境，未扩展到iOS、桌面或Web界面。交叉平台泛化性未知。
- **模型规模依赖**：零样本ConAct仅在最大骨干（235B-Thinking）上有效；较小模型（2B~8B）需要监督微调才能获益，且微调后性能仍落后于大模型。表明主动上下文管理需要较强的推理基础。
- **残余错误**：知识不足和意图理解错误未显著减少，说明方法无法弥补模型本身的知识短板或对用户意图的深层理解。
- **离线评估局限**：离线技能分析固定了输入状态，未模拟模型自主决策导致的上下文状态漂移，可能高估或低估真实行为。
- **数据收集偏差**：教师轨迹来自单一模型（Qwen3-VL-235B-Thinking），可能存在策略偏差；记忆操作频率较低（平均每个轨迹1.17次add，0.03次update，0.04次delete），对少见的更新/删除操作训练不充分。
- **计算资源**：训练8B模型需8块80GB GPU约9小时，对资源有限的研究者门槛仍较高。
