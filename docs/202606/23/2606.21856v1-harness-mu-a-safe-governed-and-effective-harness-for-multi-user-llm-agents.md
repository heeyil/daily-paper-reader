## 摘要
随着大语言模型（LLM）代理在协作工作流中的日益部署，需要强大的多用户、多主体交互机制来执行访问权限、解决权限冲突并防止未授权数据泄露。然而，当代LLM的单用户训练范式与多主体治理所需的硬约束之间存在根本性不匹配，使得基于概率的提示工程防护在多轮对抗交互中脆弱不堪。我们的关键洞察是：治理约束——谁被授权、什么被限制、谁的指令优先——是确定性的运行时变量，应由执行钩子强制执行，而非委托给LLM。本文提出Harness-MU，首个与模型无关、零微调的多用户LLM代理基础设施框架。通过将语言生成与安全编排解耦，Harness-MU在最大化合规需求满足的同时，保证了不可打破的权限边界。在Muses-Bench基准测试中，针对四种前沿开源与闭源模型，Harness-MU在所有访问控制攻击下实现了隐私保护目标，效用评分比标准基线高出0.28–0.39，指令遵循准确率提升高达48.9个百分点。Harness-MU推进了“约束工程”（Harness Engineering）理念，证明系统化基础设施对于解决LLM多主体治理挑战至关重要。代码与数据见https://github.com/YuanJrShiuan/Harness-MulUser。

## Abstract
The increasing deployment of large language model (LLM) agents in collaborative workflows demands robust multi-user, multi-principal interaction mechanisms capable of enforcing access permissions, resolving authoritative conflicts, and preventing unauthorized data disclosure. However, a fundamental mismatch exists between the single-user training paradigm of contemporary LLMs and the hard constraints required for multi-principal governance, rendering probabilistic, prompt-based safeguards vulnerable under multi-turn adversarial interactions.Our key insight is that governance constraints -- who is authorized, what is restricted, and whose instructions take precedence -- are deterministic runtime variables that should be enforced by execution hooks rather than entrusted to the LLM. We present \textbf{Harness-MU}, the first model-agnostic, zero-tuning infrastructure framework for multi-user LLM agents. By decoupling language generation from safety orchestration, Harness-MU guarantees unbreakable permission boundaries while maximizing compliant demand satisfaction. Across four frontier open-weight and proprietary models on the \textit{Muses-Bench} benchmark, Harness-MU achieves the goal of privacy preservation across all access-control attacks, outperforming the standard baseline by 0.28--0.39 in utility score and improving instruction-following accuracy by up to 48.9 percentage points. Harness-MU advances the philosophy of \textit{Harness Engineering}, establishing that systematic infrastructure is essential for solving LLM multi-principal governance challenges. The code and data are available at https://github.com/YuanJrShiuan/Harness-MulUser.

### 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：治理约束（谁被授权、什么受限制、谁的指令优先）是**确定性运行时变量**，应由执行钩子强制实施，而非委托给LLM。通过解耦语言生成与安全编排，实现模型无关的硬约束执行。
- **关键技术细节**：四个解耦运行时模块，完整实例化六组件约束框架（E, T, C, S, L, V）：
  - **Gatekeeper（准入控制层）**：在模型调用前合成非概率准入控制向量，基于声明的组织策略查询状态依赖语义权限算子Auth，生成不可变的权限记录`pk = ⟨uk, αk, Ak, Rk⟩`，不参与语言生成。
  - **Mediator（策略编排层）**：将多主体目标分解为局部执行向量，计算每个用户的执行计划`πk = ⟨wk(α), G⋆ ∩ Gk, Ωk⟩`，不直接与LLM通信，而是将策略注入下游Worker。
  - **Worker Collective（隔离生成层）**：并行实例化单主体执行环境，每个Worker仅接收隔离的上下文转录`Γk`，确保用户间无历史污染，阻断注入攻击面。
  - **ComplianceChecker（可行域投影层）**：在生成后作为确定性非线性投影算子`PAsafe`，拦截原始候选动作并投影回安全策略包络。若违反等式`Π_k Π_m [1 - Ileak(r_raw_k, m)·(1 - Auth(uk, m))] = 1`则触发故障关闭，替换为不可变拒绝负载。同时记录指令违反并触发反馈修正循环。
- **公式化目标**：多主体优化目标（式3）在隐私零泄露和全局硬目标满足的硬约束下最大化权限加权效用，任何违反硬约束的动作位于可行策略包络之外。

### 6. 论文的主要结论与发现

- **核心结论**：Harness-MU在所有访问控制攻击下实现**完全隐私保护**（Privacy=1.000，零方差），同时提升Utility和Overall分数：Overall增益从+0.240（Qwen）到+0.427（Deepseek）。在GPT上解决过度保守拒绝问题，Utility从0.257提升至0.589。
- **指令跟随**：在弱模型或冲突场景下提升显著：GPT对齐准确率从0.422升至0.912（+48.9pp），冲突准确率从0.309升至0.781（+47.2pp）。冲突模式下低权限用户准确率仍较低（~0.08–0.17），但Mediator能在兼容时服务可协商需求。
- **组件贡献**：ComplianceChecker确定性投影关闭了所有残留泄露（原始生成隐私0.932，投影后1.000），且从未误判合法交易（Utility差量为0）。IF修正循环呈现**逆缩放**现象：模型越弱触发率越高，增益越大（GPT触发66.8%获+7.94pp）。
- **系统开销**：虽增加总Token和LLM调用，但通过并行Worker使关键路径延迟降低11%（Deepseek AC场景）。

### 7. 优点

- **模型无关**：无需微调或修改底层LLM，可直接应用于任何模型。
- **确定性隐私保障**：通过外部合规检查器而非模型概率行为，实现故障关闭的强制隐私保护。
- **解耦架构**：模块化设计使策略编排、生成、验证分离，易于扩展和审计。
- **全面消融与成本分析**：定量隔离各模块贡献，并讨论延迟-Token权衡。
- **开源**：提供完整代码与数据，促进可复现研究。

### 8. 不足与局限

- **攻击者固定**：AC实验中攻击者仅使用deepseek-v4-pro，可能低估同族模型或更强的自适应攻击。
- **合成用户**：目前基于规则模拟用户，缺少真实人类交互的噪声、歧义和非正式权威信号。
- **效率开销**：尽管关键路径延迟降低，但总Token消耗增加121.8K/场景，在资源受限场景下可能不经济。
- **工具调用未覆盖**：当前未集成权限范围的工具矩阵（如数据库更新、邮件路由），限制了环境副作用评估。
- **隐私检测简单**：ComplianceChecker依赖子串匹配，可能被语义混淆或侧信道泄露绕过（论文在局限中提及未来应引入形式化验证、动态污点分析等）。
