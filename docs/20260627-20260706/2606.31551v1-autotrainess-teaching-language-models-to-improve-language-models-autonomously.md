## 摘要
训练语言模型（LM）仍然是一个高度依赖人力的过程，即使前沿语言模型代理在软件工程和其他长周期任务上变得越来越强大。一个核心挑战是，自主后训练不仅仅是编码问题：它需要代理反复规划迭代、构建与基准对齐的数据、运行稳定的训练作业、评估检查点，并在数小时的交互中保持实验状态。我们提出AutoTrainess，一个LM代理，它将这些操作暴露为代理-计算机接口的仓库，用于规划、数据准备、训练、评估和日志记录。与让代理在动作空间未充分指定的原始CLI环境中操作不同，AutoTrainess将先前的人类经验外部化为明确的工作流、规则和执行约束，引导代理实现有效且可靠的训练行为。在PostTrainBench上，AutoTrainess持续优于仅使用CLI的基线，使用GPT-5.4（Codex）时平均得分26.94，而仅CLI为23.21。它还能跨模型和工具泛化，将DeepSeek-V4-Flash（OpenCode）从12.13提升至19.58。

## Abstract
Training language models (LMs) remains a highly human-intensive process, even as frontier language model agents become increasingly capable at software engineering and other long-horizon tasks. A central challenge is that autonomous post-training is not just a coding problem: it requires the agent to repeatedly plan iterations, construct benchmark-aligned data, run stable training jobs, evaluate checkpoints, and preserve experiment state across many hours of interaction. We present AutoTrainess, a LM agent that exposes these operations as a repository of agent-computer interfaces for planning, data preparation, training, evaluation, and logging. Rather than leaving the agent to operate in a raw CLI environment with an underspecified action space, AutoTrainess externalizes prior human experience as explicit workflows, rules, and execution constraints that guide the agent toward effective and reliable training behavior. On PostTrainBench, AutoTrainess consistently outperforms CLI-only baselines, achieving 26.94 average score with GPT-5.4 (Codex) versus 23.21 for CLI-only. It also generalizes across models and harnesses, improving DeepSeek-V4-Flash (OpenCode) from 12.13 to 19.58.

## 1. 核心问题与整体含义

当前语言模型的后训练（post-training）高度依赖人工，即使前沿的LM代理在软件工程等长周期任务上表现出色，自主后训练仍然面临巨大挑战。其核心问题在于：自主后训练不仅仅是代码生成，它要求代理在数小时内反复规划迭代、构建与基准对齐的数据、稳定运行训练、评估检查点、并维护实验状态。在原始命令行界面（CLI）中，由于动作空间未充分指定，代理容易出错（如打包序列长度错误、使用错误对话模板），导致训练失败或效果不佳。AutoTrainess 提出将人类经验外部化为显式的 Agent-Computer Interface (ACI)，引导代理更可靠地执行完整后训练流程。

## 2. 方法论

- **核心思想**：设计一个训练专用的 ACI 框架，名为 AutoTrainHub，将人类专家累积的经验（工作流、规则、约束）编码为可复用的技能，指导代理按阶段执行规划、数据准备、训练、评估和日志记录。
- **关键技术细节**：
  - **数据处理模块**：包含三个显式步骤：数据选择（识别问题、选择来源方向）、数据构建（支持有限的集合操作：提取、清洗、去重、重写、重构、合成、蒸馏、模式归一化，并要求先检查基准评估接口）、数据验证（检查结构、对齐基准格式、内容质量，返回批准/返回构建/返回选择三类结果）。
  - **训练模块**：固定使用 LlamaFactory 框架，提供安装和执行脚本。要求进行全参数微调（SFT），先小验证再放大，导出 final_model/ 用于评估。对于强化学习（RL），仅在近期评估证据支持时才允许使用，且必须明确指定奖励定义。
  - **评估模块**：使用基准的正式评估入口点，输出原始结果、简洁指标摘要、15个随机样本的检查概要（含分数、输入、目标、模型输出），并总结主要失败模式及分类（数据问题、训练问题、推理/模板问题）。
  - **日志与规划模块**：每次完整迭代后追加结构化实验日志条目（包含上下文、动机、起点、数据、方法、配置、结果、分析、制品、下一步行动）。规划接口基于先前证据定义本轮目标、干预措施和成功标准。

- **Benchmark**：PostTrainBench 评估 4 个基础模型（Qwen3-1.7B、Qwen3-4B、SmolLM3-3B、Gemma3-4B）上的后训练效果，指标为加权平均分（权重与基准难度成反比）。

## 6. 主要结论与发现

- AutoTrainess 在 PostTrainBench 上显著优于 CLI-only 基线：GPT-5.4 (Codex) 平均得分 26.94 vs. 23.21（提升 ~15%）；使用 DeepSeek-V4-Flash 提升更加明显（12.13 → 19.58，+61%）。
- 消融实验显示每个接口都有不可或缺的贡献：

  - 评估接口：最直接降低评估动作失败率（7.6% → 22.8% 当移除时）。
  - 日志与规划接口：保持状态迭代，移除后训练-评估移交次数大幅下降（111 → 30），保留改进数从 7 降至 2。
- 代理行为揭示了时间模式：早期校准基准格式，中期采用针对性数据构建和训练策略（如 DPO、自蒸馏），后期聚焦失败案例分析和全评估。
- 改进的主要驱动因素是基准对齐（数据选择、模板调整、提示对齐）和针对性修正，而非通用能力扩展。

## 7. 优点

- **结构化外部化人类经验**：将高度依赖直觉和经验的训练流程编码为显式 ACI，极大降低了代理的决策不确定性。
- **模块化设计**：数据处理、训练、评估、日志四个接口可单独消融，便于诊断和扩展。
- **鲁棒性验证**：不仅验证了单一后端（Codex），还在 OpenCode 和 DeepSeek 上展示泛化能力。
- **深入的行为分析**：通过动作分类和时间分布图揭示了代理的规划-执行-反馈闭环行为，为未来设计提供了经验指导。
- **公平的基线对比**：使用相同的骨干模型、GPU 和时间预算，消融实验设计合理。

## 8. 不足与局限

- **实验覆盖限制**：消融实验仅在 Qwen3-4B 一个基础模型上进行，可能无法代表其他模型或更大模型上的效果。
- **框架绑定**：训练接口固定使用 LlamaFactory，限制了代理探索不同训练框架或自定义循环的自由度，可能遗漏更优方案。
- **骨干模型依赖**：主结果依赖 GPT-5.4 和 DeepSeek-V4-Flash 等强大闭源模型，这些模型成本高且可复现性差。论文提到尝试过更小的开源模型（Qwen3.5-35B-A3B）但表现不佳。
- **评分加权可能引入偏倚**：权重基于 instruct 与 base 模型得分差异，可能使得在某些基准上提升幅度小的任务被低估。
- **时间预算限制**：每个实验仅 10 小时，对于需要多次迭代才能达到较高性能的任务可能不够充分，且未分析更长时限下的表现。
- **未涉及更复杂的训练技术**：如 RLHF 完整流程、奖励模型训练等，仅涉及简单 SFT 和 DPO 风格训练。
- **潜在的外部有效性风险**：PostTrainBench 是特定基准集，结果可能无法直接推广到真实世界的后训练任务（如客户特定场景）。
