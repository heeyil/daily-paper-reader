Abstract —We present the AI Training Manager, a bounded LLM-based supervisory controller for adaptive machine learning training. Standard training pipelines often rely on fixed recipes or single-axis schedulers, which can struggle with mid-run failures such as severe overfitting, loss imbalance, over-regularization, exploration collapse, or unsafe exploration. Rather than replacing mathematical optimizers or acting as an unconstrained coding agent, the manager operates through a schema-conditioned inter-face: it reads structured telemetry snapshots from an active run, audits a constrained action space, and returns validated updates to training parameters such as learning rate, regularization strength, loss-weight coefficients, and exploration settings. We evaluate this architecture across supervised language mod-eling and reinforcement learning. On TinyStories, the manager detects and corrects overfitting, achieving a validation loss 60% lower than the baseline while producing auditable intervention logs. In this supervised setting, we additionally show that manager inference does not need to block the training loop: training can continue while a manager response is pending, and validated updates can be applied asynchronously once available. In a robotic manipulation reinforcement-learning task, we use the same bounded decision interface in an episodic closed-loop setting, where manager updates are applied at evaluation or checkpoint boundaries. The manager mitigates both conservative and unsafe exploration regimes. In the cautious regime, the manager rescues a failed run from 0.0 to 0.94 rollout safe success. In the reckless regime, where the fixed recipe eventually succeeds but only after the full budget and with high fluctuation, the manager reaches comparable safe success roughly 4x earlier by reducing excessive exploration.These results suggest that schema-conditioned LLMs can serve as bounded supervisory managers for live training runs, complementing conventional optimizers and schedulers with interpretable, multi-axis intervention capa-bilities.

I. I NTRODUCTION 

Modern AI training systems are not completely open-loop. Learning-rate schedulers, early stopping rules, adaptive loss weighting methods, curriculum schedules, and population-based training all adjust training behavior based on feedback. These mechanisms are useful and widely adopted, but they are usually specialized: they monitor a limited set of signals, control a limited set of knobs, or operate across multiple runs rather than as an auditable supervisor inside a single active training run. At the same time, training runs produce rich telemetry. A supervised run may expose training loss, validation loss, generalization gap, gradient norms, and auxiliary losses. A reinforcement-learning run may expose success rate, collision rate, reward components, action magnitudes, entropy, and safety metrics. Human practitioners often inspect these signals to decide how the next experimental trial should be modified. Recently, outer-loop automated research agents have begun to mimic this workflow by using LLMs to parse final metrics, rewrite training scripts, and launch subsequent runs. However, this supervisory reasoning remains largely external to the active optimization process. The question we study is whether such reasoning can be moved closer to the live training loop while keeping the system bounded, auditable, and constrained to safe interventions. We present the AI Training Manager, an LLM-based su-pervisor for closed-loop control of the training recipe. Here, “training recipe” refers to the configurable settings surround-ing the optimizer and objective, including learning-rate policy, objective weights, regularization strength, exploration settings, and reward or penalty coefficients. The manager does not update model parameters directly. It does not rewrite code, change architectures, or design new experiments. Instead, it reads a structured snapshot of the active run and selects from a predefined set of bounded interventions. A central design goal is to make manager decisions com-patible with live training rather than treating the LLM as an external offline analyst. In supervised learning, we implement this through an asynchronous application protocol. A simple implementation would pause training, call the LLM, wait for a response, and then continue. We avoid this because LLM inference latency should not determine the critical path of the gradient loop. Instead, the primary training loop continues while a manager request is pending. Each request is tied to a telemetry snapshot and versioned configuration. When the response arrives, the system validates the JSON, checks action bounds, verifies that the decision is still relevant, and applies it only at the next safe intervention boundary. Stale or invalid decisions are logged and converted to safe no-ops. In reinforcement learning, where asynchronous mid-update interventions were not stable in our setting, we use the same bounded decision interface but apply manager updates at episode or checkpoint boundaries. This paper evaluates the manager on controlled failure modes where the desired intervention is interpretable. In TinyStories language-model training, we evaluate recovery

II. R ELATED WORK 

A. Inner-Loop Heuristics and Adaptive Training Methods 

Modern training pipelines already contain several forms of feedback control. Learning-rate schedules, early stopping rules, curriculum schedules [2], and exploration-decay mech-anisms in reinforcement learning all adjust training behavior based on observed progress. Adaptive loss-balancing methods similarly modify the relative weights of multiple losses using task-specific signals such as learned uncertainty [3] or gradient magnitudes [4]. These methods are practical and widely used, but they are typically specialized: each mechanism observes a narrow signal family and controls a narrow set of knobs. Our goal is not to replace these established mechanisms. Instead, we study whether an LLM can act as a reusable su-pervisory layer over several bounded controls simultaneously. For example, in our auxiliary-head experiments, the manager can adjust the auxiliary loss coefficient, but the same interface also exposes learning-rate changes, regularization changes, and rollback/no-op decisions. The distinction is therefore not feedback versus no feedback, but specialized hand-designed feedback rules versus a schema-conditioned supervisor that reasons over heterogeneous telemetry and a constrained action space. 

B. Hyperparameter Optimization, AutoML, and Population-Based Training 

Hyperparameter optimization and AutoML methods search for effective training configurations across evaluations. Bayesian optimization [5], resource-allocation methods such as Hyperband [6], and related AutoML systems have shown that systematic search can outperform manual tuning. Population-Based Training (PBT) goes further by training a population of models concurrently, periodically exploiting stronger runs and perturbing hyperparameters to discover training schedules rather than a single fixed configuration [7]. Our setting differs in both authority and execution model. The AI Training Manager does not launch a population, copy model weights, mutate competing runs, or select among parallel trials. It supervises a single active run and proposes bounded updates to the live training recipe based on telemetry from that run. This makes the intervention trace auditable at the level of individual decisions: each manager action is tied to a snapshot, validated against an explicit schema, and applied only at safe intervention boundaries. 

C. Dynamic Algorithm Configuration and Parameter Control 

The closest classical framing to our work is dynamic algorithm configuration (DAC) and parameter control. DAC studies policies that adapt algorithm parameters during exe-cution rather than selecting a static configuration before the run [8]. This perspective is highly relevant to machine learning training, where the best learning rate, regularization strength, exploration pressure, or loss coefficient may change over time. Recent DAC benchmarks and AutoRL studies further show that dynamic control policies can be studied in controlled settings where the desired adaptation is interpretable [10], [24]. The AI Training Manager can be viewed as a language-model-based, schema-conditioned instance of this broader idea. However, unlike learned DAC policies trained offline over many problem instances, our manager uses an instruction-following LLM at run time, constrained by an explicit action schema and validation layer. This trades optimality guaran-tees for generality, interpretability, and ease of reuse across heterogeneous training failures. 

D. Automated Reinforcement Learning and Safe RL 

Reinforcement learning is particularly sensitive to training-recipe choices. Exploration settings, entropy pressure, re-ward penalties, and safety-related coefficients can determine whether a policy learns, collapses, or exhibits undesirable behavior. AutoRL methods attempt to automate parts of this pipeline, including hyperparameter selection, architecture choices, reward design, and algorithm configuration [9]. How-ever, many AutoRL approaches still evaluate candidate con-figurations across runs or require substantial search budgets. Our reinforcement-learning experiments focus on bounded mid-training recipe interventions in a single robotic-arm task. We do not claim to solve safe reinforcement learning or constrained policy optimization, which aim to provide stronger guarantees about constraint satisfaction during learning or deployment [11], [12]. Instead, we use safety-related telemetry, such as collision rate and safe success rate, as observable signals that the manager can use to adjust bounded exploration and penalty coefficients. 

E. Learned Optimizers and Meta-Learning 

Learned optimizers and learning-to-optimize methods at-tempt to replace or augment hand-designed optimization al-gorithms with learned update rules [13]–[15]. Recent work has studied learned optimization specifically for reinforcement learning, where nonstationarity, high-variance gradients, and agent-environment interactions make the problem especially difficult [16], [17]. The AI Training Manager adopts a lower-authority design. It does not replace PPO, Adam, AdamW, or the core parameter-update equations. Instead, it supervises the configurable recipe surrounding those optimizers: objective weights, regulariza-tion, exploration pressure, reward penalties, and learning-rate settings. This design intentionally preserves conventional optimizers as the mechanism for parameter updates while using the LLM only for bounded supervisory decisions. 

F. LLMs for Optimization, Hyperparameter Tuning, and Re-search Agents 

Recent work has explored LLMs as optimizers and hyperparameter-tuning agents. OPRO frames optimization as a prompting problem in which an LLM proposes candidate solutions from prior evaluations [18]. Other work uses LLMs to suggest hyperparameter configurations from dataset and model descriptions, iteratively refine candidates based on performance, or combine LLM proposals with classical HPO machinery [19], [20], [25]. Recent systems also investigate whether smaller LLMs can perform hyperparameter tuning when paired with deterministic expert blocks that summarize optimization trajectories [21]. These approaches demonstrate that LLMs can use training feedback to make optimization-relevant decisions. Our frame-work differs in three ways. First, the manager acts within an active training run rather than only proposing the next trial. Second, its agency is explicitly bounded: the LLM cannot rewrite code, alter the architecture, or invent new actions outside the schema. Third, each decision is validated before application, checked against action limits, and recorded as an auditable trace. This places the work between classical dynamic parameter control and recent LLM-based research agents, which can autonomously write code, launch experi-ments, and iterate over external research loops [22], [23]. III. T HE AI T RAINING MANAGER 

The AI Training Manager, a bounded LLM-based super-visory controller for adaptive machine learning training. It does not update model parameters directly. It does not rewrite training code, change the model architecture, introduce new losses, or create new experiments. Instead, it is called with a structured manager prompt and runtime context, and it returns a candidate decision in a required JSON format. We use the term training recipe to avoid ambiguity with the learned policy in reinforcement learning. A training recipe refers to configurable settings surrounding the optimizer, ob-jective, regularization, exploration process, reward-term coeffi-cients, or evaluation cadence. Examples include learning rate, dropout, weight decay, auxiliary loss weight, action standard deviation, entropy coefficient, movement cost, and safety-penalty coefficient. The manager’s role is to adjust these exposed recipe settings, subject to explicit bounds, based on task-specific guidance and observed training telemetry. At manager interval (k), the system constructs a manager call from two components: a prompt specification (p) and a runtime context ( ck). The prompt specification defines the manager’s role, task-family guidance, task-specific problem description, action surface, action limitations, and required output schema. The runtime context contains the current telemetry snapshot, the current recipe, relevant previous-run information, prior manager decisions, expected outcomes, and observed outcomes. A. Prompt Specification 

A. Prompt Specification The prompt specification contains the static information that defines the manager’s role, authority, and decision interface for a given training setup. In our implementation, the prompt has three main parts. First, the prompt contains task-family guidance. This section explains the general training regime in which the manager is operating, such as supervised or imitation-learning-style train-ing versus reinforcement learning. For supervised training, this guidance emphasizes generalization behavior, train-validation gaps, loss slopes, auxiliary-objective balance, regularization, and learning-rate stability. For reinforcement learning, it em-phasizes exploration quality, reward components, action mag-nitude, entropy, rollout stability, collision behavior, and task success. This portion gives the manager the “lay of the land” for interpreting telemetry within a broad class of training problems. Second, the prompt contains a problem-specific description. This section describes the concrete environment, objective, and observable metrics for the current experiment, without naming the injected failure mode. For the robotic-arm task, for example, the prompt describes the two-link arm, the initial configuration, the obstacle, the target region, the safe-success criterion, and the relevant diagnostic metrics. The same problem-specific prompt is used for both the cautious and reckless variants. Thus, the manager is given enough semantic context to interpret telemetry, but it is not told whether the current run is intended to represent conservative exploration, unsafe exploration, or a healthy recipe. Third, the prompt defines the exposed action surface. This section lists the controls that the manager is allowed to modify, the valid range or relative update bounds for each control, and the intended effect of each knob. For example, in the robotic-arm experiments, the action surface includes controls such as action standard deviation, entropy coefficient, actor learning rate, movement cost, and safety-penalty coefficient. The prompt also explains the expected qualitative effect of these controls: increasing action standard deviation can pro-mote exploration but may increase collision risk, while de-creasing it can stabilize behavior but may make the policy too conservative. The manager may only select from this declared action surface; it cannot invent new controls, change the model architecture, rewrite code, or introduce new objectives. This three-part prompt structure separates general training knowledge from problem-specific context and from executable authority. The task-family guidance helps the manager inter-pret the type of learning dynamics it is observing, the problem-specific description grounds that interpretation in the current environment, and the action-surface specification determines what interventions are admissible. The deterministic verifier then enforces this contract by rejecting malformed responses or requested updates outside the exposed bounds. 

B. Runtime Context 

The runtime context provides the evidence used for the current decision. It contains the current telemetry, current recipe settings, and a compact history of previous manager decisions. For each previous decision, the context records what was changed, what outcome was expected, and what actually happened afterward. This history can include earlier intervals of the same run and, for the AI manager condition, relevant prior-run information. Including decision history makes the manager less myopic. If a previous intervention improved validation behavior, the manager can treat that evidence differently from an interven-tion that had no measurable effect. Similarly, if a previous reinforcement-learning intervention increased exploration but caused a collision spike, the manager can avoid repeating the same aggressive update. For language-model training, telemetry includes quantities such as training loss, validation loss, train-validation gap, re-cent loss slopes, gradient norm, current learning rate, dropout, weight decay, and auxiliary losses when present. For reinforce-ment learning, telemetry includes safe success rate, collision rate, mean action magnitude, distance to target, episode return, entropy, action standard deviation, actor learning rate, and reward-term settings. The manager may choose no-op when the current recipe appears appropriate or when the available context is insufficient to justify a change. 

C. Bounded Action Surface 

The manager can only modify controls that are explicitly exposed in the action surface. The action surface lists each available control, its interpretation, valid range, allowed rel-ative update directions or magnitudes, and relevant cautions. The manager does not choose arbitrary absolute settings. In-stead, it proposes bounded relative updates such as increasing, decreasing, or holding an exposed control within predefined limits. The exposed controls differ by experiment. In the TinySto-ries overfitting and over-regularization experiments, the man-ager may adjust learning rate, dropout, and weight decay. In the auxiliary-head experiment, the manager may also adjust the auxiliary loss weight. In the robotic-arm reinforcement-learning experiments, the manager may adjust controls such as action standard deviation, entropy coefficient, actor learning rate, movement cost, and safety-penalty coefficient. This bounded action surface is central to the method. The LLM is not used as an unconstrained autonomous agent. It is used as a telemetry interpreter and decision proposer inside a deterministic control shell. The shell defines what the manager can see, what it can change, how large a relative update may be, and when a proposed decision is allowed to affect the run. 

D. Structured Output and Verification 

The manager must return structured JSON rather than free-form advice because the training system consumes the deci-sion programmatically. Each response contains a diagnosis, a control audit, a selected action, expected outcomes, and watch items for the next interval. The selected action may update one or more exposed recipe settings, or it may explicitly choose no-op. The diagnosis summarizes the current training state, such as overfitting, over-regularization, auxiliary-objective imbalance, cautious exploration, unsafe exploration, or healthy/no-action. The control audit evaluates the relevant exposed controls and indicates which are selected, rejected, or deferred. The selected-action field specifies the proposed bounded relative update. The expected-outcome field states what should im-prove during the next interval, and the watch-items field lists the metrics that should be inspected to judge whether the intervention helped, hurt, or remains inconclusive. After receiving the response, the deterministic verifier per-forms syntactic and semantic checks. It verifies that the response is valid JSON, that all required fields are present, that selected actions are part of the exposed action surface, and that proposed updates satisfy the allowed ranges and relative step limits. The verifier does not attempt to prove that the manager’s diagnosis is correct. It only determines whether the response is admissible under the interface contract. If verification succeeds, the decision is queued for appli-cation at the next safe intervention boundary. If verification fails, the response is logged but discarded, and the active training recipe remains unchanged. In this sense, the LLM never receives direct write access to the training loop. It proposes candidate decisions, while the deterministic shell decides whether those candidates are admissible. 

E. Application Protocols 

The same bounded manager interface can be used with different application timing depending on the training domain. In supervised training, we use an asynchronous protocol so that LLM latency does not block the gradient loop. At an evaluation boundary, the training loop creates a telemetry snapshot and sends a manager request. Training then continues with the current recipe while the request is pending. Each asynchronous request is tied to a snapshot interval and a versioned recipe configuration. When the response arrives, the system checks whether it is still valid for the current run state. A decision may become stale if training has advanced too far, if another decision has already changed the recipe, or if the current telemetry no longer matches the snapshot that triggered the request. A valid and still-relevant decision is queued for the next safe intervention boundary. A stale, invalid, or incompatible decision is logged and discarded. In reinforcement learning, we do not apply asynchronous mid-update interventions. In our setting, such interventions were less stable because rollout collection, policy updates, and evaluation phases have different semantics. We therefore use a boundary-synchronous protocol for the robotic-arm experi-ments. The manager is queried at rollout, update, evaluation, or checkpoint boundaries, and any verified decision is applied before the next control interval begins. This preserves the same prompt structure, action surface, verifier, and decision trace while avoiding recipe changes in the middle of a rollout or policy-update phase. 

F. Decision Logging 

Every manager request produces a decision trace. The trace records the prompt version, telemetry snapshot, current recipe, candidate response, diagnosis, control audit, selected update, expected outcome, verification status, application status, and subsequent outcome. For asynchronous supervised runs, the trace also records the snapshot interval, recipe version, re-sponse latency, whether the decision was stale, and whether it was applied. These logs serve two purposes. First, they make the system auditable: a researcher can inspect why the manager changed a setting, which alternatives it rejected, what outcome it ex-pected, and whether the next interval supported that decision. Second, they provide structured postmortem data that can guide later experiment design. For example, repeated increases in regularization may indicate that the default recipe is under-regularized, while repeated reductions in auxiliary loss weight may indicate objective imbalance.We treat these traces as structured decision records, not as hidden reasoning. IV. E XPERIMENTS AND RESULTS 

Our experiments are controlled failure-mode studies rather than broad hyperparameter-optimization benchmarks. The goal is to test whether a bounded AI manager can read live training telemetry, identify when a run has entered an interpretable failure regime, and apply a valid relative update to the exposed training recipe. The injected failure mode is known to the experimenter, but it is not named in the AI manager prompt. Across experiments, the manager can only select from the exposed action surface and can only propose bounded relative updates. It cannot rewrite code, change the model architecture, introduce new losses, or launch new experiments. The prompt contains task-family guidance, a task-specific description, ac-tion limitations, and the required JSON schema. Runtime context contains current telemetry, current recipe settings, and prior decision/outcome records. Malformed, incomplete, or out-of-bounds responses are logged but discarded. We evaluate two domains. In supervised TinyStories train-ing, we study validation degradation under small-data overfit-ting and a multi-objective auxiliary-head variant. In reinforce-ment learning, we study two opposite exploration failures in a robotic-arm reaching task: a cautious recipe that fails to discover useful reaching behavior and a reckless recipe that explores too aggressively. We compare fixed recipes against manager-controlled recipes. This lets us evaluate not only whether performance improves, but whether the manager’s interventions move the relevant telemetry in the expected direction. For all experiments the manager is using GPT 5.4 mini. 

A. TinyStories Language-Model Experiments 

We first evaluate the manager in supervised language-model training on TinyStories. These experiments test whether the manager can respond to validation degradation, train-validation gaps, and auxiliary-objective imbalance using only bounded recipe updates. The healthy baseline establishes normal behavior under a competent fixed recipe; the stress tests then reduce the training set size to create interpretable overfitting failures. All TinyStories experiments use a small GPT-style au-toregressive Transformer with 6 layers, 6 attention heads, embedding dimension 384, and context length 256. Training runs for 10k steps with evaluation every 500 steps, batch size 16, gradient accumulation 2, AdamW, and gradient clipping at 1.0. Unless otherwise noted, the base recipe uses cosine learning-rate annealing from 10 −3 to 10 −4, with no dropout and no weight decay. The manager-controlled runs start from the same base recipe but may apply bounded relative updates to exposed controls such as dropout, weight decay, learning rate, and, in the auxiliary-head experiment, the dialogue loss weight. 

1) Baseline: Figure 1 shows the healthy baseline using 20k training stories and 2k validation stories. In this regime, the standard cosine recipe is already effective: train loss decreases to 0.601, validation loss ends at 0.661, and the best validation loss is 0.652 at 7.5k steps. This baseline establishes that the model and training loop are not intrinsically unstable. When the dataset is sufficiently large relative to the model and training budget, ordinary cosine annealing is adequate, and a manager is not necessary. 

2) Overfitting Stress Test: To create a supervised failure mode, we reduce the training set from 20k stories to 2k stories while keeping the validation set at 2k stories and leaving the model, optimizer, learning-rate schedule, and training budget unchanged. This produces a classic overfitting regime: the model has enough capacity and training time to fit the small training subset, so training loss can continue improving even after validation behavior degrades. Fig. 1. Healthy TinyStories baseline. With 20k training stories and 2k validation stories, a standard cosine-annealing recipe produces a stable run: both train and validation loss decrease smoothly, and the train-validation gap remains small. 

Figure 2 shows the resulting train and validation losses. In the fixed cosine run, train loss falls monotonically to 0.112 by 10k steps. However, validation loss bottoms out early, reaching its best value of 0.852 around 3k steps, and then rises sharply to 2.09 by the end of training. This is the intended failure mode: if one only observes train loss, the run appears to improve, but the validation trajectory shows that the model is memorizing the small training set rather than learning a better general story model. The manager-controlled run starts from the same overfitting-prone recipe. Once the train-validation gap opens, the manager applies bounded regularization updates. The final train loss is 0.852, much higher than the fixed overfitting run, but this is the desired tradeoff: the manager prevents the model from continuing to optimize training loss at the expense of validation performance. The manager-controlled run finishes with validation loss 0.800 and reaches a best validation loss of 0.770 at 7.5k steps. Thus, the manager improves validation behavior not by accelerating optimization, but by recognizing that lower training loss has become a misleading objective in this regime. Figure 3 shows the corresponding recipe changes. The fixed cosine recipe keeps dropout and weight decay at zero throughout training. The manager increases dropout from 0.0 to 0.35 and weight decay from 0.0 to approximately 0.057. These changes explain the loss behavior in Figure 4: the manager-controlled model no longer drives train loss as low, but it preserves much better validation performance. This is the central supervised-learning result of the paper. The manager reads multiple signals together—train loss, validation loss, and the train-validation gap—and selects a bounded intervention that a single training-loss-driven objective would not choose.

V. D ISCUSSION AND LIMITATIONS 

The experiments support a narrow but useful claim: aschema-conditioned LLM can act as a bounded supervisory layer for controlled training failures. The manager is not optimizing model parameters directly, searching over archi-tectures, or launching new experiments. Instead, it reads structured telemetry, selects from an explicit action surface, and proposes bounded relative updates that are verified before application. This framing is important because the observed gains do not require treating the LLM as an unconstrained autonomous agent. The LLM supplies diagnostic and control-selection behavior, while the deterministic shell enforces the interface contract. The strongest TinyStories result is the overfitting stress test. In the fixed cosine recipe, train loss continues to improve while validation loss degrades sharply. The manager responds by increasing regularization and sacrificing training-set fit to preserve validation performance. This is precisely the kind of intervention a training supervisor should make: it should not blindly optimize the most immediately improving scalar, but should recognize when that scalar has become misleading. The auxiliary-head experiment extends this behavior to amulti-loss setting, where the manager reduces the weight of an auxiliary dialogue-detection objective that saturates early while preserving language-model validation performance. In the scared run it increases action standard deviation to enable exploration; in the reckless run it decreases action standard deviation to reduce excessive exploration. behavior. The same task prompt and action surface are used for both the scared and reckless arm settings, and the manager is not told which failure mode is present. In the scared setting, it increases action noise to enable exploration; in the reckless setting, it decreases action noise to reduce excessive exploration. This is the clearest evidence that the manager is not simply applying a fixed rule such as ‘increase exploration” or ‘decrease exploration.” It selects opposite bounded updates from the same action surface based on telemetry. A central design choice is that the manager’s authority is intentionally limited. This has two advantages. First, invalid or out-of-scope responses can be rejected without affecting training. Second, the resulting decision traces are auditable: each intervention can be inspected in terms of the telemetry snapshot, diagnosis, selected control, expected outcome, and subsequent behavior. This makes the manager closer to a bounded control layer than to an open-ended code-writing agent. There are several limitations. First, our experiments are controlled failure-mode studies rather than broad AutoML benchmarks. The tasks were selected because the failure mech-anisms are interpretable, not because they exhaust the space of possible training failures. Second, the experiments use rela-tively small models and compact environments. Larger models, longer runs, and more realistic reinforcement-learning tasks may expose different failure dynamics. Third, the manager re-ceives structured decision/outcome context that fixed baselines do not receive, although scheduler baselines condition on their own metric histories by design. Future comparisons should more systematically separate the value of LLM reasoning from the value of structured memory. Fourth, the auxiliary dialogue-detection task is an intention-ally simple multi-loss test. Because the auxiliary objective saturates quickly, the manager mostly learns to reduce its weight rather than negotiating a persistent tradeoff between two difficult objectives. Stronger future tests should use aux-iliary objectives that remain informative throughout training, such as entity consistency, event ordering, factual consistency, or style control. Fifth, asynchronous application worked nat-urally for supervised training but did not transfer directly to reinforcement learning in our setting. For the robotic-arm experiments, we therefore used boundary-synchronous updates at rollout, update, evaluation, or checkpoint boundaries. This suggests that the manager interface can generalize across domains, but the application timing must respect the semantics of each training loop. Finally, the manager does not provide formal optimality or safety guarantees. In reinforcement learning, safety-related quantities such as collision rate and safe success rate are used as telemetry for bounded recipe updates; the method is not a substitute for constrained policy optimization or formally safe RL. Similarly, in supervised learning, the manager im-proves validation behavior in the studied stress tests, but it does not guarantee better generalization across all datasets, architectures, or training regimes. These limitations define the next steps. The most important extensions are larger-scale supervised experiments, harder auxiliary objectives, stronger reinforcement-learning environ-ments, more seeds, and comparisons against hand-designed heuristic managers, dynamic algorithm configuration methods, and population-based approaches. Another useful direction is prompt and context ablation: removing prior decision history, removing task-family guidance, or varying the action surface would clarify which parts of the manager interface contribute most to recovery. VI. C ONCLUSION 

We presented the AI Training Manager, a bounded LLM-based supervisor for adaptive training recipes. The manager operates through a structured interface: it receives task guid-ance, telemetry, current recipe settings, prior decision/outcome context, and an explicit action surface, then returns a candidate JSON decision that is verified before it can affect training. This design keeps the LLM’s authority limited while preserving its ability to interpret heterogeneous training signals. Across TinyStories language-model training and a robotic-arm reinforcement-learning task, the manager recovers from controlled failure modes using bounded relative updates. In TinyStories, it mitigates small-data overfitting by increasing regularization and preserving validation performance despite worse training loss. In the auxiliary-head setting, it reduces the weight of a saturated dialogue-detection objective while main-taining language-model validation behavior. In reinforcement learning, it applies opposite action-noise updates for scared and reckless exploration regimes under the same prompt and action surface. These results suggest that schema-conditioned LLMs can serve as useful training-time supervisors when embedded inside a deterministic control shell. The manager should not be viewed as a replacement for optimizers, schedulers, or sys-tematic hyperparameter search. Rather, it provides a reusable supervisory layer that can read richer telemetry than ordinary single-axis rules, select among multiple bounded controls, and leave behind an auditable trace of its decisions.
