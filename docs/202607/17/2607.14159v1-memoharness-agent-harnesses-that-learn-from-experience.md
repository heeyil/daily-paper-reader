## 摘要
智能体框架（agent harness）是外部控制层，通过管理上下文、工具、编排、记忆、解码和输出处理，将基础大语言模型转变为可执行智能体。虽然框架设计强烈影响智能体行为，但大多数自动改进方法优化的是更窄的产物，如提示词、管道或工作流，且已部署的智能体通常对所有案例复用单一的全局框架。我们提出MemoHarness，一个从自身执行中学习的自适应框架优化框架。MemoHarness将框架分解为六个可编辑的控制维度，将每个案例的诊断和提炼后的全局模式存储在双层经验库中，并利用检索到的经验（无需测试时标签、反馈或额外搜索）将学到的框架适应于每个测试案例。在我们对Shell智能体、代码生成和分析推理基准的评估中，MemoHarness相比我们对比的固定框架有所改进，并展现出对未见套件和基础模型的选择性迁移。当大部分检索到的经验可缓存时，其额外上下文也能保持成本竞争力。这些结果证明，执行经验是构建比单一静态配置更具适应性的智能体框架的实用基础，而关于统计鲁棒性和组件归因的更广泛主张则留待未来工作。

## Abstract
An agent harness is the external control layer that turns a base LLM into an executable agent by managing context, tools, orchestration, memory, decoding, and output handling. While harness design strongly affects agent behavior, most automatic improvement methods optimize narrower artifacts such as prompts, pipelines, or workflows, and deployed agents usually reuse a single global harness for all cases. We introduce MemoHarness, an adaptive harness optimization framework that learns from its own executions. MemoHarness decomposes the harness into six editable control dimensions, stores per-case diagnoses and distilled global patterns in a dual-layer experience bank, and adapts the learned harness to each test case using retrieved experience without test-time labels, feedback, or additional search. In our evaluation across shell-agent, code-generation, and analytical-reasoning benchmarks, MemoHarness improves over the fixed harnesses we compare against and shows selective transfer to unseen suites and base models. Its additional context can also remain cost-competitive when much of the retrieved experience is cacheable. These results provide evidence that execution experience is a practical substrate for building agent harnesses that are more adaptive than a single static configuration, while leaving broader claims about statistical robustness and component attribution to future work.

- **核心问题**：如何让智能体框架从自身的执行经验中学习，从而在部署时无需测试标签、反馈或额外搜索就能针对每个测试案例进行自适应调整？

## 2. 方法论：核心思想、关键技术细节
### 2.1 核心思想
- 将智能体框架视为可编辑的结构化对象，而非单一黑盒。
- 在训练时通过搜索积累诊断性经验（包括成功/失败案例、维度级故障归因、全局模式），在测试时基于这些经验对全局框架进行案例级适应。
- 遵循**正确性优先（correctness-first）**原则：主要奖励决定排序，执行成本（token用量）仅作为次要打破平局条件。

### 2.2 六维框架空间（Six-Dimensional Harness Space）
| 维度 | 阶段 | 定义与示例操作 |
|------|------|----------------|
| D1 上下文组装 | 预调用输入构建 | 从指令、约束、检索材料、示例中构建模型输入；如结构化提示、添加演示、压缩上下文 |
| D2 工具交互 | 外部工具和检索使用 | 控制何时及如何调用工具或检索器；如启用检索、设置top-k、重排证据 |
| D3 生成控制 | 解码配置 | 设置采样和预算参数；如提高max tokens、降低温度、采样多个候选 |
| D4 编排 | 工作流拓扑 | 选择模型调用序列和中间推理步骤；如单次调用改为规划/执行/精化 |
| D5 记忆管理 | 跨调用状态持久化 | 确定哪些状态跨调用持久化、哪些陈旧上下文被移除；如保持状态、总结轨迹、丢弃过期上下文 |
| D6 输出处理 | 调用后输出处理 | 将原始模型输出转换为最终答案；如提取答案、验证格式、选择备用方案 |

### 2.3 双层经验库（Dual-Layer Experience Bank）
- **案例级经验 Eₜ**：为每个搜索案例和迭代存储执行条目，包括配置、轨迹、奖励、成本、诊断信号（主故障维度、次故障维度、文本分析）。
- **全局经验 Gₜ**：每N轮搜索，通过蒸馏算子从失败聚类中提取跨案例的规律性模式（常见现象、支持证据、预期修复效果）。
- 查询接口：控制器不直接读取整个库，而是通过结构化查询获取有界的证据切片，保持上下文预算可控。

### 2.4 训练时优化（Phase A）
- 从最小框架 W₀（无演示、无工具、无结构化组装、单次确定性解码、无跨调用记忆、原始输出直通）开始。
- 每轮迭代：控制器基于当前框架和累积经验构建查询，检索证据切片，提出新配置 Wₜ，对搜索集中的所有案例执行并收集轨迹、奖励、成本、诊断，更新经验库。
- **正确性优先选择**：最终通过词典序选择全局框架 W*：先最大化平均奖励，奖励相同时最小化平均token成本。

### 2.5 测试时案例自适应（Phase B）
- 对每个无标签测试案例 x=(u, φ)：
  1. 从经验库检索相似成功案例（N⁺ₖ）和失败案例（N⁻ₖ），以及全局模式。
  2. 相似度基于指令的嵌入余弦相似度。
  3. 控制器基于检索证据和全局框架 W*，输出案例特有框架 W(x)。
  4. 执行 W(x) 得到预测，无需测试时反馈或迭代搜索。

## 6. 主要结论与发现
1. **自适应框架优化提升任务成功率**：在Terminal-Bench上，MemoHarness达到0.806，比最强固定框架Codex（0.722）高出+0.084。
2. **搜索过程带来稳定提升**：三个基准上最终框架均优于初始基架（0.722→0.806, 0.900→0.967, 0.600→0.767），其中FinanceAgent（长程分析）提升最大，LiveCodeBench（接近饱和）提升较小。
3. **选择性跨套件迁移**：Terminal-Bench学到的框架在MMMU（+0.030）、StrongReject（+0.030）、SWE-Bench Pro（+0.059）上有改善，但并非所有套件均提升（LawBench部分反向）。表明迁移并非通用，但长程工具密集型搜索任务产生的框架更具泛化性。
4. **跨模型迁移有效**：在6个不同模型上，平均提升+0.098（范围+0.038到+0.233），表明学到的规则并非模型特定提示技巧，而是较可移植的执行策略。
5. **成本可控**：虽然输入token更多，但大部分可缓存，总成本低于Codex和Claude Code，高于开放框架但准确率显著更高。

## 7. 优点
- **结构化分解合理**：将框架沿推理时间流分解为六个独立维度，使搜索变为结构化编辑，而非黑盒优化，便于诊断和修复。
- **经验积累与重用**：双层经验库不仅存储标量分数，还存储维度级诊断和全局模式，使搜索迭代具有诊断性；测试时通过检索实现无反馈适应，不增加测试时计算成本。
- **正确性优先设计**：避免搜索偏向低成本但低质量配置，保持任务成功率作为首要目标。
- **全面的迁移实验**：跨套件和跨模型测试覆盖了泛化能力的多个方面，增加了结果的可信度。
- **成本分析包含缓存考虑**：更贴近实际部署场景（检索经验可重复使用）。

## 8. 不足与局限
- **统计显著性缺失**：主实验结果仅报告点估计，无置信区间或假设检验，影响了结论的严谨性。
- **基线对比公平性有限**：部分基线（如Claude Code）底层模型或工具集不同，无法单纯归因到框架差异。作者承认这是“系统级对比”而非纯框架消融。
- **组件消融不完整**：未单独消融经验库、全局模式、测试时自适应等每个组件在所有设置中的贡献，难以确定各部分的具体贡献。
- **成本分析的缓存假设**：当前缓存率（13.32/14.18）来自实验环境，实际应用中缓存利用率可能较低，导致成本上升。
- **实现依赖启发式**：控制器和诊断算子的实现使用实际启发式规则，未来应探索更通用的学习方法。
- **搜索规模有限**：仅使用10轮搜索、80/20分割，可能未充分探索解空间。作者在局限性中提及需要更大规模验证。
- **未涉及全无监督搜索**：搜索阶段仍然使用标签进行奖励计算，未来工作应研究完全无监督的在线积累。
- **仅使用单一基础模型进行搜索**：所有实验的搜索阶段均使用GPT-5.3-Codex，未测试其他模型作为搜索基座的效果。
