## 摘要
长周期智能体的优化越来越依赖基于反思的机制，其中大语言模型（LLM）作为优化器来诊断智能体失败并改进智能体策略。然而，真实执行轨迹很难直接用于优化：大量轨迹集合往往冗余且异质，导致优化效率低下且容易过拟合到低价值失败；同时，每个单独轨迹也包含许多无关步骤，而简单的上下文缩减方法（如截断或滑动窗口）会丢弃因果关键证据，产生误导性的优化信号。为解决这一困境，我们引入了STRACE（结构轨迹分析与因果提取）框架，该框架构建高信噪比的优化上下文，以实现更精确有效的优化。在批次层面，STRACE挖掘失败模式以过滤冗余轨迹并保留代表性失败；在每个选定轨迹内，它通过文本依赖图进行因果定位，去除非因果步骤并识别真正的根因模块进行优化。实验结果表明，STRACE显著优于标准上下文过滤基线。值得注意的是，在一个具有挑战性的形式化验证任务（VeruSAGE-Bench）上，它成功优化了人类专家设计的智能体，成功率提升了1.4倍（从42.5%到58.5%）。代码可在 https://github.com/moomight/STRACE 获取。

## Abstract
The optimization of long-horizon agents increasingly relies on reflection-based mechanisms, where a large language model (LLM) acts as an optimizer to diagnose agent failures and improve agent policies. However, real execution traces are difficult to use directly for optimization: large trace collections are often redundant and heterogeneous, making optimization inefficient and prone to overfitting to low-value failures; meanwhile, each individual trajectory also contains many irrelevant steps, while naive context reduction methods such as truncation or sliding windows can discard causally important evidence and produce misleading optimization signals. To resolve this dilemma, we introduce STRACE (Structural TRajectory Analysis and Causal Extraction), a framework that constructs high signal-noise optimization contexts for more precise and effective optimization. At the batch level, STRACE mines failure patterns to filter redundant traces and retain representative failures; within each selected trace, it performs causal localization over a textual dependency graph to remove non-causal steps and identify the true root-cause module for optimization. Empirical results demonstrate that STRACE significantly outperforms standard context-filtering baselines. Notably, on a challenging formal verification task (VeruSAGE-Bench), it successfully optimizes human-expert designed agents, delivering $1.4\times$ success-rate improvement (42.5% to 58.5%). The code is available at https://github.com/moomight/STRACE .

## 1. 核心问题与整体含义（研究动机和背景）
长周期智能体（long-horizon agents）的优化通常采用“反思式优化”（reflection-based optimization）范式：让 LLM 充当优化器，通过分析智能体的执行轨迹来诊断失败、改进策略。然而，实际执行轨迹存在两大困难：
- **批量层面**：轨迹集合通常冗余且异质（heterogeneous），包含大量重复的低价值失败案例，导致优化效率低下、容易过拟合；
- **单轨迹层面**：单个轨迹内包含大量无关步骤，传统上下文缩减方法（如截断、滑动窗口）会丢弃因果关键证据，产生误导性优化信号（即“上下文-噪声权衡”）。

因此，需要一种方法既能从批量轨迹中筛选出代表性失败，又能从单轨迹中准确定位真正的根因模块（而非表观故障点），从而构建高信噪比（high-SNR）的优化上下文。

## 2. 方法论：核心思想、关键技术细节
**STRACE（Structural Trajectory Analysis and Causal Extraction）** 将执行日志视为因果图而非线性文本，通过四阶段解耦流程实现优化：

- **Phase 1: 结构性建模（Structural Modeling）**  
  利用 LLM 读取智能体源码，构建文本化的执行依赖图（Execution Dependency Graph, EDG）。图中节点为原子功能模块，边分为数据依赖（下游模块消费上游模块的工件）和控制依赖（上游模块决定下游是否被调用）。该图作为后续阶段的拓扑先验。

- **Phase 2: 失败模式挖掘与轨迹过滤（Failure Pattern Mining & Trace Filtering）**  
  首先，通过确定性 Python 解析器提取轨迹的全局结果、局部节点状态和模块调用序列，并进行两种分析：  
  - **统计严重性**：计算条件概率 P(任务失败 | 某模块失败)，识别最致命的瓶颈；  
  - **结构路径模式**：追踪反复出现的异常拓扑（如无限自环、死胡同）。  
  然后，基于诊断结果对轨迹进行聚类，从每个簇中选取代表性少量样本（设定大小 s=5），形成紧凑且覆盖所有主要失败模式的轨迹子集。  
  若轨迹缺少显式节点错误信号，则退化为仅依赖全局结果与结构路径模式进行聚类。

- **Phase 3: 因果定位（Causal Localization）**  
  区分“表观节点”（manifestation node, 故障显现的位置）和“根因节点”（root cause node, 真正逻辑缺陷所在）。分为两步：  
  - **因果上下文蒸馏（Causal Context Distillation）**：从表观节点开始，利用 EDG 沿数据/控制依赖边反向递归追溯，保留能到达表观节点的闭环依赖（→*），剪除无结构影响的步骤，得到紧凑的“因果切片”（causal slice）。  
  - **根因隔离（Root Cause Isolation）**：在无噪声的因果切片中，追溯失当信息流的起点，定位首个偏离意图的模块，标记为根因节点。

- **Phase 4: 归纳式策略优化（Inductive Policy Optimization）**  
  将同根因节点的因果切片聚合，由优化器（LLM）进行归纳抽象，生成可复用的自然语言启发式规则（heuristics），而非实例级失败记录。将这些规则注入到根因模块的提示策略中，实现精准、持续的优化。

**关键创新**：通过批级别模式筛选和单轨迹因果定位，打破了“上下文-噪声权衡”，使优化器聚焦于真正需要修复的根因，避免无根据的预处理和过拟合。

## 3. 实验设计
- **数据集/基准（Benchmarks）**：
  - **HotpotQA**：多跳问答（一般推理），随机 150/300 训练测试分割，评估 Exact Match（EM）。
  - **WebArena**：通用网页交互任务，涵盖 Shopping、CMS、Reddit、GitLab 四个领域，评估 Success Rate（SR）。
  - **VeruSAGE-Bench**：领域密集型形式化验证任务，包含五个 Rust 项目（IronKV、Memory Allocator、Node Replication、NRKernel、Storage），每项目 80%/20% 分割，平均代码长度 947 行，评估 SR。

- **基线与对比方法**：
  - **Base Agent**: 未优化原始智能体。
  - **静态策略**: Naive Few-shot（随机选取示例）、Failure-Aware RAG（BM25 检索失败轨迹并作为指令参考）。
  - **上下文压缩策略**: Summary-based（全轨迹摘要后优化）、Retrieval-based（检索最相关步骤作为优化输入）。
  - **自动优化器**: TextGrad（基于梯度的全轨迹优化，因开销太大被排除在 VeruSAGE-Bench 之外）、GEPA（进化式优化，采用局部节点截断）。  
  所有方法使用相同的优化器模型（Claude Sonnet 4.5），以确保公平比较。

- **配置**：优化器温度 1.0，瓶颈阈值 k=5，代表样本数 s=5。成本按 API 价格估算（USD）。

## 6. 主要结论与发现
- STRACE 在所有基准上均取得最佳平均性能：HotpotQA 上 EM 68.5%，WebArena 上整体 SR 23.7%（+12.9% 相对于基线），VeruSAGE-Bench 上整体 SR 58.5%（+16.0% 绝对提升）。
- 在长周期、领域密集型任务（VeruSAGE）上的优势最为显著，说明因果定位对复杂失败传播尤为重要。
- 成本-性能趋势显示，随着训练轨迹数量增加，STRACE 在保持高效的同时还能提高性能，优于全轨迹（TextGrad）和局部截断（GEPA）方法。
- 消融结果确认了结构性建模、轨迹过滤、因果定位各自不可或缺的作用；去除任何阶段都会导致性能下降或成本激增。
- 案例研究证明，STRACE 能将优化从表观符号节点重新导向上游根因节点（如 assertion_reasoning_pipeline），从而打破无效循环。

## 8. 不足与局限
- **假设与适用范围**：STRACE 要求一定程度地“可见性”（visibility）——能访问智能体系统源代码、配置、提示等工件，以构建依赖图和提取节点状态。对于完全黑盒、仅提供执行轨迹的智能体，STRACE 的直接适用性有限。论文将此列为未来工作。
- **实验覆盖**：
  - 基准数量尚可，但未见在更复杂（如涉及多智能体协作、动态结构变化）场景下的验证。
  - 优化器仅使用了 Claude Sonnet 4.5，未比较不同优化器模型对方法鲁棒性的影响（仅在依赖图构建中试验了多种模型）。
- **偏差风险**：依赖 LLM 进行依赖图构建和因果推断，可能受 LLM 自身偏见和幻觉影响（尽管扰动实验表明具有鲁棒性）。
- **成本**：虽然成本可控，但若应用于极大规模轨迹集合，Phase 2 的解析和聚类计算开销可能增加。
- **伦理**：优化后的智能体若被用于不安全或恶意场景，可能加剧滥用风险。论文已提及需在部署前进行审查。
