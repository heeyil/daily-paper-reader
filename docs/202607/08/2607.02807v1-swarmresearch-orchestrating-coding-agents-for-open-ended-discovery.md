## 摘要
像自动研究这样的长时间运行的编码智能体可以持续发现开放式问题的优化方案。然而，它们倾向于收敛到单个高层次方法，然后进行低层次编辑，同时错过其他更优的问题解决方法。我们假设两个框架级设计选择导致了这种行为：在单个长时间运行的智能体中积累上下文，以及只暴露单个程序状态进行编辑。我们引入了SwarmResearch，一种编排器-子智能体框架，其中Shepherd智能体使用全局上下文引导一群搜索智能体，每个搜索智能体在其各自的git分支中操作本地上下文。在开放式优化任务中，SwarmResearch在13/15个任务上通过更高层次的探索发现了比最先进的LLM引导演化和多智能体技术更好或可比的解决方案。与固定扩展的串行和并行智能体相比，SwarmResearch的编排器引导扩展通过在不同搜索深度调整并行度，发现了性能更优的解决方案。

## Abstract
Long-running coding agents such as autoresearch can persistently discover optimizations for open-ended problems. However, they tend to converge onto a single high-level approach, then proceed with low-level edits while missing other superior approaches to the problem. We hypothesize two harness-level design choices contribute to this behavior: accumulating context in a single long-running agent and only exposing a single program state to edit. We introduce SwarmResearch, an orchestrator-subagent harness in which a Shepherd Agent uses global context to steer a population of Search Agents, each operating with local context in their respective git branch. On open-ended optimization tasks, SwarmResearch discovers better or comparable solutions to state-of-the-art LLM-guided evolution and multi-agent techniques on 13/15 tasks, driven by higher-level exploration. Compared with fixed scaling of serial and parallel agents, SwarmResearch's orchestrator-guided scaling discovers better-performing solutions by adapting parallelism at different search depths.

### 1. 核心问题与整体含义

*   **背景与动机**：长时间运行的编码智能体（如 AutoResearch）能够持续改进开放式问题（无已知最优解）的解决方案，但它们倾向于快速收敛到单个高层次方法，随后陷入低层次微调（micro-optimizations），从而错过其他更优的解决方案方向。
*   **核心问题**：作者将这种收敛行为归因于两个框架层面的设计缺陷：
    1.  **上下文积累**：单一智能体在长对话历史中不断积累针对当前方法的增量修改，使得切换到全新方法变得困难。
    2.  **单一程序状态**：所有改进都提交/回滚到同一个可编辑程序中，不保留未最优但有潜力的分支方向，本质上退化为贪婪局部搜索。
*   **整体含义**：本文旨在通过改进智能体框架的设计（上下文管理和版本控制）来解决上述瓶颈，从而在开放式探索中实现更广泛、更高层次的搜索，避免过早收敛。

### 2. 方法论

*   **核心思想**：提出 **SwarmResearch**，一种**编排器-子智能体（Orchestrator-Subagent）** 框架。通过分离全局与局部上下文、利用 git 分支进行版本管理，实现高层次的多样化探索。
*   **架构设计**：
    *   **Shepherd Agent（牧羊人智能体）**：充当编排器，拥有全局上下文（所有搜索智能体的摘要、得分）。负责：
        *   **父选择（Parent Selection）**：为新搜索智能体指定起点（来自基线或已有完成的智能体分支）。
        *   **智能体类型选择**：
            *   **Explorer（探索者）**：使用全新上下文窗口，不受先前工作锚定，适合尝试高层次重写。
            *   **Optimizer（优化者）**：继承父智能体的对话历史，适合在已有方案上做连续串行细化。
        *   **提示（Prompts）**：可以补充少量相关上下文（如已探索过的想法、互补发现）。被限制不能指定具体要研究的想法，以保持多样性。
    *   **Search Agents（搜索智能体）**：子智能体，在单独的 git 分支和 worktree 中运行，拥有局部上下文（worktree 内容，Optimizer 额外有父对话历史）。每个搜索智能体完成后提交结果并写入 findings.md。
*   **关键机制**：
    *   **上下文分离**：搜索智能体无法访问其他分支的上下文，避免多智能体系统中的“意见收敛”问题。Shepherd 通过全局信息决定策略，但只传递必要的最小上下文给子智能体。
    *   **Git 分支管理**：每个新搜索智能体从 `main` (基线) 或某已完成智能体的提交创建新分支。这允许多种搜索模式：从同一父节点扇出（探索同一方向的不同变体）、串行链（深度细化）、多父节点合并（组合不同方案）。避免了仅编辑单个程序导致的贪婪局部搜索。

### 6. 主要结论与发现

1.  **性能优越**：SwarmResearch 在 13/15 个开放式优化任务上超过或匹配 EvoX 和 CORAL，体现更有效的搜索。
2.  **高层次探索**：SwarmResearch 的平均每次尝试修改的代码行数更大（中位数是 CORAL 的 3.2 倍，EvoX 的 1.7 倍），表明其进行了更多高层次方案变革而非低层次微调。
3.  **编排器引导扩展超越固定扩展**：在 4/5 的任务中，编排器引导的变并行度扩展（Shepherd 动态分配串/并行）优于最优固定并行×串行配置，因为它能根据搜索深度自适应调整资源。
4.  **推测解码案例**：SwarmResearch 在约 11 小时内设计出比 vanilla 解码快 4.58× 的近似推测解码方案（维护 60.6% 准确率，基线 65.8%），超越 AutoResearch（1.80×）和 CORAL（2.26×）。最佳方案包括系统级优化（如批处理、自适应 MoE 专家数）和可接受速度-准确率权衡的技术。
5.  **Shepherd 智能体行为**：典型行为是初始波次探索，然后聚焦少数优秀方案，最后收缩到单一方案。其策略选择能力有限，依赖提示。

### 7. 优点
