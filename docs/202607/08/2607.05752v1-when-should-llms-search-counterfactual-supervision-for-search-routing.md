## 摘要
搜索增强的语言模型可以使用外部证据来弥补参数化知识的不足，但搜索并不总是有益的：模型可能会对其已经能够回答的问题调用搜索，或者在更合适进行纠正、澄清或弃权时依赖有噪声的证据。我们将此表述为实例级搜索路由问题：决定相对于不搜索执行，是否需要搜索来提高任务成功率。为了获得监督，我们比较同一问题的无搜索和强制搜索结果，并根据任务特定的成功构建一个包含无搜索、搜索和未解决三种状态的预言机。使用这个预言机作为评估标准和学习信号，我们通过监督微调和偏好优化训练搜索路由策略，将Gemma E2B上预言机合格样本的路由宏F1从0.7082提高到0.8235，Qwen3.5-4B上从0.7053提高到0.8365。进一步分析表明，学习到的策略减少了特定模型的路由失败：Gemma主要学会无搜索约束，而Qwen进一步减少了遗漏搜索；剩余的未解决案例揭示了涉及模型容量、检索预算、证据使用和策略行为的异质性瓶颈。

## Abstract
Search-augmented language models can use external evidence to compensate for limitations in parametric knowledge, but search is not uniformly beneficial: models may call search for questions they can already answer, or rely on noisy evidence when correction, clarification, or abstention would be more appropriate. We formulate this as an instance-level search-routing problem: deciding whether search is needed to improve task success relative to a no-search execution. To derive supervision, we compare no-search and forced-search outcomes for the same question and construct an oracle over NO SEARCH, SEARCH, and UNSOLVED based on task-specific success. Using this oracle as both an evaluation criterion and a learning signal, we train search-routing policies with supervised fine-tuning and preference optimization, improving routing macro-F1 on oracle-eligible examples from 0.7082 to 0.8235 for Gemma E2B and from 0.7053 to 0.8365 for Qwen3.5-4B. Further analysis shows that the learned policies reduce model-specific routing failures: Gemma primarily learns no-search restraint, while Qwen further reduces missed search; residual UNSOLVED cases reveal heterogeneous bottlenecks involving model capacity, retrieval budget, evidence use, and policy behavior.

# 论文总结：When Should LLMs Search? Counterfactual Supervision for Search Routing

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：搜索增强语言模型（如检索增强生成）虽然可以利用外部知识弥补参数化知识的不足，但搜索并非总是有益的。模型可能会（a）对已能回答的问题调用搜索，造成资源浪费和噪声引入；（b）在应该纠正前提、要求澄清或直接弃权的问题上，反而依赖有噪声的检索证据。现有工具调用基准大多假设工具调用是合适的，而忽略了“何时不调用”的问题。
- **核心问题**：如何为每个输入实例决定“是否应该搜索”（即搜索路由），使得搜索仅在能提升任务成功率时才被调用，从而避免不必要的搜索（过度搜索）和遗漏必要的搜索（欠搜索）。
- **整体含义**：本文首次将搜索路由形式化为一个实例级的、基于反事实结果监督的二分类决策问题，并在此基础上训练路由策略，显著提升了路由准确性。

## 2. 方法论

### 核心思想
- 通过对比同一问题的**无搜索轨迹（N(q)）** 和**强制搜索轨迹（S(q)）** 的最终响应成功与否，构建一个**反事实预言机**，为每个问题分配三种标签之一：
  - **NO SEARCH**：无搜索成功（无论搜索是否也成功）。
  - **SEARCH**：无搜索失败但强制搜索成功。
  - **UNSOLVED**：两者均失败（排除出路由监督）。

### 关键技术细节
- **轨迹收集**：
  - 无搜索轨迹：禁用搜索工具，模型直接回答。
  - 强制搜索轨迹：模型必须先调用搜索工具，然后基于返回证据回答。
  - 自由策略轨迹（P(q)）：模型在选择性搜索提示下自由决定是否搜索。
- **评估**：使用任务特定的评判器（PopQA事实准确性、KUQ前提纠正/澄清/弃权）将最终响应映射为二元成功/失败。
- **训练**：
  - **监督微调（SFT）**：在预言机合格样本（NO SEARCH 或 SEARCH）上，以预言机一致的首动作为目标进行训练。
  - **偏好优化（Preference Optimization）**：对每个合格样本构造偏好对（预言机一致的动作为 chosen，相反为 rejected），使用带选择响应 NLL 正则化的 DPO 目标优化。
- **训练数据**：仅使用预言机标签为 NO SEARCH 或 SEARCH 的样本；UNSOLVED 样本用于诊断。

## 3. 实验设计

- **数据集/场景**：
  - **PopQA**：13,763 个实体中心的事实性问题，用于测试检索对长尾知识的帮助。
  - **KUQ**：1,624 个“搜索边界”问题，包含“错误假设”和“歧义”子类，需要模型进行前提纠正、澄清或弃权。
- **Benchmark**：没有直接对比现有方法，而是与**基线的自由策略（P_pre）** 进行比较，该基线是在相同选择性搜索提示下基座模型的表现。同时对比了 SFT 和 Preference Optimization 两种训练方法。
- **对比方法**：主要对比三种策略：P_pre（基座+选择性搜索提示）、SFT（监督微调）、Preference Optimization（偏好优化）。所有策略使用相同的搜索工具（Brave LLM Context）和提示模板。

  - 实验设计较为全面，涵盖了两种不同初始倾向的模型，并分离了“过度搜索”和“欠搜索”误差方向。

## 6. 主要结论与发现

- **搜索效用因实例而异**：PopQA 中搜索显著提升成功率（如 Qwen3.5-4B 从 0.1831 到 0.5096），但 KUQ 中搜索反而降低成功率（从 0.6441 到 0.5376），说明搜索不是普遍有益的。
- **训练提升搜索路由**：SFT 和 Preference Optimization 显著提升路由宏 F1 和准确率，Gemma 从 0.7082 到 0.8235，Qwen 从 0.7053 到 0.8365。
- **不同模型学会不同的路由行为**：Gemma 主要学会“减少过度搜索”，而 Qwen 进一步减少欠搜索。
- **UNSOLVED 案例揭示异质性瓶颈**：保留的硬案例中，部分可通过更大模型、扩大检索预算或后训练策略恢复，说明失败原因多样（模型容量、检索覆盖、证据利用等），不应简单归为路由错误。

## 7. 优点

- **问题定义清晰**：将搜索路由形式化为实例级决策，并基于反事实结果构造监督信号，避免了直接人工标注“搜索是否有用”的主观性。
- **方法创新**：提出从无搜索和强制搜索的对比中学习路由，且训练信号来自任务成功本身，不依赖外部知识或额外分类器。
- **实验设计合理**：同时评估两种误差方向（过度搜索和欠搜索），并保留 UNSOLVED 作为诊断集，避免将混合失败归为路由错误。
- **应用价值高**：可推广到任何搜索增强语言模型，且训练成本低（LoRA 微调）。

## 8. 不足与局限

- **依赖模型特定的预言机**：预言机标签随模型、搜索工具、提示模板等变化，不具有绝对性；不同模型之间不能直接比较。
- **仅考虑首动作路由**：未优化后续查询改写、多跳检索、证据选择等步骤，整体搜索增强系统的质量可能仍受限于这些阶段。
- **诊断分析非因果**：UNSOLVED 的救援分析（更大模型、扩大检索预算等）只是相关性诊断，不能分离独立因果因素。
- **实验覆盖有限**：仅测试了两个较小模型（4B 参数），未在更大模型（如 70B）上验证；基准只对比了提示基线，未与现有自适应检索方法（如 Self-RAG）对比。
- **数据处理**：PopQA 和 KUQ 的评判器设计存在一定主观性（LLM-as-judge），可能引入噪声。
