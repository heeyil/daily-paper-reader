## 摘要
智能体AI中的一个重大挑战是前瞻记忆：即在其他活动进行时，能在特定未来提示或状态下执行意图的能力。我们推出了PM-Bench，这是一个基于文本的基准测试，用于评估现代LLM智能体的前瞻记忆能力。受认知科学中“虚拟周”范式的启发，PM-Bench评估LLM智能体在维持用户意图、执行延迟意图以及监控潜在环境变化方面的表现。在模拟的七天一周时间内，智能体必须持续进行当前活动，同时判断是否有任何延迟任务需要处理。我们在八种不同的智能体配置下，对八个最先进的LLM在PM-Bench上进行了比较。PM-Bench在所有设置下都极具挑战性：最佳方法（GPT-5.4智能体）在我们的评估中仅达到65.1%的F1分数。此外，没有一种提升前瞻记忆的策略在所有模型中占据主导地位。我们将PM-Bench作为一个受控测试平台发布，用于诊断这些失败，并开发支持可靠前瞻行为的训练或推理时干预措施。

## Abstract
A significant challenge in agentic AI is prospective memory: the ability to execute an intention at a specific future cue or state while other activities are ongoing. We introduce PM-Bench, a text-based benchmark for measuring prospective memory capabilities in modern LLM agents. Inspired by the Virtual Week paradigm from cognitive science, PM-Bench evaluates how well LLM agents maintain user intentions, execute delayed intentions, and monitor latent environment changes. Over the course of a simulated seven-day week, agents must continue an ongoing activity while deciding whether any deferred task is due. We compare eight state-of-the-art LLMs on PM-Bench under eight different agent configurations. PM-Bench proves challenging across all settings: the best method, a GPT-5.4 agent, reaches only 65.1\% F1 score under our evaluation. Furthermore, no single strategy for improving prospective memory dominates across models. We release PM-Bench as a controlled testbed for diagnosing these failures and developing training or inference-time interventions that support reliable prospective behavior.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：LLM驱动的智能体在长期任务中频繁出现“该做的时候不做，不该做的时候乱做”的失败，根源在于**前瞻记忆（Prospective Memory）** 能力的缺失——即在持续进行其他活动时，能记住在未来特定提示或状态时执行某个意图。
- **现有不足**：已有长期记忆评估主要关注**回溯记忆**（能否回忆过去信息），而忽视了**前瞻记忆**（能否在正确时机执行之前设定的意图）。现有基准（如SWE-bench、BrowComp等）强调软件工程、浏览、规划，但未干净地隔离“意图是否在正确未来时刻执行”这一能力。
- **重要性**：前瞻记忆失败在现实生活中占日常遗忘约50%；在航空、医疗等领域可导致重大事故。因此，需要一个专门评估LLM智能体前瞻记忆的受控测试平台。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：借鉴认知科学中的**虚拟周（Virtual Week）** 范式，构建一个**文本模拟的七天日程**，智能体必须持续进行当前活动（前进剧情），同时判断是否有延迟任务到期。
- **关键技术细节**：
  - **场景结构**：七天，每天10-13个时间锚定步骤，共80步。每步提供叙事场景、三个“当前活动”选项（A/B/C前进）、一组匿名任务手柄（task_N），以及可主动查询的状态通道（时钟、邮件、日历等11个）。
  - **任务类型**：基于事件（event-based）和基于时间（time-based）的任务；跨天任务（cross-day，提前一天指示，后续某天触发）；取消/重调度/覆盖（update事件）；常态任务与非规则任务；需要监控隐藏状态通道的任务。
  - **交互协议**：每步“先查询后行动”——智能体可先发起零个或多个状态查询（如查时间、查邮件），然后提交最终行动：一个当前活动选择 + 一组选中的任务手柄子集。
  - **评分指标**：主要指标为**集合级F1（Set-F1）**，累加每步真实到期集与智能体所选集的TP、FP、FN。同时记录命中/晚执行/遗漏/误报/更新违规/依赖违规等细粒度错误。
  - **可解性验证**：所有场景通过结构校验（唯一ID、时间单调性、提示可达性、更新合法性等）和最佳策略回放审计（确保每个非取消任务在完美监控下可按时完成）。

## 3. 实验设计

- **基准场景**：一个固定的七天虚拟周（同一条场景），所有方法和模型使用同一场景。
- **模型集合**：8个前沿LLM：GPT-5.4、GPT-5.3-Codex、Llama-3.3-70B-Instruct、Mistral-Large-3-675B-Instruct、Mistral-Small-3.2-24B-Instruct、Qwen3-8B/14B/32B。
- **方法对比（8种智能体配置）**：
  1. **单智能体基线**：无外部记忆、无心跳、无子智能体。
  2. **待办清单单智能体**：维护一个上下文中的待办清单，每步更新。
  3. **可选心跳单智能体**：模型可自行决定开启或关闭周期性心跳（每30或60虚拟分钟）。
  4. **自动心跳60分钟**：系统自动每60分钟触发心跳。
  5. **自动心跳30分钟**：系统自动每30分钟触发心跳。
  6. **层级多智能体（Union-Query）**：一个协调器 + 三个子智能体（事件观察者、状态观察者、更新观察者），子智能体提出查询建议，运行时执行去重后的查询并集，协调器做最终决策。
  7. **多数投票回放†**：对同一层级痕迹，采用子智能体多数投票选择任务。
  8. **一致投票回放†**：对同一层级痕迹，采用所有子智能体一致同意才选择。

## 6. 主要结论与发现

1. **前瞻记忆远未解决**：最佳方法（GPT-5.4 + 可选心跳）仅达65.1% Macro Set-F1。
2. **无通用最优方法**：不同模型偏好不同配置；待办清单在减少误报方面最佳，心跳在召回上更好，但会带来更多误报。
3. **精度-召回权衡**：能恢复更多到期任务的方法往往更加激进（假阳性高），保守方法则欠行动。
4. **主动监控仍困难**：需要监控隐藏状态通道的任务（非时钟）表现最差，最佳通道级命中率仅16.7%。
5. **跨天任务和更新敏感任务未解决**：所有方法在这两类任务上的命中率均低于50%。
6. **层级多智能体监控开销大但收益低**：发出1661次查询（单智能体最多203次），但Set-F1仅45.2%，说明“监控多”不等于“执行好”。

## 7. 优点

- **可解性验证**：通过完美策略审计确保场景无内在矛盾，失败可归因于模型行为。
