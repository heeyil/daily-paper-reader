## 摘要
部署的LLM代理依赖于代理上下文，即由操作框架组装的外部文本控制内容。在这项工作中，该上下文的可变组件是一个持久化的系统级指令，该指令根据操作经验进行更新，而模型、工具和框架保持不变。在长时间演化过程中，随着积累的指令增加和交互，纯文本维护使得验证变得越来越困难。我们提出了图正则化代理上下文演化（GRACE），它将持久指令组件维护为一个类型化语义图，并在已修改节点的局部类型邻域内验证提议的更新。接受的图更新被重构为对部署时使用的文本指令检查点的增量编辑。我们在源自$\tau^2$-bench的固定电信代理框架内，在受控分布偏移协议下评估GRACE。在五个独立重复中，GRACE将严格可靠性（以pass^3衡量）从Gemini 2.5 Flash的零样本值0.091提升到最终检查点的0.673$\pm$0.136。这超过了在同一保留集上Gemini 3.1 Pro的零样本参考值0.242，而纯文本HCE基线最终为0.191$\pm$0.051。这些结果确定了可靠长时域上下文演化的两个需求：使验证局部化的结构基础，以及保持积累指令内容可用的整合机制。

## Abstract
Deployed LLM agents rely on agentic context, the model-external textual control content assembled by an operational harness. In this work, the mutable component of that context is a persistent system-level instruction that is updated from operational experience while the model, tools, and harness remain fixed. Over long evolution horizons, flat-text maintenance makes verification increasingly difficult as accumulated instructions grow and interact. We propose Graph-Regularized Agentic Context Evolution (GRACE), which maintains the persistent instruction component as a typed semantic graph and validates proposed updates within the local typed neighborhoods of modified nodes. Accepted graph updates are reconstructed as incremental edits to the textual instruction checkpoint used at deployment. We evaluate GRACE within a fixed telecom agent harness derived from $τ^2$-bench under a controlled distribution-shift protocol. Across five independent replications, GRACE improves strict reliability, measured by pass^3, from the Gemini 2.5 Flash zero-shot value of 0.091 to 0.673$\pm$0.136 at the final checkpoint. This exceeds a Gemini 3.1 Pro zero-shot reference of 0.242 on the same held-out set, while the flat-text HCE baseline finishes at 0.191$\pm$0.051. These results identify two requirements for reliable long-horizon context evolution, a structural substrate that makes verification local and a consolidation mechanism that keeps accumulated instruction content usable.

- **核心问题**：在长时间跨度的部署中，LLM代理依赖的**持久系统级指令**需要从操作经验中持续更新。然而，传统的**纯文本（flat-text）维护**方式随着指令增长和交互，导致内部矛盾累积、验证困难，进而引发“上下文崩溃”（context collapse）和可靠性严重下降。
- **研究背景**：先前工作（如ACE、SCOPE、Dynamic Cheatsheet）已经尝试通过增量更新、冲突解决等方式改进文本指令演化，但均未明确研究**表示基底（representation substrate）** 对验证有效性的影响。当演化周期变长时，纯文本中规则之间的关系隐含在线性顺序中，验证必须依赖长上下文处理，成本高且易失效。
- **整体含义**：论文揭示了实现可靠长时域上下文演化的两个关键要求：**使验证局部化的结构基底**（structural substrate）和**保持积累指令内容可用性的整合机制**（consolidation mechanism）。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：维护一个**类型语义图（typed semantic graph）** 作为持久指令的中间表示，仅在修改节点的**局部类型邻域（typed neighborhood）** 内进行结构验证，通过确定性操作对图进行编辑，再将接受的图变化**增量重构**为文本指令检查点供部署使用。
- **关键技术细节**：
  - **图基底**：图 $G_t = (V_t, E_t)$，节点类型 $\mathcal{A} = \{\text{identity}, \text{norm}, \text{knowledge}\}$，边类型 $\mathcal{R} = \{\text{supports}, \text{refines}, \text{sequence}\}$。网络模式 $\mathcal{T}_G$ 定义了源-目标类型的允许关系。
  - **编辑操作代数** $\mathcal{O}_{\mathcal{T}_G} = \{\text{AddNode}, \text{AddEdge}, \text{ModifyNode}, \text{RemoveNode}, \text{RemoveEdge}, \text{Merge}\}$，所有操作必须保持模式合规。
  - **演化流水线**分三阶段：
    1. **操作规划**：LLM根据诊断报告和当前图 $G_{t-1}$ 提议一组图编辑 $\Delta_t^{\text{edit}}$，应用后得到中间图 $\tilde{G}_t$，再通过关系维护模块补充关系编辑 $\Delta_t^{\text{rel}}$，生成候选图 $G_t^{\text{cand}}$。
    2. **结构验证**：对候选图进行模式合规检查，然后在**受影响的节点集 $U_t$ 的 $k$ 跳局部邻域** $H_t = N_{k^\star}(U_t, G_t^{\text{cand}})$ 内进行结构分析（SA），检测两类语义问题：
       - **矛盾（Contradiction）**：在重叠适用范围内内容不可同时满足。
       - **冗余（Redundancy）**：同类型节点间存在语义蕴含且图结构未区分。
       针对检测到的问题，通过合规操作修复，得到验证后的图 $G_t$。
    3. **增量重构**：将图变化 $L_{G_t} = G_t \triangle G_{t-1}$ 作为增量编辑应用于先前的文本指令 $\ell_{t-1}$，保留未改变的内容和格式，生成新指令 $\ell_t$。
- **公式流程**：  
  $(\tilde{G}_t, U_t^{\text{edit}}) = \text{Apply}(F_{\text{edit}}(G_{t-1}, o_t), G_{t-1})$  
  $(\hat{G}_t, U_t^{\text{rel}}) = \text{Apply}(F_{\text{relation}}(\tilde{G}_t, U_t^{\text{edit}}), \tilde{G}_t)$  
  $G_t = \text{Repair}(SA(\hat{G}_t, U_t = U_t^{\text{edit}} \cup U_t^{\text{rel}}))$  
  $\ell_t = F_{\text{recon}}(\ell_{t-1}, L_{G_t})$

- **GRACE 实现了持续累积改进**：在最终检查点 $\ell_{10}$，严格可靠性 pass^3 达到 **0.673 ± 0.136**，远超 Gemini 2.5 Flash 零样本 (0.091) 和 Gemini 3.1 Pro 零样本 (0.242)。pass@3 接近天花板（0.979）。

- **仅有图结构不足以保证持久改进**：GRACE without SA 后期同样退化（pass^3 从 0.458 降至 0.248），表明**矛盾避免**和**整合机制**两者缺一不可。
- **结构分析（SA）的作用**：有效降低矛盾计数（GRACE 与 w/o SA 矛盾数相近且明显低于 HCE），且通过检测冗余/进行合并（Merge 操作），抑制无控制的节点增长，保持图的可管理性和内容的可维护性。
- **迁移保持**：GRACE 在分布偏移后显示出正或零的**反向迁移（BWT）**，而 HCE 表现出持续的负迁移，进一步验证了结构化表示的鲁棒性。
