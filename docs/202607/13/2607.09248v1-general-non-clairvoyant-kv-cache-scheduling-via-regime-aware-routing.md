## 摘要
我们研究了在硬性键值（KV）缓存内存预算下，批量大型语言模型（LLM）推理的非窥视调度问题。每个请求有一个已知的提示长度但未知的响应长度，其内存占用包括固定的提示部分和随每个解码令牌增长的响应部分。在每个解码轮次中，调度器选择一个可行的活跃请求批次；驱逐一个请求会丢弃其累积的缓存状态，浪费先前的计算。目标是针对知道所有响应长度的最优窥视调度，最小化总完成时间。我们首次提出了在无需额外假设的情况下，对于任意提示长度和任意响应长度的常数竞争比算法。我们的算法并非依赖于单一的通用调度策略，而是基于一种新颖的状态感知路由框架构建。专门的子调度器处理不同的内存增长几何形态，而元调度器在它们之间时间共享内存预算，并根据每个作业的执行逐渐揭示其行为来动态路由。该框架还在在线到达情况下，为完工时间和总完成时间提供了常数竞争比保证。

## Abstract
We study non-clairvoyant scheduling for batched Large Language Model (LLM) inference under a hard Key-Value (KV) cache memory budget. Each request has a known prompt length but an unknown response length, and its memory footprint comprises a fixed prompt component together with a response component that grows with each decoded token. At each decoding round, the scheduler chooses a feasible batch of active requests; evicting a request discards its accumulated cache states, wasting prior computation. The goal is to minimize total completion time against the optimal clairvoyant schedule that knows all response lengths.   We present the first constant-competitive algorithm for arbitrary prompt lengths and arbitrary response lengths with no additional assumptions. Rather than relying on a single universal scheduling policy, our algorithm is built on a novel regime-aware routing framework. Specialized sub-schedulers handle different memory-growth geometries, while a meta-scheduler time-shares the memory budget across them and dynamically routes each job as its execution progressively reveals its behavior. This framework also yields constant-competitive guarantees for makespan and for total completion time under online arrivals.

#### 2. 论文提出的方法论：核心思想、关键技术细节

-   **核心思想：状态感知路由（Regime-Aware Routing）**
    1.  **问题分解**：作者发现，没有单一的全局调度规则能同时保证高内存利用率和符合面积（memory-time area）顺序的完成顺序。因此，他们根据作业的内存增长几何形态，将作业划分为三个互斥的“状态”（Regime）：
        -   **大型作业（Large Jobs）**：提示长度大于阈值Λ（例如M/4）。此类作业几乎独占内存。
        -   **提示密集型作业（Prompt-Heavy Jobs）**：提示长度≤Λ，且响应长度≤提示长度。其内存占用在常数因子内接近固定宽度（形状像矩形）。
        -   **响应密集型作业（Response-Heavy Jobs）**：提示长度≤Λ，且响应长度>提示长度。其内存占用远超提示，面积主要由响应长度决定。
    2.  **分而治之**：为每个状态设计专用的、常数竞争的**子调度器**。
        -   **大型/提示密集型作业子调度器（Rect 算法）**: 利用其内存增长有限的特性，将其抽象为“矩形条带调度”问题。使用**预算加倍贪心（Budget-Doubling Greedy）** 策略，以几何增长的尝试（attempt）长度来应对未知响应长度，并维护一个单调的等待队列以保证调度顺序。
        -   **响应密集型作业子调度器（Response 算法）**: 利用其面积与响应长度的平方成正比的特点。设计了一个**几何切片调度（Area-Geometric Slicing）** 框架，使用指数增长的相位（Phase）和代理提示长度，结合流水线调度（SPS）子例程，保证每个相位内的高利用率。
    3.  **动态路由**：由于非窥视设置下无法预先知道作业属于哪个状态，本文设计了一个**路由元调度器（Route 算法）**。它将小作业先送入提示密集型分支；如果作业在生成了si个令牌后仍未完成，则被“认证”为响应密集型，并路由到响应密集型分支。大型作业则直接进入大型作业分支。元调度器采用类似“时间片轮转”的方式，按指数增长的阶段（Stage）调度各个子调度器。

-   **关键技术细节**：
    -   **内存-时间面积（Memory-Time Area, \( A_i \)）**：定义\( A_i = s_i \cdot o_i + o_i(o_i+1)/2 \)，表示一个作业成功运行所需的累积内存-时间资源。这是一个关键的下界工具。
    -   **预算加倍**：在`Rect`算法中，每次尝试的长度按2的幂次增长，若未完成则杀死重启。这保证了一个作业的总活跃时间最多是其实际处理时间的常数倍。
    -   **SPS（Staggered Pipeline Scheduling）子例程**：一种用于调度具有相同提示和响应长度的作业的高效流水线技术，通过错开作业的起始时间来实现高并行度和内存利用率。
