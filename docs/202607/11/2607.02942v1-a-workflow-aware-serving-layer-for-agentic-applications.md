## 摘要
智能体AI应用形成了一种新兴的服务工作负载，其中每个请求都会创建一个工作流：一个由LLM和工具调用构成的有向无环图，该图暴露了每个节点的模型选择以及诸如验证器之类的可选质量算子。这种工作负载介于两个现有层之间：模型服务引擎高效执行单个调用但无法看到工作流结构，而智能体框架固定了工作流但无法看到后端负载，因此两者都无法在服务时条件下联合选择每个节点的模型、验证器和后端。我们提出了Dyserve，一个填补这一空白的工作流感知服务层。Dyserve将每个工作流中每个节点的模型和验证器选择编译成一个整数线性规划（ILP），该规划在异构后端池上运行，其定价基于跨工作流迁移的技能条件离线配置文件。这与仅通过每个模型的吞吐量扫描进入的硬件相结合，并通过加权将强模型和验证集中在错误传播最远的节点上。由于没有单一的延迟-质量偏好能适应所有工作负载混合，Dyserve在准入时在几个压力级别下预求解该规划，并在负载下在工作流的未提交后缀之间切换这些策略，使求解器远离负载切换路径；失败的工具调用触发一次性的剩余重新求解，以保留已提交的工作。

## Abstract
Agentic AI applications form an emerging serving workload in which a request creates a workflow: a directed acyclic graph of LLM and tool calls that exposes per-node model choices and optional quality operators such as verifiers. This workload falls between two existing layers. Model-serving engines execute individual calls efficiently but cannot see workflow structure, while agent frameworks fix the workflow but cannot see backend load, so neither jointly chooses each node's model, verifier, and backend under serving-time conditions.   We present Dyserve, a workflow-aware serving layer that fills this gap. Dyserve compiles each workflow's per-node model and verifier choices in one integer linear program (ILP) over a heterogeneous backend pool, priced by skill-conditioned offline profiles that transfer across workflows. This couples with hardware entering only through per-model throughput sweeps, and is weighted to concentrate strong models and verification on the nodes whose errors propagate the furthest. Because no single latency-quality preference fits every workload mix, Dyserve pre-solves the program at several pressure levels at admission and shifts a workflow's uncommitted suffix among these strategies under load, keeping the solver off the load-shift path; a failed tool call triggers a one-time residual re-solve that preserves committed work.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究背景**：大型语言模型（LLM）正被用于复杂的智能体（Agent）应用，这些应用通常需要多次LLM调用和工具调用，形成一个有向无环图（DAG）工作流。现有系统分为两层：**模型服务引擎**（如vLLM、SGLang）高效执行单个调用，但无法看到工作流整体结构；**智能体框架**（如LangGraph、AutoGen）处理工作流逻辑，但无法感知后端负载。
- **核心问题**：这两层之间存在一个空白——**没有系统能够在服务时，联合优化工作流中每个节点的模型选择、验证器选择和资源后端选择**。这种分离导致无法充分利用工作流固有的资源弹性（不同策略在延迟与质量上差异可达一个数量级），也无法在负载变化时动态调整未执行部分的策略。
- **整体含义**：论文论证了联合优化模型和验证器的必要性、拓扑位置与节点语义对节点脆弱性的影响，以及运行时负载变化会使得准入时的最优策略失效。因此，需要一个**工作流感知的服务层**来填补这一空白，在保持工作流逻辑不变的前提下，适应运行时状态。

### 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：Dyserve是一个位于智能体编排器与模型服务引擎之间的中间层。它接受逻辑工作流和可用后端列表，编译出每个节点的物理服务策略（模型+验证器+后端），并在运行时动态调整未提交部分。
- **关键技术细节**：
    1. **工作流抽象**：每个请求表示为有向无环图（DAG），节点分为LLM节点和工具节点。LLM节点具有执行选项集合（模型+后端）和验证策略集合（如自细化、自一致性、法官门等）。工具节点为固定调用。
    2. **准入时策略编译（Admission-Time Policy Compilation）**：
        - **技能条件离线画像（Skill-Conditioned Profiling）**：通过离线测试，对每个（模型，验证策略，技能类别）组合测量质量、残差错误率、调用结构和输出token需求。画像与硬件无关（硬件仅需一次吞吐量扫描），可跨工作流复用。
        - **工作流感知ILP**：构建一个整数线性规划（ILP），目标函数最大化拓扑加权质量，同时惩罚成本、失败风险和关键路径延迟。特别地：
            - 拓扑权重 \(\rho(n)\) 基于下游可达节点数，并由操作者控制的努力参数 \(e\) 调节。
            - 脆弱性系数 \(v_n\) 来自反事实故障注入实验，依赖于角色、技能和扇入数。
            - 延迟建模为每个串行阶段中最慢并行子调用的耗时之和，而非总token数。
        - 该ILP规模线性于边数和候选对，典型工作流（≤11节点，3模型，8策略）可在数十毫秒内求解。
    3. **运行时适应（Runtime Repair）**：
        - **负载适应**：为避免在线求解干扰，Dyserve在准入时对 \(K\) 个压力等级预求解ILP，得到策略梯子（Strategy Ladder）。运行时通过滞后控制器监测各模型类队列深度，仅通过指针交换切换当前策略至预解的快照，不触发求解器。
        - **工具失败恢复**：当工具调用失败时，Dyserve回溯到锚点LLM节点，将错误信息作为反馈，按“模型升级→最强验证器→重新运行最高配置”的顺序进行有限升级，并触发一次性的剩余ILP重求解（仅对未提交后缀）。

### 6. 论文的主要结论与发现

- **Dyserve实现了最高的准确率**：在四个工作负载上，准确率比最高准确率基线高出3~10个百分点，同时延迟降低了1.1~6.8倍。
- **异构集群只有在编译控制下才有益**：同构集群上编译即可获得优势；异构集群如不编译（base），性能反而下降；而Dyserve编译后，异构集群达到最优配置，准确率更高且延迟更低。
- **运行时梯子有效保护了过度承诺的计划**：在双脉冲突发下，质量偏好的准入计划（未保护）P95延迟达823秒，goodput仅18.1%；而加上预求解梯子后，P95降为136秒，goodput回升至67.2%，与最佳静态计划（73.7%）仅差6.5个百分点。对于正确调优的计划，梯子保持静默，不产生干扰。
- **工具失败恢复显著提高**：失败恢复率84%，对比平坦重试的55%和无重试的0%。
- **控制路径开销低**：准入编译中位数27ms，p95 56ms；梯子重解仅12~15ms，均在请求路径外。

### 7. 优点：方法或实验设计上的亮点

- **方法创新**：首次提出将**模型选择与验证器选择联合编译**为ILP的问题，并设计了**技能条件、硬件可移植的离线画像**，避免了重复画像。**拓扑权重与脆弱性系数**的结合比单纯拓扑更准确。**预求解策略梯子**避免了在线求解的干扰，实现了轻量级负载适应。
- **实验设计亮点**：通过**反事实故障注入**量化节点脆弱性，揭示了拓扑不足以决定节点重要性。进行**削融实验**（联合vs单轴、拓扑均匀等）验证了每个组件的必要性。**突发实验**使用了双脉冲设计和开放循环到达，避免了协调缺失（coordinated omission），结果可靠。
- **工程实现**：支持LangGraph和Flow两个编排器的适配器，仅需2.1K行代码，无需修改编译器和运行时。提供了33个适配器测试和端到端演示。

### 8. 不足与局限

- **实验覆盖有限**：服务评估仅在4-GPU单节点集群上进行（(2+2)和(3+1)异构），未测试更大规模或更复杂的跨节点集群。模型家族仅限于单一（参数规模变化），未涉及不同架构模型。
- **动态图扩展**：Dyserve仅处理已物质化的DAG（静态展开），对于动态扩展的工作流（如ReAct风格的逐步决策）需要重新编译，尚未涵盖。
- **画像依赖技能标签**：节点质量依赖人为标记的“主导技能”标签，可能不准确；论文也承认画像质量直接影响计划质量。
- **剩余完成时间近似**：负载梯子使用了全图解决的限制，而非对剩余子图精确重解，可能偏离最优。
- **与Murakkab/Orla的对比不足**：由于它们不控制模型-验证器选择，论文仅在削融实验中用单轴ILP作为替代，缺少更直接的系统级比较。
- **工具失败恢复的开销**：虽然恢复率高，但额外增加了37%的平均延迟，且论文未讨论对高失败率场景的健壮性。
