## 摘要
LLM智能体的行为不仅取决于底层模型，还取决于其框架：一个可执行程序，用于构建上下文、调用工具、验证中间结果并从失败中恢复。现有方法在部署前优化此类框架，在训练或开发数据中搜索固定的智能体工作流，并在测试时冻结。当测试分布、失败模式或工具交互与开发期间所见不同时，这会限制适应性。我们提出是否可以在评估过程中利用智能体在测试输入上生成的无标签执行轨迹来优化框架。我们引入测试时框架进化（TTHE），将可执行框架视为测试时适应的状态。评估期间，TTHE维护一个候选框架群体，并通过一个智能体提议者对其执行轨迹进行推理来改进它们，无需黄金标签或任务特定监督；然后，一个评判者根据执行衍生的代理信号提交改进的框架，选定的程序持续控制后续输入。关键的是，TTHE不更新模型权重，不需要黄金标签，也不训练单独的适应模型：求解器、提议者和评判者是围绕同一冻结LLM的不同角色和框架，因此所有适应都通过周围程序的改变发生。在文本到SQL、竞赛编程、软件工程、数据科学编码和智能体工具使用任务中，TTHE改进了固定的ReAct风格基线框架，产生了持久、可检查的改进，而不是预搜索的工作流或每次查询的重试。这些结果将LLM智能体的测试时适应重新定义为可执行控制程序的进化，并将执行衍生的代理可靠性确定为鲁棒无监督智能体改进的核心挑战。

## Abstract
The behavior of an LLM agent is determined not only by the underlying model, but also by its harness: the executable program that constructs context, invokes tools, verifies intermediate results, and recovers from failures. Existing approaches optimize such harnesses before deployment, searching training or development data for a fixed agent workflow that is then frozen at test time. This limits adaptation when the test distribution, failure modes, or tool interactions differ from those seen during development. We ask whether the harness can instead be optimized during evaluation itself, using only the unlabeled execution traces the agent produces on the test inputs. We introduce Test-Time Harness Evolution (TTHE), which treats the executable harness as the state of test-time adaptation. During evaluation, TTHE maintains a population of candidate harnesses and refines them through an agentic proposer that reasons over their execution traces, without gold labels or task-specific supervision; a judge then commits an improved harness from execution-derived proxy signals, and the selected program persists to govern subsequent inputs. Crucially, TTHE does not update model weights, require gold labels, or train a separate adaptation model: solver, proposers, and judge are different roles and harnesses around the same frozen LLM, so all adaptation occurs through changes to the surrounding program. Across text-to-SQL, competitive programming, software engineering, data-science coding, and agentic tool-use tasks, TTHE improves fixed ReAct-style baseline harnesses, yielding persistent, inspectable improvements rather than a pre-searched workflow or per-query retries. These results recast test-time adaptation for LLM agents as evolution over executable control programs and identify execution-derived proxy reliability as a central challenge for robust unsupervised agent improvement.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **背景**：现代LLM智能体通常构建为可执行系统，其行为不仅由冻结的模型权重决定，还受到周围“框架”（harness）的显著影响。框架是一个可执行的控制程序，负责构建上下文、调用工具、验证中间结果、从失败中恢复等。同样的模型放在不同的框架中，输出结果可能完全不同。
- **现有方法的局限**：大多数方法在部署前优化框架，在训练或开发数据上搜索固定的智能体工作流，并在测试时冻结。这导致当测试分布、失败模式或工具交互与开发阶段不同时，框架无法适应。
- **核心问题**：能否在评估过程中，仅利用智能体在测试输入上产生的无标签执行轨迹来优化框架？模型权重保持冻结，没有外部标签或人工修复信号，也不引入额外的适应模型。
- **整体含义**：这项工作重新定位了LLM智能体的测试时适应：适应对象不再是模型参数，而是高层的可执行控制程序；适应信号来自执行轨迹而非标注数据；适应结果是一个可检查、可复用的程序，而非一次性的响应修正。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：测试时框架进化（TTHE）将可执行框架作为测试时适应的状态。在评估过程中，维护一个候选框架群体，通过智能体提议者（proposer）推理无标签执行轨迹来改进它们，然后由评判者（judge）根据执行衍生的代理信号提交一个改进的框架，该程序持续控制后续输入。
- **关键技术细节**：
  - **框架定义**：框架是一个Python类，是任意可执行的控制程序，其搜索空间严格大于仅提示模板或固定工作流。
  - **优化信号**：无黄金标签、无参考输出、无隐藏测试。信号来自执行轨迹的详细记录（提示词、完成、工具调用、stdout/stderr、中间工件、运行时状态、错误等）以及三个轻量级代理信号：
    - 执行健康度（s∈{0,1}）：输出无错误且形状一致。
    - 往返一致性（b∈[0,1]）：输出被反转为描述并与输入匹配。
    - 公开测试通过率（p∈[0,1]）：仅当公开测试可用时。
  - **算法流程（Algorithm 1）**：
    - 输入：冻结模型M、测试流x₁…x_T、批量大小B、分支数G、轮数R、基线框架H₀。
    - 对于每个未标签批次X：
      - 执行当前框架H在X上，获取轨迹和缓存输出。
      - 从H初始化G个分支。
      - 对每个轮次r=1到R：
        - 从当前活动分支集合，每个分支的提议者编辑其父框架产生一个子框架（无效则回退）。
        - 执行新生成的子框架在X上。
      - 最终分支集合由评判者使用无标签证据（代码、轨迹、代理信号）提交一个框架作为Hₜ₊₁。
      - 使用缓存输出进行测量（仅使用黄金标签进行报告，不进入循环）。
  - **无标签循环**：求解器、提议者、评判者均由同一个冻结骨干模型驱动（主实验用deepseek-v4-flash），只是通过不同框架调用。提议者和评判者运行在Claude Code智能体框架内（但所有模型调用路由到同一冻结骨干，没有更强的教师）。
- **公式**：
  - 测量目标 J⋆(H) = E_{(x,y)~D} [ I[o_H(x) ≡ y] ]，其中I是指示函数，≡代表任务特定正确性（如SQL结果集等价、代码通过所有隐藏测试）。

## 6. 论文的主要结论与发现

- **有效性**：TTHE在所有评估领域提升了固定ReAct基线：
  - BIRD: 12.0% → 50.0%（+38.0pp）
  - LiveCodeBench: 30.0% → 38.3%（+8.3pp）
  - SWE-bench: 20.0% → 35.0%（+15pp）
  - DS-1000: 38.0% → 44.0%（+6pp）
  - claw-eval: 48.9% → 69.8%（+20.9pp，28/30任务改进）
- **跨模型泛化**：BIRD上，MiMo V2.5（32.0%→52.0%）、Kimi K2.5（28.0%→48.0%）均提升，表明方法不依赖特定模型。
- **涌现策略**：进化出的框架是可检查的源代码，揭示了可解释的策略，如值接地、验证-修复循环、复制优先调试、防瘫痪策略等。
- **非单调性**：搜索预算与性能非单调，G=1下多轮反而有害，G=3下多轮有益；批次大小在B=10时最优。
- **主要瓶颈**：选择遗憾（14个点已经在候选池中但未被提交）和覆盖率（约30%的任务从未被任何候选生成）。评判者存在校准错误，代理信号因此成为核心挑战。

## 7. 优点

- **问题定义创新**：首次将测试时适应的状态定义为可执行框架而非模型参数，开辟了新方向。
- **完全无标签**：不依赖任何黄金标签、引用答案或隐藏测试，仅在执行轨迹和轻量级代理信号上运行，实用性强。
- **持久且可检查**：进化的框架是源代码，可以审查、调试和转移，不同于黑盒权重调整或单次响应修正。
- **无需额外模型训练**：所有角色使用同一个冻结骨干模型，无教师模型、无外部修复模型。
- **广泛的任务覆盖**：涵盖结构化查询、竞赛编程、软件工程、数据科学和工具使用，展示了通用性。
- **细致的诊断分析**：通过后验oracle定位了选择遗憾和覆盖率缺陷，指出了未来改进方向。

## 8. 不足与局限

- **实验充分性**：
  - 仅单种子运行，缺乏统计显著性评估。
  - 缺乏计算匹配的基线（例如相同预算下的best-of-N采样或开发时优化器）。
  - 缺乏前向泛化评估（prequential scoring）和打乱批次的重叠分析。
  - claw-eval切片按难度排序，跨批次累积效果可能与批量局部适应混淆。
- **模型调用偏见**：提议者和评判者使用Claude Code作为智能体编码接口，但所有模型调用路由到同一冻结骨干，未比较使用不同编码接口的效果。
- **可行性限制**：
  - 仅适用于可执行的、沙盒化的任务；对于开放世界或安全关键部署，需要额外护栏。
  - 进化出的框架可能过拟合代理信号（如通过公共测试但隐藏测试失败）。
  - 搜索预算非单调增长，最优配置需要调参。
  - 每次批量进化需要多次模型调用和代码编辑，计算开销较大。
- **核心挑战**：执行代理信号并非完美；评判者常提交较差候选项，选择遗憾是主要损失源。
- **未讨论**：方法对提示词泄漏、模型骨架差异的鲁棒性；多种子平均值和方差；计算成本的具体量化。
