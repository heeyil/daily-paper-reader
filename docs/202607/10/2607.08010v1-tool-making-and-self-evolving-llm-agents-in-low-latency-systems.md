## 摘要
生产级LLM智能体常常因为每个请求都重新生成相同程序步骤的代码而浪费延迟和可靠性。我们用一个智能体工具制作流水线替代这种推理时的编码循环，该流水线在部署前将重复的标准操作程序步骤编译成经过验证、带有版本管理的工具。工具制作器在实时环境中进行合成，收集执行轨迹、观察后端模式与值、生成候选工具，并根据标注案例对其进行修复。运行时，生产智能体直接调用这些工具，仅在必要时回退到代码生成。我们在一个履行中心的警报分类系统中部署了该方法，在该系统中，智能体根据一个包含44个节点的标准操作程序对异构指标后端进行警报诊断。在生产环境中，工具调用将p50延迟降低了42%。在1500个历史警报上，通过抑制重复步骤的逐次运行方差，端到端错误率最高降低了53%。由于工具返回紧凑的结构化判定结果，它们还支持更简单的直接调用架构，在受控消融实验中进一步将p50延迟降低了62%。版本化工具还改善了可审计性，并暴露了规范差距和上游数据漂移。我们的结果表明，自进化智能体可以使工业LLM系统更快、更可靠、更易于操作。

## Abstract
Production LLM agents often waste latency and reliability by regenerating code for the same procedural steps on every request. We replace this inference-time coding loop with an agentic tool-making pipeline that compiles repeated SOP steps into validated, versioned tools before deployment. The tool-maker grounds synthesis in the live environment as it collects execution traces, observes backend schemas and values, generates candidate tools, and repairs them against labeled cases. At runtime, the production agent calls these tools directly and falls back to code generation only when needed. We deploy the approach in a Fulfillment Center alarm-triage system, where an agent diagnoses alarms against a 44-node SOP over heterogeneous metric backends. In production, tool calls reduce p50 latency by 42%. On 1,500 historical alarms, they reduce end-to-end error rate by up to 53% by suppressing run-to-run variance in repeated steps. Because tools return compact structured verdicts, they also enable a simpler direct-call architecture, reducing p50 latency by a further 62% in a controlled ablation. Versioned tools also improve auditability and expose specification gaps and upstream data drift. Our results show that self-evolving agents can make industrial LLM systems faster, more reliable, and easier to operate.

# 详细论文总结

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：生产环境中，LLM智能体在每次请求时重复生成代码来执行相同的标准操作程序（SOP）步骤，导致高延迟、高成本和运行结果的不一致（run-to-run variance）。例如，智能体每次都要重新解释同一个SOP文本、重新发现后端 schema、重写类似查询代码。
- **核心挑战**：SOP文本常缺乏执行细节（字段名、数据类型、空值处理、边界情况），代码生成容易出错。在延迟敏感的工业系统（如物流中心告警分类）中，延迟和正确性紧密耦合：故障会迅速级联，错误操作成本高。
- **整体含义**：提出一种“自进化智能体”模式——智能体离线将重复过程编译为验证过的工具，运行时直接调用工具，从而降低延迟、提升可靠性、改善可审计性，并暴露上游环境问题。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：用离线工具制作（tool-making）替代在线代码生成。每个SOP节点被编译成一个确定性的工具，返回判定（true/false/no data）和支持细节。运行时智能体直接调用工具（或由主智能体直接调用），仅当工具不可用或失败时才回退到代码生成。
- **关键技术细节**：
  - **工具制作流水线**（图1）：
    1. **数据收集（Data Collector）**：在实时环境中运行基线CodeAct子智能体，对部分训练案例执行查询，收集执行轨迹（查询代码、MCP响应、观察到的 schema、判定的正确性）。这提供了SOP文本缺失的执行细节。
    2. **工具制作（Tool-maker LLM）**：基于SOP文本、决策树位置、数据收集轨迹，生成候选工具代码（Python函数）。
    3. **测试-修复循环（Test-Repair Loop）**：在完整训练集上测试候选工具，将失败案例反馈给反射器LLM（Reflector）生成简短诊断，然后工具制作LLM根据诊断重写工具。循环最多3轮，选择训练集通过率最高的候选。工具版本化，独立于智能体模型。
  - **运行时架构改造**：基线为层次化多智能体（主智能体+子智能体），子智能体通过CodeAct循环生成代码。改进后，子智能体直接调用工具，或主智能体直接调用工具（直接调用架构），进一步消除子智能体层的延迟开销。
  - **智能体微调**：使用更强的教师模型（如GLM-4.7）生成轨迹，通过拒绝采样保留正确轨迹，对学生模型（Qwen3 32B、GLM-4.5-Air）进行LoRA微调。为了处理工具缺失情况，训练数据中随机禁用25%的工具，并包含25%全无工具轨迹，使智能体学会回退到代码生成。

## 6. 论文的主要结论与发现

1. **性能提升显著**：
   - 工具调用使生产p50延迟降低42%，子智能体输出token减少58%，代码生成token减少80%。
   - 直接调用架构（主智能体直接调工具）在消融中额外降低p50 62%（控制模型和推理框架）。
   - 端到端错误率降低36%（Qwen3）至53%（GLM-4.5-Air），工具消除因生成代码变异性导致的两类错误：偏离SOP和报告格式错误。

2. **组件必要性**：
   - 数据收集轨迹（+D）和修复循环（+R）都必要：移除修复循环降低9.6pp pass@1，移除轨迹降低5.2pp。仅全配置（SOP+D+R）超过90% pass@1（94.5%）。
   - 修复计算不能替代轨迹；轨迹中代码、元数据、评分各贡献增益。

3. **瓶颈在规范质量**：
   - 剩余误差集中于4个节点（SOP未覆盖边界条件：负分母、空值、模糊的聚合窗口）。通过补充数据和澄清SOP，pass@1从94.5%提升至99.9%（Claude Opus验证）。
   - 修复循环可能“学坏”，在训练集上拟合但偏离SOP（如用全10分钟窗口替代5分钟窗口）。

4. **数据效率**：仅5个训练样本即可达91.8% pass@1（接近全量94.5%）；无标签配置达91.6%（近3pp差距）。

5. **生产洞察**：
   - 工具日志暴露上游环境不一致（字段类型差异、定义不匹配、格式无声变更），这些在代码生成变异性下不可见。
   - 工具版本化支持快速回滚和配置更新，不依赖智能体重训练。
   - 参数化工具（阈值作为输入）可适用于不同站点。

6. **跨模型复用**：GLM-4.7生成的工具可直接服务于Qwen3 32B和GLM-4.5-Air智能体，无二次生成。

## 7. 优点：方法或实验设计上的亮点

- **创新的离线在线分离**：将重复工作从延迟关键路径移出，保留代码生成的灵活性作为回退，是一种实用工程模式。
- **环境接地（grounding）**：通过实际执行轨迹而非仅依靠SOP文本，有效解决文档-代码差距。
- **自进化循环**：工具制作、测试、修复的闭环，可自动检测失效并重新生成，降低人工维护。
- **全面的消融与统计**：细致分析各组件贡献，使用置信区间和显著性检验，可信度高。
- **生产部署验证**：不仅离线实验，还在真实生产系统运行超过5000个告警，证明实用性。
- **暴露隐藏问题**：工具确定性日志发现了之前代码生成变异掩盖的数据不一致和漂移，提升了系统可观测性。

## 8. 不足与局限

- **领域限制**：仅应用于一个特定领域（出库码头告警分类），泛化到其他SOP、runbook、或更自由的流程未经验证。论文承认“generalization to other application domains is not established”。
- **人类监督仍需**：修复循环和日志监控仍保留人类审核环节，完全自动化尚未保证。特别是澄清歧义SOP本质上是人类任务。
- **工具生成依赖标注**：新节点仍需要标注案例（尽管标签效率高且无标签方案接近但有差距），完全去除标注仍需努力。
- **模型能力要求**：工具制作需要强LLM（如GLM-4.7/Claude Opus），弱模型（Qwen3 235B、GLM-4.7 Flash）增益有限甚至退化（+D损伤性能），限制了适用性。
- **成本权衡**：工具制作是一次性token成本（800K），但+修复循环增加计算，且强模型离线成本高，对某些应用可能不划算。
- **性能指标局限**：延迟报告为归一化相对值（因商业原因未披露绝对延迟），读者难以转化为绝对数值。同时，主要关注p50/p99，未深入分析尾延迟分布。
- **未探索多智能体协作**：当前是单智能体+工具，未与其他自进化方法（如技能库、内存检索）直接比较。
