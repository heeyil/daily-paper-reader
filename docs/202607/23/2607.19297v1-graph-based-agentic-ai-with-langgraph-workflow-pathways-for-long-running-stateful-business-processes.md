## 摘要
本文是一份关于基于图的工作流路径的实践指南，适用于业务流程中长期运行、有状态、多步骤的生成式AI系统。我们不将LangGraph（一种用于有状态智能体的低级编排框架）视为模型质量的基准目标，而是提供了三个可执行的方案——带修复循环的SQL分析、带证据门控的智能体检索增强生成，以及带中断和检查点恢复的人工参与策略审查——以展示类型化状态、条件路由、确定性工具、重试、中断、检查点和追踪如何协同工作。LangGraph的定位是基于工作流复杂度的匹配，而非通用默认选择：对于基本的工具使用，更简单的ReAct风格或纯SDK循环可能更好；对于结构化提取和验证，模式优先的工具更合适；而当提示或程序优化是主要目标时，DSPy更优。每个方案都解释了何时LangGraph值得额外的结构，以及哪些实现模式能使路由、暂停和审计轨迹成为明确的产品行为，而非隐藏的提示逻辑。

## Abstract
This paper is a practitioner guide to graph-based workflow pathways for long-running, stateful, multi-step generative AI systems in business processes. Rather than treating LangGraph, a low-level orchestration framework for stateful agents, as a model-quality benchmark target, we present three executable recipes -- SQL analytics with repair loops, agentic retrieval-augmented generation with evidence gating, and human-in-the-loop policy review with interrupt and checkpoint recovery -- to show how typed state, conditional routing, deterministic tools, retries, interrupts, checkpoints, and traces fit together. LangGraph is positioned by workflow-complexity fit, not as a universal default: simpler ReAct-style or plain SDK loops may be better for basic tool use, schema-first tools for structured extraction and validation, and DSPy when prompt or program optimization is the main goal. Each recipe explains when LangGraph is worth the extra structure and which implementation patterns make routes, pauses, and audit trails explicit product behavior rather than hidden prompt logic.

## 2. 论文提出的方法论：核心思想、关键技术细节、算法流程

### 核心思想
- **图式工作流**：将多步LLM应用表示为有向图（StateGraph），每个节点执行一个原子步骤，条件边根据状态决定下一步，通过类型化状态对象在节点间传递数据。
- **显式路由**：将控制流从提示词中解放出来，放入可检查、可测试、可审计的图结构中。路由决策由应用逻辑（如错误检查、证据评分、风险等级）而非模型文本决定。
- **适配性决策**：LangGraph不是万能方案，应根据工作流复杂度选择：简单工具使用用ReAct/SDK循环；结构化提取用模式优先；提示优化用DSPy；只有需要状态持久化、分支、修复、人工暂停时才用LangGraph。

### 关键技术细节
- **核心原语**：Typed State（类型化状态）、Nodes（节点）、Conditional Edges（条件边）、Checkpointer（检查点）、Interrupt（中断）。
- **路由函数**：小型、无状态函数，根据当前状态返回路由标签（如"execute"、"retry"、"fail"、"interrupt"），使路由决策可测试。
- **节点边界稳定原则**：节点边界对应产品事件（如模式查找、SQL验证、文档评分、风险评分、中断审查），每个节点应可成为审计日志、重试规则或仪表盘的单元。
- **检查点机制**：使用`InMemorySaver()`（开发）或SQLite/Postgres（生产）持久化线程状态，支持跨进程重启后的恢复。

## 6. 论文的主要结论与发现

1. **LangGraph的价值在于显式路由而非模型质量提升**：图式编排使验证错误、弱证据、高风险决策等路由成为可见的、可测试的产品行为，而非隐藏在提示文本中。
2. **决策树选择框架**：根据工作流需求选择最简方案——线性任务用SDK循环，结构化输出用模式优先，提示优化用DSPy，只有需要状态持久化、分支恢复、人工暂停时才用LangGraph。
3. **节点边界与产品事件对齐**：稳定的节点边界对应审计日志、重试规则、仪表板中的事件，是构建可维护工作流的关键。
4. **检查点对HITL至关重要**：去掉检查点会破坏中断/恢复能力，导致高风险决策无法完成。
5. **路由函数应保持小型**：路由函数只读取状态并返回小标签，使图可检查；模型负责内容生成，应用逻辑负责路由决策。
6. **三个食谱可执行且可审计**：提供完整代码（SQL分析、智能体RAG、HITL策略审查）验证了模式的有效性。
