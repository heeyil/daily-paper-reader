## 摘要
大型语言模型及基于LLM的智能体被广泛用作个人聊天助手、企业副驾驶和自主工作流智能体。在这些应用中，记忆（即保留、访问和推理在长上下文及多次交互中积累信息的能力）在决定智能体可靠性方面起着关键作用。我们提出RECON（基于混淆叙事的长上下文推理基准），用于评估长上下文中的组合推理。RECON涵盖三个领域（刑事、医疗和金融）的24个案例文件，每个文件长度为5万至10万token，并在六项记忆密集型任务上测试智能体：重建多跳证据链、传播级联失效、解决源冲突、反事实推理、满足时间约束以及时间事实检索。最近的记忆基准测试评估智能体是否能检索分散的事实或检测事实是否发生变化，而RECON评估变化发生后会发生什么：智能体是否能追踪哪些下游结论受到影响、哪些通过独立支持而幸存，以及替代时间线本会如何展开。我们的评估揭示了当前架构的重大局限性：即使最强的非Oracle系统准确率也仅达到22.4%，检索和推理各自构成了挑战。

## Abstract
Large language models and LLM-based agents are widely used as personal chat assistants, enterprise copilots, and autonomous workflow agents. In all these applications, memory (the ability to retain, access, and reason over information accumulated over long contexts and multiple interactions) plays a crucial role in determining the reliability of any agent. We introduce RECON (Reasoning over Extended Contexts with Obfuscated Narratives), a benchmark for evaluating compositional reasoning over long contexts. RECON spans 24 case files across three domains (criminal, medical, and financial), each ranging from 50k to 100k tokens, and tests agents on six memory intensive tasks: reconstructing multi-hop evidence chains, propagating cascading invalidations, resolving source conflicts, counterfactual reasoning, satisfying temporal constraints, and temporal fact retrieval. Recent memory benchmarks evaluate whether agents can retrieve scattered facts or detect if a fact has changed whereas RECON evaluates what happens after the change, whether agents can trace which downstream conclusions are affected, which survive through independent support, and how alternative timelines would have unfolded. Our evaluation reveals substantial limitations across current architectures: even the strongest non-Oracle system reaches only 22.4% Accuracy, with retrieval and reasoning each surfacing as challenges.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：现有 LLM 代理（如 Claude Code、ChatGPT、企业副驾驶）需要在长上下文和多轮交互中维持一致、演化的理解，然而当前记忆基准（如 LongMemEval、LoCoMo）仅评估**事实检索**或**变化检测**（即“事实是否改变”），缺乏对**变化后推理能力**的评估——例如追踪哪些下游结论被级联无效化、哪些通过独立支持幸存、替代时间线如何演化。
- **整体含义**：记忆能力直接决定代理在临床、法律、金融等高风险场景的可靠性。RECON 首次将记忆建模为**溯源图（provenance graph）**而非状态机，强调结论具有推导历史，以评估代理在**事实交互、矛盾、级联**场景下的组合推理能力。

## 2. 方法论：核心思想、关键技术细节、流程
- **核心思想**：通过**确定性生成管道**构建包含显式依赖结构的案件文件，保证所有答案均可从有向无环图（DAG）计算得出，LLM 仅参与受限的表面语言实现，不参与因果结构或答案键定义。
- **技术细节**：
  - **属性条件蓝图合成**：采用类文法引擎 `G = (D, A, P, C)`，其中 D 为领域目录，A 为属性集合，P 为带前置条件/建立后件的产生式规则，C 为全局不变量（如时间单调性）。规则按需触发，生成蓝图。
  - **骨架扩展与时间强制**：将蓝图展开为带时间戳的事件、证据、因果依赖、无效化、源冲突、反事实依赖、并行时间流（监控、访问、交易等）。所有节点 `τ(v) = (day, time)` 满足偏序关系。
  - **溯源 DAG 与证明迹**：全局 DAG 的节点为事件、证据、结论，边表示因果/修订/无效化/冲突解决关系。每个问题附带一个最小生成子图作为证明迹，确保答案可推导。
  - **确定性任务合成**：算法化地从 DAG 生成六类问题（链重构、级联传播、源冲突、反事实、时间约束、时间事实检索），并分配固定配额（表 4）。
  - **语言实现**：LLM 将骨架条目转换为调查风格叙事，但强制保留时间戳、实体 ID、证据内容等不可变字段，事后交叉验证。
- **流程**：种子配置 → 属性条件蓝图 → 结构化骨架 → 溯源 DAG → 任务合成 → 叙事实现 → 验证（6 阶段：正则检查、结构验证、溯源验证、LLM 语义审计、鲁棒性审计、人工验证）。

## 6. 主要结论与发现
- **整体表现**：RECON 对所有架构均构成挑战。最强非 Oracle 系统准确率仅 22.4%（Gemini-2.5-Pro）；Oracle 也仅达 54.6%，表明即使完美检索下推理仍存瓶颈。
- **任务差异**：
  - **级联传播**：记忆系统（Supermemory）表现优于长上下文（0.708 vs 0.361），因其擅长追踪独立支持。
  - **反事实与时间约束**：所有系统表现最差，Oracle 也仅 48.3% 和 38.4%，说明链式推理是主要瓶颈。
  - **时间事实检索**：长上下文和 RAG 优于 Oracle，因结构化表示抽象掉了叙事细节。
- **检索 vs 推理**：80% 的失败发生在检索成功时，推理是主导瓶颈；但源冲突、链重构等任务中检索仍是主要障碍（Oracle–LLM 差距达 59–64 pp）。
- **token 效率**：RAG 和记忆系统比长上下文节省 8–20 倍 token，但仍无法匹配长上下文在多源推理任务上的表现。
- **链长与准确率负相关**：Oracle 准确率从链长 3–4 的 100% 降至链长 10 的 50%，非 Oracle 系统几乎在所有链长上接近零。

## 7. 优点（方法或实验设计）
- **确定性生成管道**：所有结构、依赖、答案均由代码定义，LLM 仅做语言实现，确保答案可追溯、可复现，避免 LLM 引入的无序性。
- **任务覆盖全面**：六类任务覆盖了记忆密集型组合推理的核心模式，远超现有基准（仅检索或状态跟踪）。
- **人类验证**：200 问题三盲标注，Fleiss κ=0.69，86.3% 匹配基准答案，证明题目合理可行。
- **鲁棒性审计**：匿名化、时间戳归一化、DAG 安全扰动测试确保性能不依赖表面线索。
- **污染过滤**：11.8% 问题因可闭卷作答被移除，提高评估纯净度。
- **开源与可复现**：基准、评估框架、生成代码全部公开。

## 8. 不足与局限
- **合成数据**：案件叙事由 LLM 在确定性骨架上生成，虽然受约束，但仍可能引入风格偏差。模型若在类似合成叙事上训练可能获益。
- **模式泄漏**：了解内部骨架模式的结构化提取管道可能通过答案格式而非推理作答。
- **人类基线输入不对称**：人类标注者获得的是问题相关证据包（比完整文件更聚焦），而 LLM 需要处理整个 100k 文件或自检索上下文，因此人类准确率（63.0%）不能直接同等比较。
- **领域与规模有限**：仅三个领域 24 个案例，每个案例 50k–100k tokens，未覆盖更长（百万 token）或更多样化的场景（如源代码、科学研究）。
- **评分设计**：多项选择题的负分惩罚可能过度惩罚低信心模型，但作者通过 Abstain% 提供了补充视角。
- **记忆系统评估**：仅使用两种 LLM 作为求解器，且依赖云服务（Mem0、Supermemory），其版本变化可能影响结果稳定性。
