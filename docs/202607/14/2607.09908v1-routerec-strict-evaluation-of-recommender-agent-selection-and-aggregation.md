## 摘要
推荐系统越来越面临在异构代理（协同过滤、序列模型、基于内容的检索器和基于LLM的重排序器）之间进行选择的问题，然而没有一个代理是普遍最优的。我们使用RouteRec框架将这种选择视为在成本约束下的任务感知代理排名，该框架比较了请求级别的硬选择与四个传统推荐代理和一个LLM重排序代理上的项目级别学习聚合。在MovieLens-1M上，完整质量预言机具有显著的提升空间（HR@10 = 0.584），证实了跨代理信号的存在。然而，在无泄漏的五折交叉验证协议下，硬选择仍然低于BM25（0.223对比0.254），且选择性LLM升级并未改善这一结果。同样的协议对学习聚合产生了不同的结果：其仅廉价变体在HR上与BM25持平，并具有更高的NDCG点估计（0.123对比0.114），而门控全代理聚合达到HR@10 = 0.295，其中70.2%的调用使用了LLM。由此得出的教训不是路由问题已解决，而是对于这种稀疏固定候选场景，请求级别选择一个完整代理列表过于粗糙；项目级别聚合是更有前景的行动空间。

## Abstract
Recommender systems increasingly face a choice among heterogeneous agents -- collaborative filters, sequential models, content-based retrievers, and LLM-based rerankers -- yet no single agent is uniformly best. We study this choice as task-aware agent ranking under cost constraints using RouteRec, a framework that compares request-level hard selection with item-level learned aggregation over four traditional recommender agents and one LLM reranker agent. On MovieLens-1M, the full quality oracle has substantial headroom (HR@10 = 0.584), confirming that useful cross-agent signal exists. Under a leakage-free 5-fold out-of-fold protocol, however, hard selection remains below BM25 (0.223 vs. 0.254), and selective LLM escalation does not improve it. The same protocol yields a different outcome for learned aggregation: its cheap-only variant matches BM25 in HR and has a higher NDCG point estimate (0.123 vs. 0.114), while gated all-agent aggregation reaches HR@10 = 0.295 with 70.2\% LLM calls. The resulting lesson is not that routing is solved, but that request-level selection of one complete agent list is too coarse for this sparse fixed-candidate setting; item-level aggregation is the more promising action space.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究背景**：推荐系统面临异构代理（协同过滤、序列模型、内容检索、LLM 重排序器等）的实际部署选择。但单一代理无法在所有请求上都最优，LLM 虽强但成本高，且并非总优于传统方法。
- **核心问题**：**“如何根据每个请求（task-aware）在成本约束下做出正确的代理选择？”** —— 即从“哪个模型最好”转向“哪个模型应处理这个请求”的代理路由问题。
- **整体含义**：现有工作多关注“总有一个更好”，但忽略了路由策略的**可学习性**与**决策粒度**。本文旨在严格评估两种路由范式：请求级硬选择（完整列表选一）与物品级学习聚合（跨代理证据融合），并揭示其在不同条件下的有效性。

## 2. 方法论：核心思想、关键技术、流程

### 2.1 核心思想

- 将路由问题建模为**任务感知的效用最大化**，效用函数综合推荐质量（NDCG、HR）与成本（延迟、token 消耗）。
- 设计两个变体：
  - **RouteRec-Select**：两阶段硬选择（先选最佳廉价代理，再决定是否升级到 LLM）。
  - **RouteRec-Stack**：物品级学习聚合（对候选物品进行二次排序）。

### 2.2 关键技术细节

#### 代理池（5个代理）
- 廉价代理（A_C）：Popularity、BPR-MF、SASRec、BM25。
- 昂贵代理（A_E）：Qwen-2.5-7B-Instruct 重排序器（默认）；消融实验使用 grok-4-1-fast-reasoning。

#### 请求编码
- 提取 16 维特征（用户历史长度、活动度、类别熵、长尾比例、指令有无等），通过 2 层 MLP 得到 32 维嵌入 h_x。

#### 代理能力编码
- 手工设计 12 维能力向量（信号类型、指令支持、场景优势等），通过 MLP 得到 16 维嵌入 e_a。

#### 廉价探测模块（Cheap Probe Module）
- 在路由前执行三个廉价代理（SASRec、BPR-MF、BM25）的**诊断探测**，计算 15 维分歧特征 d_x（重叠、Jaccard、Kendall τ、熵、多样性等）。
- **直觉**：代理一致时任务简单，分歧大时可能需要更强大的代理。

#### 两阶段路由器
- **Stage 1：廉价选择器**  
  计算每个廉价代理的兼容性分数 s_c(x,a)，选择分数最高的廉价代理 a*_c。
- **Stage 2：升级门控**  
  估计升级边际效用 g(x) = σ(f_g([h_x; d_x]))，若 g(x) > τ 则使用 LLM 代理，否则返回 a*_c。
- **训练损失**：L = L_rank + η·L_esc，其中 L_rank 是廉价代理的成对排序损失（使用纯质量标签），L_esc 是升级门的二元交叉熵（基于验证集效用标签）。

#### 学习聚合（Stacking）
- 形成去重候选集（各代理 top-10 的并集），使用可部署特征（代理成员、倒数排名、归一化分数、流行度等），训练逻辑回归或梯度提升分类器。
- 变体：StackCheap（仅廉价列表）、StackAll（含 LLM 列表）、StackGate（用廉价堆叠置信度决定是否调用 StackAll）、StackBudget 系列（限制 LLM 调用比例）。

## 3. 实验设计

### 3.1 数据集
- **MovieLens-1M**（6,040 用户，3,706 物品，1M 评分）

### 3.2 评估协议
- **严格的无泄漏 5 折 OOF**（out-of-fold）：每折训练 350 用户，验证 50 用户，测试 100 用户。重复 3 个种子取平均±标准差。

- **融合基线**：RRF（倒数排名融合）、Borda、ScoreAvg（分别有廉价版和全代理版）。
- **选择基线**：随机、最优固定单代理、仅请求特征、独热代理。
- **质量上界**：CheapOracle（按实例选最佳廉价代理）、Oracle（最佳代理含 LLM）。

## 6. 主要结论与发现

1. **预言机存在提升空间**：CheapOracle HR=0.508，Oracle HR=0.584，说明跨代理互补信号确实存在。
2. **硬选择无法利用互补性**：
   - RouteRec-Select 廉价版 HR=0.223，低于 BM25 基线 0.254；
   - 加入升级门控后 HR=0.215，LLM 调用 10.4% 反而下降，无法可靠识别 LLM 有用的案例。
3. **学习聚合效果更优**：
   - RouteRec-StackCheap 匹配 BM25（HR=0.254，NDCG 略高 0.123 vs 0.114）；
   - 预算化堆叠（StackBudget@25）以 13.4% 的 LLM 调用达到 HR=0.262；
   - StackGate 以 70.2% 调用达到 HR=0.295，超越所有单代理；
   - StackAll（100% LLM）HR=0.299，仍未达到 CheapOracle 水平。
4. **决策粒度是关键瓶颈**：请求级完整列表选择过于粗糙；物品级聚合可恢复部分丢失的跨代理信号。
5. **升级门控不可靠**：在有限样本下无法学到有效的 LLM 触发策略。

## 7. 优点

- **问题定义清晰**：明确将路由分为硬选择与聚合两种范式，并引入严格效用函数统一比较。
- **实验设计严谨**：采用无泄漏 5 折 OOF、多种子、报告置信区间；区分可部署信号与预言机标签，避免特征泄漏。
- **方法轻量可复现**：模型仅 11K 参数，CPU 秒级训练，利于复现和扩展。
- **提供了关键洞见**：否定“路由已解决”的假设，指出“物品级聚合优于请求级选择”的设计原则。
- **公开所有代理输出日志**（缓存 LLM 推理），消除随机性，保证评估确定性。

## 8. 不足与局限

- **数据与场景局限**：仅 MovieLens-1M 一个数据集，500 用户；固定候选池 SASRec 生成，限制了 LLM 的发挥（目标暴露率仅 36.4%）；不适用于全检索或开放域。
- **监督信号稀疏**：留一法导致每用户仅一个正例，多数代理得分为零，路由学习的统计能力严重不足。
- **消融差异不显著**：特征组移除后的 HR 变化极小（±2-3%），无法可靠判定特征重要性；论文自己也承认这点。

- **未评估在线或实时场景**：离线 OOF 协议未考虑用户反馈循环和动态延迟约束。
