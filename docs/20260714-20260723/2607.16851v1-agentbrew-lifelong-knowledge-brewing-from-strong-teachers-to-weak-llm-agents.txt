## Abstract 

Deploying LLM agents typically requires a compact test-time student, even if a stronger teacher is available during training. We study knowledge brewing: distilling a teacher’s interactive experience into a persistent external memory for the student. Crucially, this requires no weight updates, expert demonstrations, ground-truth labels, or test-time teacher access. This setting poses two challenges: environments provide only sparse, binary feedback, and teacher-authored notes must be inherently tailored to be concretely executable by a substantially weaker student. To address these hurdles, we propose AgentBrew , comprising two cou-pled components. First, a failure-triggered teacher–Ralph Loop mitigates sparse feedback by transforming student failures into environment-validated notes. Sec-ond, student-aware synthesis calibrates teacher knowledge to the weak executor’s operational granularity, yielding model-specific, actionable guidance. Extensive evaluations and comprehensive ablations across coding, math, and tool-use tasks demonstrate that this asymmetric, training-free brewing paradigm produces highly capable yet deployable LLM agents. 

## 1 Introduction 

Large language model (LLM) agents now solve interactive tasks in coding, math, and tool-use environments [ 1– 4]. In production, however, the agent that must serve requests is often a small or API-only model: fine-tuning may be unavailable, inference cost must stay low, and model weights are treated as fixed after deployment. A natural alternative is teacher–student agent brewing 1: a stronger teacher agent T (large LLM) distills interaction experience during a training period, while a compact student agent S (small LLM) runs alone at test time with only retrieved external knowledge. The central question is how to convert brewing-time interaction into durable notes that raise S’s effective capability without updating its parameters. Two challenges make this setting substantially harder than either self-evolution or standard distillation alone. Challenge I: sparse environment feedback. Real benchmarks and deployed APIs typically expose only a terminal binary reward R ∈ { 0, 1}—pass or fail—together with the agent’s own rollout, but no ground-truth solution, expert trajectory, or step-level label. When S fails, the only extra signal is the failure trajectory τS (x); when it succeeds, no teacher intervention is needed. Challenge II: teacher–student capability gap ( i.e. , knowledge is model-specific). In the deployment scenario above, T is intentionally much stronger than S, yet the notes stored in memory must be written by T and executed by S. Existing work addresses only parts of the problem. Self-evolving agents [ 5 , 6 ] can im-prove from environment feedback without weight updates, but they usually (i) evolve the same model that both acts and learns, (ii) continue adapting at test time, and (iii) are not built for asymmetric teacher-student "brewing", where notes must be validated for a fixed weak executor. Agent distilla-tion [ 7, 8 ] can compress teacher behavior into trajectories or reusable artifacts, but it typi-cally depends on labeled pairs or successful teacher rollouts, and it transfers knowledge at a teacher-level abstraction rather than as student-executable notes certified to work on the weak executor. Accordingly, we organize our investigation around three questions: 

Q1 (Learning signal). When Env exposes only binary R and no expert data, how can a weak agent extract useful knowledge from failures, and how can we verify that knowledge before storing it? 

Q2 (Capability gap). How can a strong teacher author knowledge that a substantially weaker student can execute—not merely parse—given that both generation and validation are biased by the gap? 

Q3 (Lifelong accumulation). How can validated, student-calibrated knowledge accumulate across tasks in external memory-without updating student weights S-while keeping test-time inference lightweight and stable? We propose a brew–serve framework (Figure 2) that maps Q1–Q3 to three coupled components: a failure-triggered teacher–Ralph Loop , student-aware note synthesis , and a lifelong memory M.We call it AgentBrew : brewing is the Ralph Loop [ 9] that iteratively distills certified notes from failures, and student-aware synthesis tailors each note to its executor as a brewer adapts method to the ingredients. Brewing on Dtrain proceeds as follows. Student S rolls out each task in Env ;when R=0 , teacher T reads (x, τ S (x)) —the only supervision beyond binary pass/fail—and drafts a structured candidate note. The note is authored with student-aware prompting and an action-oriented schema so that T targets S’s reading level and operational granularity rather than its own. The Ralph Loop then stages the note and re-rolls S on the same task: recovery ( R=1 ) certifies the note using environment feedback alone, and rejects guidance that remains plausible to T yet unusable for the weak executor. A curator deduplicates certified notes and appends them to M, which grows across the training stream without updating S. Serving on Dtest reads from frozen M⋆: T is inactive, 

S retrieves skill-scoped top-k notes, and completes each task in a single rollout with no iterative reflection or online memory writes. AgentBrew is guided by a simple principle: knowledge is not universal—it must fit the agent that executes it . Our main contributions are: • Brew–serve under sparse feedback. We develop a lifelong knowledge-building method for a fixed weak agent using only binary feedback from the environment, without expert trajectories, test-time labels, or any weight updates. • Student-calibrated transfer. We introduce student-aware note synthesis that proactively authors teacher knowledge at S’s operational granularity, paired with Ralph recovery as a reactive filter that certifies notes on the weak executor rather than on teacher plausibility alone. • Comprehensive Evaluation. (Q1) a failure-triggered teacher turns sparse binary feedback from Env into structured candidate notes, with a curator merging environment-certified entries into M (Sec. 2.3); (Q2) reactive Ralph validation plus proactive student-aware synthesis bridge the capability gap between teacher and student agents (Sec. 2.4); and (Q3) a lifelong memory M accumulates validated notes across tasks without updating the student LLM (Sec. 2.5). Here we summarize how they interact:

Given a new task x, S retrieves the top-k notes from frozen memory M⋆ (via skill-scoped routing 3and reranking), prepends them to its prompt, and completes x in a single environment rollout-without an iterative reflection loop or additional validation calls. If retrieval returns no notes, S falls back to its vanilla policy. 

2.3 From Binary Feedback to Validated Knowledge 

The brewing phase must extract durable knowledge from failures when Env exposes only a terminal reward R ∈ { 0, 1}: no step-level labels, no expert trajectories, and no oracle explanations. We address Q1 by treating each failure as a supervision event. Here we detail the failure-triggered activation pipeline, the note representation written by T , and curation before durable writes.

During brewing, each task x ∈ D train begins with a rollout by student agent S, producing τS (x) and reward R ∈ { 0, 1}. Binary R is the sole external supervision signal from Env : no step-level labels, no expert trajectories, and no oracle explanations are provided. If R = 1 , the pipeline advances to the next task without invoking T . If R = 0 , we trigger T on the tuple (x, τ S (x), FAIL ). Unlike Supervised Fine-Tuning (SFT), T never conditions on a reference solution; it diagnoses the failure mode from the student trajectory and task text alone, then proposes a corrective note. This design makes learning possible in settings where only pass/fail feedback is available. Algo. 1 summarizes the control flow. 

2.3.2 Structured Note Generation 

Given a failure, T outputs a candidate note n as structured JSON rather than a worked example. Each note is an abstract, reusable rule intended for retrieval on future tasks with similar structure: • note_key : unique identifier (reused when updating an existing rule); • trigger_pattern : task features that should activate the note; • corrective_rule : an executable fix directive for S;• minimal_steps : a short ordered checklist lowering interpretation cost for a weak LLM; • tags : domain or skill labels for routing; • quality_score : initial confidence, later updated from validation statistics. Notes deliberately avoid storing full solutions to x; they compress failures into transferable guardrails. Granularity and wording are calibrated to the student agent’s capability. 

2.3.3 Note Curator 

When the Ralph Loop returns validated = TRUE , Algo. 1 passes the environment-certified note to curator agent C before any durable write. C does not re-judge factual correctness—recovery in Env 

already certifies utility to S on the brewing task x. Its role is strictly operational on the memory store :4(i) Deduplication: compare n against existing entries in M via lexical and semantic similarity; merge near-duplicates by updating an existing note_key instead of appending redundant rules. 

(ii) Quality tracking: adjust quality_score from historical validation outcomes (acceptances, recoveries, and downstream usage). 

(iii) Write gating: persist n to M only if it passes redundancy and quality thresholds; otherwise discard despite a successful Ralph Loop round. Accepted notes are appended under an append-only policy, extending M without overwriting prior knowledge or modifying the student LLM. 

2.4 Bridging the Capability Gap 

Teacher agent T is substantially stronger than student agent S, leading to the capability gap [ 10 ]between to agents. We address Q2 with a two-stage design. First, reactive validation : the 

Ralph Loop [9] re-rolls S on the failed task with a candidate note staged in memory and accepts 

n only when Env returns R=1 , using terminal reward—not teacher or student self-judgment—as the certification signal. Second, proactive synthesis : student-aware prompting and note schema calibrate what T writes before validation, so proposals match S’s reading level and operational granularity. Ralph provides the environment-grounded filter; proactive synthesis improves validation yield and transferability. 

• Why same-level generation fails. Prior work on knowledge transfer, such as ACE [ 11 ], assumes that the knowledge producer and consumer have comparable reasoning ability. When T is substan-tially stronger than S, this assumption breaks down [ 12 , 13 ]: notes often omit intermediate reasoning, use mismatched vocabulary, or specify steps at a granularity too coarse for S to follow. Such notes may appear plausible to T yet leave S unable to recover. 

2.4.1 Reactive Validation: Ralph Loop 

Teacher-generated notes can be plausible yet useless or harmful to S. Self-judgment [ 5] by T or 

S is insufficient because capability gaps bias both generation and critique. The Ralph Loop is the environment-validation stage of brewing: it asks whether a candidate note actually helps S recover on the failed task, using only terminal rewards from Env .

Provisional staging. After T proposes n, ADD PROVISIONAL (M, n ) places n in a provisional 

slot—visible to RETRIEVE (M, x ) during re-attempts on x, but not yet part of durable long-term memory. Provisional entries simulate “what if this note were already in M?” for the validation rollouts; ROLLBACK PROVISIONAL removes the slot if validation fails, and COMMIT promotes a curated note only after C approves persistence.

2.5 Lifelong Memory 

AgentBrew addresses Q3 by treating external memory M as the sole durable state: validated notes accumulate across Dtrain , and student agent S reads from a frozen M⋆ at test time. No gradient updates or adapter weights are applied; effective capability grows because M grows. We structure M

to limit cross-domain interference, append new knowledge without erasing old entries, and retrieve notes with a serving pipeline tuned for single-pass inference. 

2.5.1 Skill-Scoped Memory Architecture 

A monolithic note pool induces cross-scope interference : notes from one skill are retrieved on unrelated tasks. We therefore partition M into disjoint skill scopes {M s}s∈S , each indexing notes for a single capability or environment slice. Every note stores tags and an optional scope identifier; during curation, the teacher writes it to the corresponding Ms. At test time, retrieval first selects a scope subset Mscope ⊆ M and ranks notes only within that subset-never over the full pool. Scope routing is orthogonal to note ranking. When x does not clearly specify a scope, we choose 

Mscope using three signals, in order: (i) explicit scope or tool names in x; (ii) lexical or semantic similarity between x and a scope-description catalog maintained alongside M; (iii) actions observed in the student rollout τS (x) when (i)–(ii) are inconclusive. 62.5.2 Continual Accumulation and Stability 

During brewing, accepted notes are merged with append-only semantics: curator deduplication may update an existing note_key , but validated rules are never deleted to make room for new ones, avoiding catastrophic forgetting of earlier skills. Each note record tracks usage statistics and 

quality_score so frequently harmful or redundant entries can be down-weighted at retrieval time without erasing history. Train and test streams are strictly separated. While processing Dtrain , M is writable. On Dtest ,

M⋆ is frozen read-only : no teacher calls, no provisional writes, and no statistic updates that leak test-task content back into the store. This guarantees that performance on held-out tasks measures generalization from brewed knowledge, not online adaptation to test labels or test-task feedback.

The top-k notes are formatted and prepended to S’s prompt; S then performs one ROLLOUT in Env . If retrieval returns an empty set, S falls back to its vanilla policy with no memory augmentation. 

## 3 Evaluation 

3.1 Experimental Setup 3.1.1 Tasks and Benchmarks 

We evaluate AgentBrew across three representative agent scenarios— coding , math , and tool-use —each probing a distinct capability profile: synthesizing executable programs from natural-language specifications, performing multi-step symbolic reasoning, and completing long-horizon tasks through interactive API calls. More details are in App. A.2. 

• For the coding tasks, we evaluate AgentBrew on MBPP [14 ], a benchmark of roughly 1,000 entry-level Python programming tasks, and MBPP+ [ 15 ], an extension that applies stricter unit testing to a curated subset to reduce spurious passes. 

• For the math tasks, we evaluate AgentBrew on MATH [16 ], comprising challenging competition-level problems, and GSM8K [ 17 ], which tests multi-step grade-school arithmetic reasoning. We score final-answer accuracy using the Qwen2.5-Math [18] evaluation framework. 

• For the tool-use tasks, we evaluate AgentBrew on AppWorld [ 2 ], an interactive benchmark where agents write and execute Python code to complete realistic, multi-app daily tasks in a simulated environment with multiple APIs. 

3.1.2 Models, Baselines and Metrics 

• LLM Models. In our experiments, we use deepseek-chat-v3.1 [ 19 ] by default for the teacher agent and qwen3-14b [20] by default for the student agent. 

• Baseline Methods. The main baseline methods we compare against include: Simple RAG ,

ExpeL [ 6], Reflexion [5], ACE [11 ], ReAct [21 ], and LoRA [ 22 ]. Among these, Simple RAG refers to saving the successful trajectories of the teacher agent during training, and directly retrieving these trajectories during testing to help the student agent complete the task. 

• Evaluation Metrics. We use accuracy for math tasks, the strict EvalPlus [ 15 ] benchmark for coding tasks, and Task Goal Completion (TGC) for tool-use tasks ( i.e. , AppWorld) as our evaluation metrics. 73.2 Main Performance Evaluation

(i) Strong-teacher brewing vs. student-only reflection. ReAct and Reflexion (Table 1, Groups I–II) rely entirely on the weak student at test time: ReAct acts in a single pass, while Reflexion adds multiple reflection rounds that re-attempt each task after observing explicit binary feedback from the environment. The Reflexion setup with k> 1 is therefore a stronger inference protocol than our single-rollout serving (Group III), as it consumes additional test-time trials and outcome signals that are unavailable to all other methods. Even so, AgentBrew clearly outperforms ReAct across all three scenarios, and despite using only one test-time attempt, matches or exceeds Reflexion ( k=2 )on coding and tool-use—MBPP (61.22% vs. 54.22%), MBPP+ (71.43% vs. 70.78%), AppWorld Normal (31.55% vs. 29.76%), and AppWorld Challenge (15.35% vs. 14.63%). Reflexion’s advantage is concentrated on math benchmarks, where iterative self-correction from explicit pass/fail signals is most beneficial for multi-step arithmetic reasoning. These trends support that routing binary failure feedback through a strong teacher and validating notes before storage yields more useful knowledge than unaided student self-reflection, particularly when weak-model reflection is insufficient for complex code generation and API interaction. 

(ii) Student-calibrated transfer vs. teacher-agnostic memory. Simple RAG, ExpeL, and ACE also use a strong teacher to build external memory (Group III), but none explicitly calibrates that memory to the weak student or validates it on the executor that must use it at test time. AgentBrew consistently outperforms ACE across math, coding, and tool-use, and achieves the best or tied-best results against Simple RAG and ExpeL on most benchmarks—with particularly clear gains on harder coding and tool-use splits such as MBPP+ and AppWorld test_challenge . These results support that bridging the teacher–student capability gap through student-aware authoring and environment-side validation is key to turning teacher knowledge into guidance that a fixed weak agent can actually follow.

A natural concern is whether notes validated by recovery on a specific brew task encode task-specific patches rather than transferable knowledge. The structured note schema is explicitly designed to prevent this: fields such as trigger_pattern , corrective_rule , and not_applicable_when

## 4 Related Work 

4.1 Self-Improving Agents from Environment Feedback 

LLM agents interleave language reasoning with environment actions ( e.g. , ReAct [ 21 ] and Toolformer-style tool-use [ 27 ]). Beyond one-shot prompting, a self-evolving agent literature studies how agents improve from interaction rather than offline expert labels, often leveraging sparse environment feed-back as the main supervision for accumulating reusable competence [ 28 ]. Representative approaches differ in what they evolve and what feedback they require: verbal self-improvement systems treat lan-guage as mutable memory by writing critiques after failures ( e.g. , Reflexion [ 5 ] and Self-Refine [ 29 ]), while experience-driven skill libraries extract reusable procedures from interaction to grow pools or code-based repositories ( e.g. , ExpeL [ 6] and Voyager [ 30 ]). More modular self-evolution and long-horizon memory frameworks push adaptation toward reusable protocols or persistent external stores ( e.g. , Alita [ 31 ] and OMNE [ 32 ]), and playbook-style context engineering closes execution-reflection-curation loops by revising structured bullets ( e.g. , ACE [ 11 ]). Overall, most self-evolving agents assume same-capability evolution and adapt online at test time, learning from successes or rich feedback, whereas AgentBrew instead performs offline "brewing" of environment-validated, student-aware notes under binary supervision only, and then serves them via a frozen memory at test time. 

4.2 Knowledge Transfer and Lifelong Memory for LLM Agents 

Knowledge distillation has been widely used to compress large language models and has recently been extended to agent settings by transferring behavior from a teacher to a student across different transfer channels [ 33 , 34 ]. At the trajectory level, structured span supervision and retrieval-augmented rollouts train students to imitate interleaved reasoning-action traces [ 7 , 35 ]. In contrast, structure-level distillation compresses trajectories into interaction graphs or subgoal plans to preserve high-level task structure [ 36 , 37 ]. Training-free approaches further store reusable external artifacts: for example, AgentDistill extracts Model-Context-Protocols (MCPs) from successful teacher trajectories into an inference-time MCP-Box [ 8]. RAG Distillation similarly distill teacher-to-student knowledge from failure-extracted hints, but internalize it into model weights via fine-tuning at the teacher’s abstraction level, without calibrating knowledge to the student’s capability or supporting incremental accumulation without re-training [ 38 ]. Related memory-based and lifelong-retrieval settings maintain external knowledge without parameter updates [ 32 , 39 , 40 , 2].Overall, our work addresses the limitations of protocol-like artifact transfer and teacher-dependent abstractions by generating student-validated corrective notes from student failures using only binary rewards. 

## 5 Conclusion 

We study a practical deployment scenario for LLM agents where a strong teacher is available during training but only a fixed, weaker student serves at test time—with no weight updates, no teacher access, and only binary pass/fail feedback. The core challenge is that teacher knowledge, however insightful, often fails when executed by a substantially weaker student. We introduced AgentBrew, a training-free brew–serve framework built on the principle that knowledge must be calibrated to its executor. Its three components directly address the key obstacles: a failure-triggered teacher with Ralph Loop validation converts sparse binary feedback into environment-certified notes (Q1); student-aware synthesis paired with reactive validation bridges the teacher–student capability gap (Q2); and an append-only skill-scoped memory accumulates knowledge across tasks without modifying the student model (Q3). Extensive experiments across coding, math, tool-use, and a real-world Terminal-Bench deployment demonstrate that AgentBrew consistently lifts a fixed weak student, surpassing teacher-agnostic memory baselines. 12 References

Algo. 3 formalizes the test-time inference procedure introduced in Sec. 2.5.3 and Sec. 2.2. For each test task x, the instruction is first encoded into a retrieval query q that captures key textual signals such as task descriptions and tool names (line 2). The skill-scoped router ROUTE SCOPE then selects a partition Mscope ⊆ M ⋆ following the scoping policy in Sec. 2.5.1—explicit scope hints in x are matched first, then lexical similarity against category descriptions, and finally runtime cues when available—confining subsequent retrieval to a relevant skill subset and preventing cross-domain interference (line 3). quality_score ρ(n) and a frequency penalty ϕ(n); soft suppression dampens notes below a relative-score floor without hard-dropping them, preserving recall when the candidate pool is sparse. The top-k
