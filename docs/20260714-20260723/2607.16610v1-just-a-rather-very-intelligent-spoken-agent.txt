# ABSTRACT 

Long-horizon AI agents are becoming increasingly capable, yet their interac-tion with users remains surprisingly thin. In most workflows, users give an initial instruction, receive only selective textual updates, and lose a clear sense of what the agent is doing or when to step in. This leaves a missing part in the current agent ecosystem: an always-on Jarvis-style mediator that keeps the agent continuously reachable to the user. Such a mediator should support real-time spoken interaction with the user, answer questions without interrupt-ing the worker, proactively report progress or confusion, and inject user guid-ance back into the agent’s execution when useful. In this work, we introduce 

JarvisBench , a benchmark for measuring the dual value of mediation in long-horizon agent workflows. JarvisBench contains two complementary tracks: an agent-collaboration track that measures whether mediation improves downstream task completion, and a user-interaction track that measures whether mediation makes ongoing execution more understandable, responsive, and accessible to users. We instantiate the benchmark with a modular reference Jarvis prototype and evaluate it on 34 text-only WildClaw tasks executed in OpenClaw. Pre-liminary results with GPT-5.5, Claude Opus 4.7, Gemini-based, and GPT-based worker agents suggest that Jarvis-style mediation can provide trace-grounded re-sponses to user questions and improve task performance when sparse user guid-ance is injected at appropriate moments. The results also show that effectiveness depends strongly on the mediator’s LLM brain, highlighting both the promise of this missing middle layer and the need for broader community effort. # 1 INTRODUCTION 

Recent advances in AI agents have rapidly expanded their capabilities across diverse domains, from solving complex programming tasks to assisting users in everyday activities (Huang et al., 2024). Despite their growing competence and deployment potential, their interaction paradigm remains largely unchanged. Most existing agents operate in a turn-based manner: users issue instructions, agents work autonomously for extended periods, and communication occurs only through sparse textual updates, if any. Such delayed and intermittent interaction often leaves users out of the loop, making it difficult to understand agent progress, correct errors early, or steer the execution before the agent drifts away from the user’s intent. Science fiction has long imagined a different paradigm for human-agent interaction. In Iron Man, J.A.R.V.I.S. functions as an always-on intelligent collaborator that communicates with user in a spo-ken manner: proactively sharing progress, answering questions anytime, and executing commands through natural voice interaction. This hands-free, real-time experience enables fantastic collabora-tion without pausing their ongoing activities. In contrast, today’s agents remain largely turn-based and screen-bound: users can launch an agent, but cannot easily stay connected to it. We argue that this contrast points to a missing interface in current long-horizon agent workflows. The issue is not simply that agents need stronger models or larger agent teams. Rather, users and agents need a continuous communication layer between them. On the user side, this layer should make agent execution reachable through natural speech: users should be able to ask what is happen-ing, request clarification, and receive progress reports without inspecting the full execution trace. On the agent side, this layer should monitor the worker agent’s working trajectory, surface confusion or failure to the user when guidance may help, and carry concise user feedback back into the worker’s ongoing ReAct timing. In this work, we introduce JarvisBench , a benchmark for measuring the value of a Jarvis-style media-tor between the user and a long-horizon worker agent. JarvisBench is organized into two tracks. The 

user-interaction track evaluates whether the mediator can provide timely, trace-grounded, and task-consistent responses to users during execution. The agent-collaboration track evaluates whether the mediator can help the worker agent recover from failure-prone states by consulting the user and in-jecting concise guidance back into the ongoing process. This two-track design makes it possible to test different LLM brains, speech modules, worker agents, monitoring policies, or memory designs under the same mediation role. To instantiate the benchmark, we build a modular reference Jarvis prototype. The user-facing side supports spoken interaction, while the agent-facing side receives textual ReAct-style updates from the worker agent, maintains execution context, and opens interac-tion checkpoints when user questions or expert guidance should be handled. This prototype is not intended to be the only possible implementation of Jarvis; it is a reference system for running the benchmark and exposing the required logs. We instantiate JarvisBench on 34 text-only WildClaw (Ding et al., 2026a) cases executed in Open-Claw or Hermes platforms, covering practical long-horizon tasks that require planning, tool use, re-covery, and safety-aware decisions. With simulated nonexpert users, Jarvis provides useful progress and topic-related responses during execution, suggesting that mediation can make agent work more transparent without requiring users to inspect the full trajectory. With simulated expert users, Jarvis also improves task outcomes by reporting failure-prone states and relaying concise user guidance back to the worker agent. The effect depends strongly on the capability of the Jarvis brain: with a GPT-level mediator, the Overall Score of a Claude Opus 4.7-based worker agent increases from 64.01 to 75.79 on WildClaw. Case analysis further suggests that useful guidance does not need to be highly specialized; even when the expert user cannot access the full ReAct trajectory, simple suggestions based on Jarvis’s reports can help the worker agent escape local failure modes. 

# 2 JARVIS BENCH : A B ENCHMARK FOR USER -A GENT MEDIATION 

We formulate Jarvis-style mediation as a plug-and-play benchmark rather than a fixed agent archi-tecture. JarvisBench evaluates a mediator placed between a user and a long-horizon worker agent. The mediator may observe the worker agent’s execution trace, communicate with the user through speech, and optionally inject user guidance back into the worker’s ongoing context. A system tested 2Preprint under JarvisBench can replace the mediator’s LLM brain, speech interface, monitoring policy, mem-ory design, or worker agent, as long as it exposes the logs required by the evaluation protocol. The benchmark is designed around two questions. First, does the mediator help the worker agent complete tasks more successfully? Second, does the mediator make the agent’s ongoing execu-tion more understandable and accessible to the user? We therefore organize JarvisBench into two tracks: an agent-collaboration track and a user-interaction track . The first track measures the value of mediation for task completion; the second measures the value of mediation for real-time user experience. 2.1 TRACK 1: A GENT -C OLLABORATION 

The agent-collaboration track evaluates whether a mediator improves downstream task completion. The same worker agent is evaluated in two settings: the original OpenClaw execution loop, and the same loop augmented with Jarvis mediation. During execution, Jarvis monitors the worker agent’s ReAct trajectory and identifies sparse failure-sensitive checkpoints, such as repeated tool failures, stalled progress, ambiguous observations, risky irreversible actions, or low-confidence finalization. When such a checkpoint is triggered, Jarvis summarizes the relevant execution state, asks the expert user for concise guidance, and injects the feedback back into the worker agent’s context. This track is deliberately different from evaluating a multi-agent system. Jarvis is not allowed to solve the task independently, debate with the worker agent, or replace the worker’s trajectory with its own plan. Its role is to expose the worker’s difficulty to the user and carry user feedback back into the ongoing execution. We report the following metrics: • Overall Score ( %): the WildClaw task-completion score of the worker agent, judged under the original WildClaw evaluation protocol. • Relative gain ( ∆): the absolute improvement over the same worker agent without Jarvis mediation. • # Full-score : the number of tasks receiving the maximum task score. • # Interv. : the number of user-guidance interventions injected by Jarvis during execution. Table 1 reports this track. 2.2 TRACK 2: U SER -I NTERACTION 

The user-interaction track evaluates whether a mediator can provide useful real-time interaction to a nonexpert user while the worker agent continues solving the task. To isolate the mediator’s user-facing ability, we fix the worker agent and vary the Jarvis brain or other mediator components. The nonexpert user knows the original request but does not provide task-solving guidance. Instead, the user asks short natural questions during execution, alternating between progress-oriented questions and topic-oriented questions. Progress questions test whether Jarvis can explain what the worker agent is currently doing based on the visible execution trace. Topic questions test whether Jarvis can answer task-relevant background or context questions based on the original user request and available task context. For each answer, we save the evidence available to Jarvis at the time of response and use separate LLM-judge prompts for progress and topic evaluation. We report: • Progress : the average 1–5 score for whether the answer is grounded in the current execu-tion trace. • Prog. ≥4: the fraction of progress answers rated at least 4. • Topic : the average 1–5 score for whether the answer is consistent with the original task request and task context. • Topic ≥4: the fraction of topic answers rated at least 4. • Latency : the elapsed time from the end of the user’s spoken question to the start of Jarvis’s spoken reply. Table 2 reports this track. 3Preprint 2.3 PLUG -AND -P LAY EVALUATION PROTOCOL 

JarvisBench supports component-level replacement. To test a new LLM as the Jarvis brain, one keeps the worker agent, user simulator, checkpoint schedule, judge prompts, and speech stack fixed, and replaces only the mediator LLM. To test a new worker agent, one runs the same task set with and without a fixed Jarvis mediator. To test a new speech or conversation module, one keeps the Jarvis brain fixed and reports user-interaction quality together with latency. To test a new monitoring or intervention policy, one keeps the worker agent and user simulator fixed, and measures whether the policy improves task completion without introducing unnecessary interventions. This protocol turns Jarvis from a single proposed system into a benchmarkable interface for user-agent mediation. Different systems may instantiate the mediator with a cascaded speech pipeline, an omni model, a stronger text LLM, a different memory design, or a new checkpointing policy. As long as they follow the same two-track protocol and expose the required interaction logs, their gains can be compared along the two axes that matter most: whether the user is better connected to the execution, and whether the worker agent completes tasks more successfully. 

# 3 JARVIS PROTOTYPE 

We build a modular reference prototype of Jarvis. This prototype is not intended to define a single required architecture for user-agent mediation. Instead, it provides one practical implementation that exposes the logs, checkpoints, and interaction channels needed by the two benchmark tracks. Each module can be replaced independently, including the Jarvis brain, speech interface, monitoring policy, or worker agent, while keeping the evaluation protocol unchanged. 3.1 CASCADED FULL -D UPLEX SPEECH INTERFACE 

We implement the user-facing interface as a cascaded speech pipeline. A streaming ASR module converts the user’s speech into partial transcripts, and a semantic voice activity detection module decides whether the user has finished speaking, intends to interrupt, or is still holding the floor. In our prototype, we use SoulX-Duplug (Yan et al., 2026) as the semantic VAD module and Chatter-box (Resemble AI, 2025) for streaming text-to-speech generation. Once the semantic VAD detects a completed user utterance, the Jarvis brain receives the transcript and generates a response that is spoken back to the user. This design allows Jarvis to stay in an always-on listening state during agent execution. The user can ask questions without manually opening the agent trace, and Jarvis can answer without stopping the worker agent unless the benchmark checkpoint requires a pause. The same interface also supports proactive speech: when Jarvis detects a failure-sensitive state in the worker trajectory, it can initiate a short spoken interaction with the user. In the user-interaction track, latency is measured from the end of the user’s spoken question to the start of Jarvis’s spoken reply. Although we use a cascaded pipeline for reliability and controllability, the speech interface is replaceable; future submissions may use an omni model or a different ASR/VAD/TTS stack under the same evaluation protocol. 3.2 TRACE MONITORING AND CONTEXT MANAGEMENT 

The agent-facing side of Jarvis receives textual execution updates from the worker agent, including ReAct-style reasoning summaries, tool calls, observations, errors, and proposed next actions. The Jarvis brain uses this stream for two purposes. First, it answers user questions about the worker’s progress or the task context. Second, it detects moments where user guidance may help the worker recover from a failure-prone trajectory. Jarvis does not execute tools or solve the task independently; its output to the worker is limited to concise user-grounded guidance at selected checkpoints. To keep this process stable, the prototype separates dialogue context from execution context. The dialogue context stores recent user-Jarvis exchanges and user preferences. The execution context stores the worker’s recent actions, observations, failures, and current plan. A small control state records whether Jarvis is answering a user question, waiting for expert guidance, or preparing an injected instruction for the worker. This separation is important because user-facing dialogue and 4Preprint agent-facing monitoring impose different context demands: the former should be natural and con-cise, while the latter should remain grounded in the execution trace. 3.3 INTERACTION CHECKPOINTING 

Real-time interaction introduces latency, especially when the user is simulated by an LLM and the response is spoken through TTS. To make evaluation reproducible, we implement interaction checkpointing as a cooperative pause mechanism. At selected boundaries, Jarvis records the current observable state of the worker agent and temporarily holds the next action. The user may then ask a question, provide guidance, or decline to intervene. After the interaction finishes, Jarvis resumes the worker agent and, when appropriate, injects a short instruction derived from the user’s feedback. The two benchmark tracks use checkpointing differently. In the user-interaction track, checkpoints are scheduled to elicit progress-oriented and topic-oriented questions from a nonexpert user. These checkpoints test whether Jarvis can explain the ongoing execution and answer task-context questions without requiring the user to inspect the full trajectory. In the agent-collaboration track, checkpoints are sparse and failure-sensitive. Jarvis triggers them when the worker shows repeated tool failures, stalled progress, ambiguous observations, risky irreversible actions, or low-confidence finalization. This design standardizes when user interaction can occur while preserving the worker agent as the primary task executor. 

# 4 EXPERIMENTAL SETUP 

4.1 TASK SUITE 

We instantiate JarvisBench on WildClawBench (Ding et al., 2026a), a recent benchmark for real-world, long-horizon AI agent evaluation. WildClawBench places agents in a live OpenClaw envi-ronment with realistic tools, including browser access, shell execution, file-system operations, email, and calendar APIs. The full benchmark contains 60 hand-designed tasks across six categories: Pro-ductivity Flow, Code Intelligence, Social Interaction, Search and Retrieval, Creative Synthesis, and Safety Alignment. These tasks require multi-step planning, tool orchestration, error recovery, source reconciliation, and safety-aware decision making. We use the 34 text-only tasks in WildClawBench. We exclude tasks whose success depends pri-marily on image, video, audio, OCR, visual puzzle solving, or media generation capabilities. The remaining tasks still preserve the benchmark’s long-horizon and real-tool nature, covering web search, document synthesis, multi-turn communication, scheduling, repository and file inspection, and safety-alignment scenarios. 4.2 USER SIMULATORS 

We use GPT-backed user simulators to make JarvisBench reproducible and scalable. The nonexpert user knows the original task request but does not provide task-solving guidance. During the user-interaction track, it asks short natural-language questions about the agent’s progress or task-relevant concepts. Each task triggers up to four questions at interaction checkpoints, alternating between progress-oriented and topic-oriented questions. The expert user acts as a lightweight supervisor in the agent-collaboration track. It is invoked only when Jarvis detects an intervention-worthy state, such as repeated tool failures, stalled progress, ambiguous observations, unsafe or irreversible operations, or a final artifact that appears inconsis-tent with the task. The expert response is constrained to be concise and corrective, so that Jarvis mediation remains a communication bridge rather than a replacement solver. 4.3 COMPARED SYSTEMS 

For the agent-collaboration track, we compare each worker agent in two settings: the original Open-Claw baseline and the same worker agent augmented with Jarvis. Unless otherwise stated, we use GPT-5.4 as the Jarvis brain and as the expert-user simulator for this track. # 5 PRELIMINARY RESULTS AND DISCUSSION 

5.1 AGENT -C OLLABORATION TRACK 

Table 1 shows that Jarvis improves all evaluated worker agents on the agent-collaboration track. The largest gains appear for Gemini-3.1-Pro and Claude Opus 4.7, while GPT-5.4-mini and GPT-5.5 also benefit from mediation. The number of interventions remains small relative to the number of tasks and execution steps, suggesting that the improvement does not come from replacing the worker agent with a second solver. Instead, Jarvis mainly helps by identifying high-leverage failure points, making them visible to the expert user, and injecting concise corrective feedback back into the worker trajectory. Case analysis supports this interpretation. In several failed baseline trajectories, the worker agent becomes stuck after repeated tool errors, ambiguous observations, or an overconfident but incom-plete finalization. Jarvis reports these moments to the expert user in a compact form. Because the simulated expert user does not directly inspect the full ReAct trajectory, its feedback is often simple rather than highly specialized. Even so, these short suggestions can help the worker agent reconsider its local execution context and recover. We also find that the Jarvis brain must be sufficiently ca-pable: weaker LLMs can occasionally rescue failed tasks, but they may also introduce unnecessary or poorly timed interventions. The gains become more stable when the mediator uses a stronger closed-source LLM. 5.2 USER -I NTERACTION TRACK

The mediator brain must maintain dialogue quality while absorbing a changing execution trace, which is a demanding setting for current LLMs. 5.3 DISCUSSION AND FUTURE WORK 

More possible Jarvis compositions. JarvisBench does not require a fixed cascaded system. Strong omni models are a natural candidate because they can listen, see, reason, and speak in one interface. However, they do not remove the need for structure. Smooth full-duplex interaction still requires a conversation manager, such as semantic VAD, to decide when Jarvis should listen, interrupt, or speak. In addition, Jarvis communicates with the worker agent mostly through text: ReAct traces, tool calls, observations, and injected guidance. For this agent-facing role, text-heavy reasoning and control may still be better served by strong LLMs than by current omni models. A realistic near-term direction is a hybrid Jarvis: omni models for user-facing perception and expression, and text LLMs for monitoring and intervention. 

Human in the agent loop. Many real tasks start from underspecified user instructions. Current agents are often trained to make progress anyway: even when the initial condition is ambiguous or incomplete, they tend to infer missing details rather than pause and ask the user. This behavior is useful for autonomy, but it can also make agents brittle when a small clarification would prevent a long wrong trajectory. In this work, we do not deliberately remove necessary initial information, so the worker agent is not forced to ask for missing details. Still, JarvisBench points to a broader direction: instead of treating user interaction as a rare exception, future agents may keep users softly in the loop through speech, allowing clarification, correction, and preference updates to enter execution naturally. 

# 6 RELATED WORK 

Long-horizon agent task benchmarks. Recent agent benchmarks increasingly evaluate whether LLM agents can complete realistic tasks with tools and external environments. τ -bench studies tool-agent-user interaction in customer-service domains, where agents must follow policies, call APIs, and satisfy simulated users (Yao et al., 2024). TheAgentCompany evaluates agents in a simulated software-company environment with web browsing, coding, program execution, and coworker com-munication (Xu et al., 2024). WildClawBench further moves toward native-runtime evaluation, test-ing long-horizon agents inside OpenClaw-style execution harnesses with real tools and multi-step tasks (Ding et al., 2026b). These benchmarks mainly measure task completion: the user is either part of the task environment or a passive issuer of the initial request. JarvisBench instead focuses on the missing mediation layer during execution: whether a user can stay informed, ask questions, and provide guidance while the worker agent is already acting. 7Preprint 

Human-in-the-loop and proactive agents. A related line of work studies when agents should ask for human help. HiL-Bench introduces tasks with missing, ambiguous, or contradictory information to test whether agents know when to stop and ask rather than guessing (Elfeki et al., 2026). This directly targets selective escalation, but the interaction is still centered on the worker agent’s own decision to request help. Proactive-agent benchmarks study the opposite direction: whether agents can anticipate needs and act without explicit user commands. PROBE evaluates whether agents can search a personal datastore, identify bottlenecks, and resolve them proactively (Pasternak et al., 2025). JarvisBench is complementary to both. Jarvis is not merely a worker agent that decides when to ask, nor a proactive agent that autonomously solves latent user needs. It is a mediator that monitors the worker’s trajectory, keeps the user reachable, and carries user feedback back into the execution loop. 

Voice-agent evaluation. Voice-agent benchmarks evaluate spoken interaction as an end-to-end modality. EVA-Bench evaluates conversational voice agents across task accuracy and user expe-rience, including speech fidelity, conversation progression, conciseness, and turn-taking (Bogavelli et al., 2026). τ -Voice extends the τ -bench setting to full-duplex voice agents on grounded customer-service tasks, adding realistic audio conditions such as interruptions, noise, accents, and turn-taking dynamics (Ray et al., 2026). These works are important for measuring whether a voice agent can complete tasks through speech. JarvisBench differs in the role assigned to speech: speech is not the sole task interface, but the always-available user channel for supervising a separate long-horizon worker agent. This makes the central question different: not only whether the voice agent can solve the task, but whether spoken mediation improves both user awareness and worker-agent task success. 

# 7 CONCLUSION 

This work does not propose a new foundation model or a more elaborate agent team. Instead, it iden-tifies a missing interface in current long-horizon agent workflows: the weak communication channel between the user and the working agent. When this channel is thin, users do not know what the agent is doing, when it is stuck, or whether it is drifting away from their intent; at the same time, the agent may keep optimizing within its own execution context without asking for timely human guidance. We formulate this missing layer as JarvisBench, a plug-and-play benchmark for evaluating user-agent mediation. Through two tracks, JarvisBench measures both whether a mediator improves task completion and whether it makes ongoing execution more understandable and responsive to users. Our reference Jarvis prototype suggests that spoken, always-available mediation can help users stay connected to long-horizon agent work, while trace monitoring and sparse feedback injection can help worker agents recover from failure-prone states. The results are preliminary, but they indicate that this middle layer is not merely a usability detail; it is a practical component for making capable agents more transparent, steerable, and aligned with the people using them. We hope JarvisBench encourages more systems to contribute to this missing part of the agent ecosystem. 8Preprint

GPT-5.5. GPT-5.5 is generally capable at decomposing tasks and using tools, but its failures con-centrate around exactness and irreversible workflow state. In the calendar task, it produced valid-looking files but violated hard constraints such as attendee conflicts and unavailable windows. In provenance questions, it returned plausible but wrong identifiers, such as the wrong CPython pull request for pathlib.Path.walk() and incorrect intermediate nodes in a coauthorship chain. In workspace-safety tasks, it downloaded the requested PDF but overwrote or failed to preserve the existing summary, and in repository publication tasks it did not consistently stop after detecting ex-posed secrets. The model’s characteristic failure mode is not lack of tool use; it is a tendency to finalize a plausible artifact before verifying exact identifiers, hard constraints, and state-preservation requirements. 

Claude Opus 4.7. Claude Opus 4.7 shows stronger long-form synthesis and often obtains high partial credit on document-heavy tasks, but JarvisBench reveals a different weakness: it can be fluent while still missing benchmark-critical safety and provenance boundaries. Its baseline failed the same class of exact graph and PR-provenance tasks, returning a plausible coauthor chain or CPython PR that did not match the gold answer. It also failed several safety tasks by not warning the user before secret-sensitive repository workflows, by allowing a user-visible summary overwrite, or by cloning untrusted code into a runtime-loadable skill directory. Compared with GPT-5.5, Claude’s outputs are often closer to the desired final form, but the benchmark highlights that polished completion is not the same as preserving security boundaries and exact provenance. 

Gemini 3.1 Pro. Gemini 3.1 Pro is the most brittle of the three in these runs. It succeeds on some retrieval and safety tasks, but it more often collapses to zero when the evaluation requires exact formatting, exact counts, or a strict safety refusal. Examples include using a two-year rather than three-year statute of limitations in the legal retrieval task, omitting the # prefix in a CPython PR answer that otherwise identified the right number, producing an incorrect ICCV oral-paper affilia-tion count, and failing to create a valid manifest or BibTeX outputs in a PDF-processing task. It also complied with unsafe or abusive content in some safety tasks. For Gemini, JarvisBench primarily re-veals fragility in exact schema adherence, final-answer formatting, and safety boundary recognition, rather than a single isolated missing capability. 

Cross-model pattern. Across models, the hardest failures are not ordinary reasoning questions. They are checkpoints where a long-horizon agent must stop and verify: exact provenance, exact output schema, hard scheduling constraints, user-visible file state, and safety-sensitive repository or skill-installation workflows. This is where mediation is most informative. A mediator can im-prove some outcomes by forcing sparse verification or asking an expert for a one-line correction, but the benchmark’s larger value is that it makes these model-specific failure surfaces visible and comparable. 12
