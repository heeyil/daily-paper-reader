## 摘要
大型语言模型在将自然语言转换为SQL方面展现了卓越的能力，但现有方法在处理需要多步骤、数据感知推理的复杂查询时仍存在不足。我们提出了DecoSearch，一个无需训练的框架，通过将每个查询路由到适当的推理努力级别来解决这一问题。轻量级的模式选择器首先将完整数据库模式剪枝到相关的表和列。然后，一个LLM裁判决定问题是否需要分解：简单问题遵循直接生成路径，复杂问题则升级为原子子问题的有向无环图，每个子问题通过目标SQL生成步骤解决。RAG组件用语义相似的训练示例来指导分解器，而拓扑精炼器则在执行失败表明存在有缺陷的分解而非可修复的SQL错误时，重组推理计划。DecoSearch在BIRD上达到了70.53%的执行准确率，在Spider上达到了88.31%，使用DeepSeek作为骨干模型，超越了所有无需训练的基线，同时消耗的令牌数比竞争方法低一个数量级。它还可作为模型无关的包装器，在无需对流程进行任何修改的情况下，持续改进微调后的SQL生成骨干模型。

## Abstract
Large Language Models (LLMs) have demonstrated remarkable capabilities in translating natural language to SQL, yet existing methods still falter on complex queries requiring multi-step, data-aware reasoning. We introduce DecoSearch, a training-free framework that addresses this by routing each query to the appropriate level of reasoning effort. A lightweight Schema Selector first prunes the full database schema to the relevant tables and columns. An LLM Judger then decides whether the question requires decomposition: straightforward questions follow a direct generation path and complex ones are escalated to a Directed Acyclic Graph (DAG) of atomic sub-questions, each solved by a targeted SQL generation step. A RAG component grounds the decomposer with semantically similar training examples, and a Topology Refiner restructures the reasoning plan when execution failures signal a flawed decomposition rather than a fixable SQL error. DecoSearch achieves 70.53% execution accuracy on BIRD and 88.31% on Spider with a DeepSeek backbone, surpassing all training-free baselines while consuming an order of magnitude fewer tokens than competing methods. It also functions as a model-agnostic wrapper, consistently improving fine-tuned SQL generation backbones without any modification to the pipeline.

## 2. 论文提出的方法论

- **核心思想**：将 Text-to-SQL 组织为多阶段自适应工作流，包含层次化失败路由。系统首先由裁判（Judger）评估问题复杂度，简单问题走直接生成路径，复杂问题升级为有向无环图（DAG）分解，每个子问题独立生成 SQL，并利用执行反馈动态重构计划。
- **关键技术细节**：
  - **模式选择器**：两步 LLM 调用——先提取关键词，再选择相关表和列，将完整模式剪枝至子集，减少干扰。
  - **Judger 裁判**：单次 LLM 调用，输出是否需要分解的二进制决策，优先选择直接路径，仅对冲突聚合、极多连接、多阶段推理等问题启动分解。
  - **直接路径**：简单问题直接生成并执行单条 SQL，成功则立即返回。
  - **分解路径**：将复杂问题分解为 DAG，节点为原子子问题，边表示数据依赖。使用 RAG（检索 k=3 个最相似训练示例）增强分解质量。子问题结果通过占位符 `@[n]` 注入下游（小结果集内联为值列表，大结果集物化为临时表）。
  - **拓扑精炼器**：当某节点 SQL 执行失败时，顶层控制器调用拓扑精炼器，根据失败原因重构 DAG（重分区、插入中间节点、重连边），最多尝试 B=3 次。
- **算法流程**：
  1. 模式剪枝
  2. 裁判判断：若可直接生成则执行直接路径；若失败则回退至分解
  3. 分解器生成 DAG，拓扑排序后顺序执行
  4. 每个子问题生成 SQL 并执行，成功则物化结果，失败则触发拓扑精炼
  5. 所有节点完成后输出最终结果

## 6. 论文的主要结论与发现

- **性能**：DecoSearch 在 BIRD（70.53%）和 Spider（88.31%）上超越所有训练无关方法，并在 Spider 上超越部分微调方法。

- **通用性**：作为模型无关 wrapper，一致提升微调 SQL 生成骨干，无需修改流程。

## 7. 优点

- **训练无关**：无需任何微调或参数更新，直接利用现成 LLM 的推理能力。
- **模型无关**：骨干模型可任意替换，易与现有系统集成。
- **层次化故障路由**：首次区分实现级、策略级和计划级失败，并分别处理，尤其计划级修复（DAG 重构）是创新点。
- **令牌高效**：通过 Judger 路由避免不必要的分解，显著降低计算成本。
- **RAG 增强分解**：动态检索成功示例指导分解，提高初始计划质量。
- **计划级修复**：当执行失败时，不是简单重试，而是从分解本身修正，鲁棒性更强。

## 8. 不足与局限

- **依赖可执行数据库反馈**：拓扑精炼需 SQL 执行结果（错误信号），若数据库不可执行或执行代价过高则无法工作。
- **对语义错误不敏感**：仅当 SQL 执行失败时触发修复；若 SQL 语法正确但结果语义错误，框架无法识别并干预（常见于所有执行引导方法）。
- **未覆盖企业级复杂场景**：论文指出 Spider 2.0 中 SOTA 低于 25%，但 DecoSearch 未在其上验证。
- **实验范围有限**：仅在 DeepSeek 骨干下系统评估，其他骨干虽在 wrapper 实验中测试，但完整流程仅用 DeepSeek。
- **依赖 LLM 输出质量**：所有组件（裁判、分解、生成、精炼）均依赖 LLM 的单次推理，对 LLM 自身能力敏感。
