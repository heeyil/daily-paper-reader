## 摘要
实际用户通常能够访问来自不同提供商的多个大型语言模型（LLM），这些LLM往往在不同领域表现出色，但没有一个能全面占优。因此，将每个任务路由到最合适的模型对于性能和成本都至关重要。现有的路由器将这一问题视为静态的一次性分类任务。然而，我们识别出这些路由器的性能瓶颈在于信息不足：仅仅在任务维度级别上用性能统计数据增强一个普通的LLM路由器，就能获得15.3%的相对增益，超过了基于相同维度级别先验信息的启发式路由器。受此发现启发，我们提出了Agent-as-a-Router，一个将路由形式化为C-A-F循环（上下文→行动→反馈→上下文）的框架。它通过在部署期间积累执行经验来弥合信息差距。我们将该框架实例化为ACRouter，由编排器、验证器、记忆模块组成，并引入了CodeRouterBench，一个包含约10K个任务实例的评估环境，这些实例具有来自8个前沿LLM的验证评分，支持在流式任务上进行基于遗憾的路由器比较。实验表明，ACRouter在分布内任务上实现了最低的累积遗憾，并泛化到分布外的智能编程任务，证明我们的路由框架积极弥合了信息差距。代码和基准已发布在 https://github.com/LanceZPF/agent-as-a-router。

## Abstract
Real-world users typically have access to multiple Large Language Models (LLMs) from different providers, and these LLMs often excel at distinct domains, yet none dominate all. Consequently, routing each task to the most suitable model becomes critical for both performance and cost. Existing routers treat this as a static, one-off classification problem. However, we identify the performance bottleneck for these routers as information deficit: simply augmenting a vanilla LLM router with performance statistics at the task-dimension level yields a 15.3% relative gain, surpassing a heuristic router built on the same dimension-level priors. Motivated by this finding, we propose Agent-as-a-Router, a framework that formalizes routing as a C-A-F loop (Context->Action->Feedback->Context). It closes the information gap by accumulating execution-grounded experience during deployment. We instantiate this framework as ACRouter, composed of an Orchestrator, a Verifier, a Memory module, and introduce CodeRouterBench, an evaluation environment comprising ~10K task instances with verified scores from 8 frontier LLMs, enabling regret-based router comparison on streaming tasks. Experiments show that ACRouter achieves the lowest cumulative regret on in-distribution tasks and generalizes to out-of-distribution agentic-programming tasks, demonstrating that our routing framework actively closes the information gap. Codes and benchmarks are released at https://github.com/LanceZPF/agent-as-a-router.

## 2. 方法论

### 2.1 核心思想：C-A-F循环

- 将路由建模为上下文多臂赌博机（Contextual Multi-Armed Bandit）问题。在每个任务上，路由器观察上下文（包括任务提示、元数据、历史记忆），然后选择一个模型作为行动，接收经过验证的反馈（性能和成本），并将该反馈合并到上下文中用于后续决策。
- 循环迭代：`c_i → a_i → f_i → c_{i+1}`，其中 c 为上下文，a 为行动（选定模型），f 为反馈（分数和成本）。

### 2.2 关键技术细节

- **ACRouter 实例化**：由三个核心模块构成：
  - **Orchestrator（编排器）**：综合多种信息（DimensionBest 先验、Memory 中检索的 top-10 邻居、任务元数据），使用微调的 Qwen3.5-0.8B 模型与启发式规则加权投票做出最终决策。
  - **Verifier（验证器）**：在沙箱中执行模型输出，聚合多种信号（AST 解析、沙箱执行、提示内测试、基于规则信号）得到统一性能得分 u_i ∈ [0,1]。
  - **Memory（记忆模块）**：在线向量存储，基于任务嵌入（voyage-code-3 / BGE-large）的余弦 kNN 检索（k=10），FIFO 限制 20K 条记录，存储选择的模型、性能、成本和验证轨迹。

- **奖励与遗憾**：
  - 单任务奖励：`r_i(a_i) = ε1 * s_i(a_i) + ε2 * κ_i(a_i)`，其中 ε1>0（奖励性能），ε2<0（惩罚成本）。
  - 累计遗憾：`CumReg_N(π) = Σ (r*_i - r_i)`，r*_i 为每个任务的最优模型奖励（预计算的全知矩阵），用于流式路由评价。

- **上下文多臂赌博机等价性**：策略 π 诱导多项式分布，历史 h_i 包含此前所有决策与反馈。

### 2.3 算法流程（文字说明）

1. 初始化 Memory（可从探测集预填充）。
2. 对于任务流中的每个任务 t_i：
   - Orchestrator 从 Context（当前任务提示、维度信息、Memory 检索历史、DimensionBest 先验）中整合信息。
   - 生成行动 a_i（选择模型 m_{a_i}）。
   - 将任务发送给该模型，得到输出。
   - Verifier 在沙箱中评估输出，得到分数 s_i 和成本 κ_i。
   - 将 (a_i, s_i, κ_i) 写入 Memory。
   - 更新 Context 供下次使用。
3. 循环直至任务流结束。

## 6. 主要结论与发现

1. **信息赤字是路由的主要瓶颈**，而非推理能力不足。提供维度级性能统计即可使 LLM 路由器超过启发式方法。
2. **没有单一模型主宰所有编码维度**。不同维度最佳模型不同（例如 GLM-5 在算法设计最优，Qwen3-Max 在测试生成最优），且最强模型（Opus）成本最高。
3. **ACRouter 在分布内和分布外任务上均达到最低累积遗憾和最高平均性能**，且成本低于固定选择最强模型（Always-Opus）。
4. **静态学习方法在分布内表现尚可，但在分布外任务上崩盘**（平均性能低于随机），而 ACRouter 和在线赌博机由于在线更新而更具泛化性。

## 7. 优点

- **框架清晰且理论稳固**：将路由形式化为 C-A-F 循环，与上下文多臂赌博机等价，遗憾度量自然。
- **诊断实验精准揭示瓶颈**：通过简单消融（加统计信息）证明信息赤字是核心，说服力强。
