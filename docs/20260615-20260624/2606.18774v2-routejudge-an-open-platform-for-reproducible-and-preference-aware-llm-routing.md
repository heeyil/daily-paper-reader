## 摘要
我们提出了RouteJudge，一个用于LLM路由系统的在线成对偏好评估框架，并提供了公开平台（https://routejudge.cn）。与模型级响应评估不同，RouteJudge关注路由级决策质量。对于每个用户查询，多个路由策略在相同的模型池和预算约束下独立推荐候选模型。然后，通过匿名成对比较将所选模型的响应呈现给用户，并将由此产生的用户偏好归因于被比较响应背后的路由策略。每条评估记录存储查询、路由决策、模型响应、偏好标签、成本、延迟和任务元数据，从而支持对LLM路由器进行偏好感知、成本感知和任务条件分析。为了支持RouteJudge中路由方法的持续扩展，我们进一步发布了ORBIT（最优路由与预算推理工具箱），这是一个模块化、可扩展的工具箱，标准化了LLM路由的端到端工作流程。ORBIT为基准加载、查询表示、路由器实现、预算感知评估和方法比较提供了统一接口，使研究人员能够在一致的协议下开发和评估路由算法。它还是RouteJudge的提交和集成层：研究人员可以在ORBIT中实现路由方法，在现有路由基准上验证它们，并提交兼容的路由器进行在线基于偏好的评估。ORBIT的代码可在https://github.com/LAMDA-Model-Reuse/ORBIT获取。

## Abstract
We present RouteJudge, an online pairwise preference evaluation framework for LLM routing systems, with a public platform available at https://routejudge.cn. Different from model-level response evaluation, RouteJudge focuses on router-level decision quality. For each user query, multiple routing strategies independently recommend candidate models under the same model pool and budget constraints. The selected model responses are then presented to users through anonymous pairwise comparisons, and the resulting user preferences are attributed back to the routing strategies behind the compared responses. Each evaluation record stores the query, routing decisions, model responses, preference labels, cost, latency, and task metadata, enabling preference-aware, cost-aware, and task-conditioned analysis of LLM routers. To support the continuous expansion of routing methods in RouteJudge, we further release ORBIT (Optimal Routing and Budgeted Inference Toolbox), a modular and extensible toolbox that standardizes the end-to-end workflow of LLM routing. ORBIT provides unified interfaces for benchmark loading, query representation, router implementation, budget-aware evaluation, and method comparison, allowing researchers to develop and evaluate routing algorithms under consistent protocols. It also serves as the submission and integration layer for RouteJudge: researchers can implement routing methods within ORBIT, validate them on existing routing benchmarks, and submit compatible routers for online preference-based evaluation. The code of ORBIT is available at https://github.com/LAMDA-Model-Reuse/ORBIT.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：大型语言模型（LLM）的部署正从单一模型转向异构模型池，需要自动为每个查询选择最合适的模型，即 **LLM路由**。然而，现有路由评估方法大多依赖静态基准、标准答案或自动评分，将路由质量简化为固定目标（如匹配基准最优模型），忽略了真实用户偏好的**多样性**（如对简洁性、详细程度、推理风格、成本延迟容忍度的不同需求）。
- **核心问题**：路由决策质量应该以用户实际是否偏好被选模型产生的响应来衡量，而非仅与离线标签或自动评分一致。当前缺乏**偏好感知**、**可复现**、并能持续扩展的LLM路由评估平台。
- **解决方案意义**：作者提出 **RouteJudge** 在线平台和 **ORBIT** 工具箱，构建开放生态系统：前者通过匿名成对比较将用户偏好归因于路由策略，后者标准化路由开发、离线评估与提交流程。两者结合实现了从“固定目标离线评分”到“偏好感知在线评估”的范式转变，为LLM路由提供了更贴近部署现实的评估基准。

## 2. 方法论：核心思想、关键技术细节、公式或算法流程（文字说明）

### 2.1 RouteJudge 评估框架

- **核心思想**：将评估焦点从“模型响应质量”转移到“**路由决策质量**”。对于同一查询，多个路由策略在相同模型池和预算约束下独立推荐一个模型；然后选出两个被推荐次数最多的模型（构成对决对），匿名展示两者的响应给用户，用户给出四类偏好标签（A胜、B胜、平局、两者都差）。
- **关键技术细节**：
    - **评估记录结构**：每条记录 `Z = (x, h, I, C, M_C, d, v, m_A, m_B, y, s, c, ℓ, τ, η)` 包含查询、历史、多模态输入、预算、可行模型集、所有路由器的选择、模型级投票分布、对决模型对、用户偏好标签、路由器级归因分数、成本、延迟、任务元数据等。
    - **路由器归因**：偏好标签（y）先映射为比较分数 `(s_A, s_B)`：A胜→(1,0)，B胜→(0,1)，平局→(0.5,0.5)，两者都差→(0,0)。然后根据路由器所选模型是否参与对决来归因：若参与则获得对应分数，否则标记为“∅”（非参与）。非参与本身也作为统计信息（参与率）。
    - **任务元数据**：每个查询分配任务标签（编程、数学、翻译、创意写作、分析等），支持任务条件分析。

### 2.2 ORBIT 工具箱

- **核心思想**：标准化路由开发的端到端流程，提供统一接口（数据加载、查询表示、路由器实现、训练、评估、部署），消除不同实现间的预处理不一致。
- **关键技术细节**：
    - **预算感知路由公式**：给定模型池 M 和预算 C，可行模型集 `F(q;C) = {j in M | c_j(q) ≤ C}`。路由策略 `π_C` 映射查询表示到 M 中的模型，目标是最大化期望效用 `E[a_{π_C(ϕ(q))}(q)]`，同时满足预算约束。
    - **统一入口点**：`python main.py --dataset [Dataset Name] --method [Method Name]`，通过配置分离数据集和路由器参数。
    - **可扩展性**：支持添加新基准、新编码器（嵌入层）、新路由算法（继承 `BaseRouter`）。
    - **集成到 RouteJudge**：通过 PR 提交 ORBIT 兼容的路由器，先进行历史回放评估（根据历史记录中用户偏好计算路由器是否选择了被偏好的模型），再上线进行在线评估。

## 6. 主要结论与发现

1. **离线性能与在线偏好存在差异**：在离线基准上表现好的路由器（如某些学习型评分路由器）在在线用户偏好评估中不一定占优；相反，简单非参数或矩阵分解方法（如 RouteLLM-MF、kNNRouter）表现突出。这验证了偏好感知评估的必要性。
2. **成本-偏好不可由单一维度决定**：低成本的模型有时能达到较高胜率，而最强模型需要高成本，路由策略应基于预算和任务上下文自适应选择。
3. **不同路由器行为差异显著**：模型选择分布（图D.1）显示有些路由器倾向集中于少数模型，有些覆盖面广。头对头对比（图D.2）表明优势并非全局一致，存在专业化。
4. **ORBIT 和 RouteJudge 形成互补**：ORBIT 提供可控可复现的离线开发，RouteJudge 提供真实用户偏好反馈，两者结合构成开放评估生态系统。

## 7. 优点

- **创新评估视角**：将评估从“模型级响应质量”转向“路由级决策质量”，直接反映用户偏好。
- **全面记录元数据**：每条记录包含查询、路由决策、模型响应、偏好、成本、延迟、任务标签等，支持多维度分析（偏好感知、成本感知、任务条件）。
- **匿名比对与路由器归因**：避免身份偏差，非参与路由器标记为∅，使参与率和胜率结合分析。
- **开放可扩展**：通过 ORBIT 标准化接口，研究者可轻松添加新路由方法、新基准、新编码器；通过 PR 提交后经历史回放再上线，保障质量。
- **模块化设计**：ORBIT 中编码器、路由器、数据集可独立更换，支持复现和消融。
- **提供公开平台和代码**：routejudge.cn 和 GitHub 仓库（ORBIT），促进社区共建。

## 8. 不足与局限

- **实验规模有限**：
    - 离线仅用一个基准（RouterEval），未在多基准（如 MixInstruct、MMR-Bench）上进行系统比较，泛化性存疑。
    - 在线投票数仅109份，统计效力不足；且用户群体可能偏斜（如只来自特定渠道），偏好代表性有限。
- **缺乏消融实验**：未展示对编码器、训练数据分割、预算网格等的敏感性分析，难以判断方法对超参数的稳健性。
- **未提供训练算力与时间**：影响方法实用性的评估缺失，尤其是路由方法可能本身有训练开销。
- **模型池静态**：当前仅17个候选模型，且多为闭源API。真实场景模型池可能更大、更动态，实验外推性受限。
- **历史回放阶段的潜在偏差**：新提交路由器通过历史记录评估，但历史记录本身受当时路由策略选择影响，可能造成偏差（例如某些模型被选少导致缺乏对比对）。
- **延迟影响未在偏好中体现**：虽然记录了延迟，但用户判断时成本/延迟信息被隐藏，因此偏好未直接反映延迟敏感度（仅通过预算间接约束）。
- **多模态支持初步**：平台支持多模态输入，但目前线上分析未见多模态用例。
