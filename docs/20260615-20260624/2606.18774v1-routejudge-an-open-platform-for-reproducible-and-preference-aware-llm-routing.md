## 摘要
我们介绍了RouteJudge，这是一个用于LLM路由系统的在线成对偏好评估框架，并提供了一个公共平台，网址为https://routejudge.cn。与模型级别的响应评估不同，RouteJudge专注于路由器级别的决策质量。对于每个用户查询，多个路由策略在相同的模型池和预算约束下独立推荐候选模型。然后通过匿名成对比较将所选模型的响应呈现给用户，并将由此产生的用户偏好归因于所比较响应背后的路由策略。每条评估记录存储查询、路由决策、模型响应、偏好标签、成本、延迟和任务元数据，从而实现对LLM路由器的偏好感知、成本感知和任务条件分析。为了支持RouteJudge中路由方法的持续扩展，我们进一步发布了ORBIT（最优路由和预算推理工具箱），这是一个模块化且可扩展的工具箱，标准化了LLM路由的端到端工作流程。ORBIT为基准加载、查询表示、路由器实现、预算感知评估和方法比较提供了统一接口，使研究人员能够在一致的协议下开发和评估路由算法。它也是RouteJudge的提交和集成层：研究人员可以在ORBIT中实现路由方法，在现有路由基准上验证它们，并提交兼容的路由器进行在线偏好评估。ORBIT的代码可在https://github.com/AIGNLAI/LAMDA-ORBIT获取。

## Abstract
We present RouteJudge, an online pairwise preference evaluation framework for LLM routing systems, with a public platform available at https://routejudge.cn. Different from model-level response evaluation, RouteJudge focuses on router-level decision quality. For each user query, multiple routing strategies independently recommend candidate models under the same model pool and budget constraints. The selected model responses are then presented to users through anonymous pairwise comparisons, and the resulting user preferences are attributed back to the routing strategies behind the compared responses. Each evaluation record stores the query, routing decisions, model responses, preference labels, cost, latency, and task metadata, enabling preference-aware, cost-aware, and task-conditioned analysis of LLM routers. To support the continuous expansion of routing methods in RouteJudge, we further release ORBIT (Optimal Routing and Budgeted Inference Toolbox), a modular and extensible toolbox that standardizes the end-to-end workflow of LLM routing. ORBIT provides unified interfaces for benchmark loading, query representation, router implementation, budget-aware evaluation, and method comparison, allowing researchers to develop and evaluate routing algorithms under consistent protocols. It also serves as the submission and integration layer for RouteJudge: researchers can implement routing methods within ORBIT, validate them on existing routing benchmarks, and submit compatible routers for online preference-based evaluation. The code of ORBIT is available at https://github.com/AIGNLAI/LAMDA-ORBIT.

## 1. 核心问题与整体含义（研究动机和背景）
- **研究动机**：现有LLM路由评估方法大多基于离线基准（静态基准、黄金答案、自动评分），将路由质量简化为固定指标（如准确率），忽略了实际用户交互中存在的多元化偏好（如不同用户对简洁性、详细程度、风格、成本敏感度的差异）。这种“固定目标评估”无法反映真实部署场景下的路由决策效果。
- **整体含义**：论文提出应转向“路由器级决策质量”评估，即路由选择是否导致用户实际更偏好的响应，而非仅仅匹配离线最优模型。为此，构建了一个结合在线偏好评估平台（RouteJudge）和标准化开发工具箱（ORBIT）的开放生态系统，推动LLM路由评估向可复现、偏好感知、成本感知、任务感知方向发展。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：通过在线匿名成对比较，将用户对模型响应的偏好信号归因于背后的路由策略，直接评估路由决策（而非模型响应本身）的质量。
- **关键技术细节**：
  - **评估记录**：每条记录包含用户查询、历史上下文、多模态输入、预算、预算可行模型集、所有路由器的决策、用户比较的模型对、偏好标签（A胜/B胜/平局/两者皆差）、成本、延迟、任务标签等。
  - **评估流程**：用户提交查询与预算 → 路由器从预算可行模型集中独立推荐模型 → 平台选择得票最高的两个模型进行匿名成对比较 → 用户给出偏好标签 → 结果揭示并归因给路由器（若路由器选择的模型进入比较则得分；否则记为未参与）。
  - **ORBIT工具箱**：标准化开发流程，提供统一的数据加载、查询嵌入、路由实现、训练、验证、提交接口。支持基于规则、检索、分类、回归、图、奖励模型等多种路由方法。
  - **提交与集成**：研究人员在ORBIT中实现路由器，提交后先进行历史回放评估（在已有用户偏好记录上模拟），再进入在线评估，确保可复现性和自洽性。

## 6. 主要结论与发现
- **离线与在线评估互补**：离线基准上的路由性能并不能完全转化为在线用户偏好，一些离线强路由在在线评估中胜率较低（如HybridLLM在线胜率仅18.18%），而简单的非参数或矩阵分解方法（如RouteLLM-MF）反而表现更好。
- **成本-偏好非线性**：模型成本与用户偏好并非正相关，低成本模型也能获得较高胜率，说明路由应依据预算和任务自适应选择，而非一味追求高价模型。
- **路由器参与率重要**：仅靠胜率不能全面评价路由策略，参与率（即路由器选择的模型进入比较的频率）也影响评估可靠性。高胜率低参与率可能意味着评价覆盖不足。
- **开放平台可复现**：ORBIT标准化接口使不同路由方法能在一致条件下开发和比较，消除预处理、编码器等不一致因素，提升研究可复现性。

## 7. 优点
- **创新评估范式**：首创将路由决策质量直接与用户偏好挂钩，而非依赖固定基准，更贴近实际部署场景。
- **开源生态完整**：提供在线平台（RouteJudge）和离线工具箱（ORBIT），涵盖全流程（开发、离线评估、提交、在线评估），降低门槛，促进社区贡献。
- **多维度分析**：支持成本感知、任务条件分析、参与率、头部对决等，超越单一排名，提供更丰富的诊断信息。
- **匿名双盲设计**：消除用户判断时的偏见（如模型品牌、成本提示），提高偏好信号的可靠性。
- **可扩展性**：ORBIT设计模块化，易于添加新基准、新编码器、新路由算法，并自动兼容在线平台，鼓励持续迭代。

## 8. 不足与局限
- **在线数据量有限**：目前仅109个用户投票，统计显著性不足，结论可能受样本偏差影响（如用户群体、任务分布未知）。平台需要更多时间和用户参与才能得出可靠排名。
- **离线基准覆盖不足**：仅展示了RouterEval上的离线实验结果，未在更多基准（如MixInstruct、RouterBench、MMR-Bench）上报告类似曲线，基准泛化性未验证。
- **消融实验缺失**：对路由方法各组件的贡献（如编码器类型、训练数据量、特征）没有系统消融，难以分离各因素影响。
- **未讨论安全与偏见**：偏好评估完全基于用户主观判断，未考虑毒性、有害内容、公平性等维度，可能导致路由偏向迎合用户潜在偏见。
- **算力信息缺失**：未提供训练/推理的硬件和计算成本，使得方法的实际部署开销难以评估。
- **任务标签粗粒度**：当前仅分 Coding/Math/Translation等几大类，缺乏细粒度任务难度、领域等划分，可能掩盖路由在不同子场景下的表现差异。
- **多模态支持初步**：虽然框架支持多模态输入，但当前在线平台和初步实验均以文本为主，多模态路由评估未做展示。
