## 摘要
在稀疏混合专家语言模型中，相同的token id是否意味着相同的路由器状态和产生它的相同专家？在重复锚点处固定发出的token id，我们发现并非如此：产生它的专家仍然分离任务上下文、轨迹历史和推理努力模式。这种残差结构支持测试时控制：在边界锚点（最终响应转换）和分隔锚点（打开答案，例如\boxed{}或代码围栏）附近，路由邻域已经与仅标记读取出的最终答案盆地对齐，并且在答案打开时读取路由时最强。我们将其操作化为RAD（路由一致解码），一种无需答案字符串的多轮次选择器：它定位一个固定锚点，通过其锚点窗口的MoE路由状态表示每个轮次，并返回最密集的加权Jaccard K近邻路由盆地中心，无需解析、规范化、执行或对答案字符串进行投票。在10个稀疏MoE配置（gpt-oss, Qwen3-MoE）和涵盖数学、GPQA和代码的6个数据集上，RAD与多数投票相当，其中字符串投票是适定的，具有小的正配对差异（RAD 73.9 / RAD+DC 74.2 vs. 多数投票 73.6）。与多数投票一样，RAD不是验证器：密集的“错误”盆地仍然可能获胜。其价值在于接口：相同的选择器在代码上直接给出pass@1，其中精确字符串投票是不适定的，并且相同的路由密度原则，重新锚定到代理边界，在SWE-bench Verified上随机改进最佳16补丁选择，其中补丁没有要投票的答案字符串。

## Abstract
In sparse Mixture-of-Experts language models, does the same token id imply the same router state and the same experts producing it? Holding the emitted token id fixed at repeated anchors, we find it does not: the experts that produce it still separate task context, trajectory history, and reasoning-effort mode. This residual structure supports test-time control: near \emph{boundary} anchors (the final-response transition) and \emph{delimiter} anchors (which open the answer, e.g.\ \texttt{\textbackslash boxed\{} or code fences), routing neighborhoods already align with final-answer basins at a marker-only readout and strongest when the routing is read at the answer opening. We operationalize this as \textbf{RAD} (Routing Agreement Decoding), an answer-string-free multi-rollout selector: it locates a fixed anchor, represents each rollout by its anchor-window MoE routing states, and returns the densest Weighted-Jaccard $K$-NN route-basin center, without parsing, normalizing, executing, or voting over answer strings. Across 10 sparse-MoE configurations (gpt-oss, Qwen3-MoE) and 6 datasets spanning math, GPQA, and code, RAD is on par with Majority where string voting is well-posed, with small positive paired deltas (RAD $73.9$ / RAD+DC $74.2$ vs.\ Majority $73.6$). Like majority voting, RAD is not a verifier: a dense \emph{wrong} basin can still win. Its value is the interface: the same selector gives direct pass@1 on code, where exact-string voting is ill-defined, and the same routing-density principle, re-anchored to the agentic boundary, improves best-of-16 patch selection on SWE-bench Verified over random, where patches have no answer string to vote on.

### 1. 核心问题与整体含义（研究动机与背景）
论文研究稀疏混合专家（MoE）语言模型中一个根本问题：**相同的 token id 是否一定对应相同的内部路由状态和生成该 token 的专家**？通过控制锚点 token 的 id 固定，作者发现**并非如此**：产生同一 token 的专家仍会依据任务上下文、轨迹历史和推理努力程度而分离。这种残差结构表明，MoE 路由状态比 token 身份承载更多信息。基于此发现，论文提出 **RAD（Routing Agreement Decoding，路由一致性解码）**，一种完全**无需解析、归一化或投票最终答案字符串**的多 rollout 选择器。该方法通过利用路由几何信息来聚合 rollouts 之间的一致性，在多数投票适用的场景下与之持平，而在代码生成、agentic 补丁选择等答案字符串不易获取或不适定的场景下提供了直接有效的选择机制。

### 2. 方法论
- **核心思想**：利用锚点（anchor）对齐 rollouts，提取每个 rollout 在锚点窗口中 MoE 路由的稀疏向量（各层 top-k 专家及其权重），通过加权 Jaccard 相似度衡量 rollout 间的路由一致性，并选择具有最高 K 近邻密度的 rollout 作为代表，而不读取任何最终答案字符串。
- **关键技术细节**：
  - **锚点类型**：轨迹锚点（如 "So"）、边界锚点（如最终响应转换标记 <|channel|>final<|message|>）、分隔锚点（如 \boxed{} 或代码围栏）。默认使用分隔锚点。
  - **路由表示**：对每个 rollout，取锚点之后 **W=16** 个 token 的 MoE 路由向量（各层 top-k 专家 id 和 softmax 权重）的平均，得到向量 \( z_i \)。
  - **相似度计算**：加权 Jaccard (Weighted Jaccard)：
    \[
    \mathrm{WJ}(u,v) = \frac{\sum_e \min(u_e, v_e)}{\sum_e \max(u_e, v_e)}.
    \]
  - **选择策略**：计算每个 rollout 与其它 rollout 的 WJ 相似度矩阵 \( S \)，对每个 rollout 取前 \( k_{\mathrm{rad}}=10 \) (默认) 个最近邻，汇总其相似度得到密度 \( q_i \)，选择 \( i^* = \arg\max q_i \)。
  - **完全无答案字符串**：选择过程中不解析、不比较任何最终答案内容；仅在选择后读出答案用于评估。
  - **融合版本 RAD+DC**：利用 DeepConf 的置信度分数（标量），先保留置信度最高的 50% rollout，再应用 RAD 密度选择。

### 6. 主要结论与发现
1. **相同 token id 不一定对应相同路由状态**：即使发射 token id 固定，产生该 token 的专家仍能区分任务、问题和推理努力程度。
2. **路由邻域与答案盆地对齐最强于分隔锚点窗口**：通过在分隔锚点（如 \boxed{} 后 16 token）的读取窗口，KNN 纯度最高，且与选择器准确率呈正相关。
3. **RAD 是软共识而非验证器**：它选择最密集的路由盆地，而非判断正误；当多数 rollout 错误时，RAD 也会选择错误。
4. **与文本多数投票性能相当，且更具扩展性**：在数学+GPQA 上与 Majority 持平（RAD 73.9 vs Majority 73.6）；在代码生成和 agentic 补丁选择这些字符串投票不适定场景，RAD 仍能工作并显著提升。
5. **信号冗余且鲁棒**：核函数二值/加权 Jaccard 等价，仅需浅层路由即可，无需全网络深度。

### 7. 优点
- **完全无需答案字符串**：不依赖解析、归一化、执行或投票最终答案，使其适用于代码、对话等需灵活输出的场景。
- **统一的接口**：同一原则可应用于数学、科学、代码生成和 agentic 任务，无需任务特定修改。
- **理论奠基扎实**：通过分析路由的残差结构和邻域纯度，为方法提供清晰的认知机理。
- **实践效率高**：路由捕获开销仅约 7%，选择阶段在 CPU 上快速完成。
- **跨架构可迁移**：在三种不同 MoE 架构（gpt-oss、Qwen3、Qwen3-Next）上均有效。

### 8. 不足与局限
- **不是验证器**：RAD 只能反映共识，而非正确性；错误共识下会选择错误结果，与多数投票同进退。
- **依赖 MoE 路由访问**：不适用于密集模型或闭源 API，需白盒路由信息。
- **锚点覆盖不完全**：部分 rollouts 缺少所需锚点（约 7% 问题舍弃），可能产生有偏子集。
- **对 GPQA 提升微弱**：相比数学和代码，RAD 在科学问答数据集上的优势较小甚至为负（在 GPQA 上 RAD 准确率略低于 Majority）。
- **统计显著性有限**：与 Majority 的配对差距较小且不显著（p≈0.52），主要优势在于扩展性而非绝对准确度提升。
- **agentic 提升仍有限**：SWE-bench 上的 end-to-end 提升约 2-3pp，仍远低于 oracle 上限（约 75% vs 95%+），且仅基于单次 rollout 池。
