## 摘要
诸如autoresearch之类的长期运行编码智能体能够持续发现开放式问题的优化方案。然而，它们往往收敛到单一的高级方法，然后在进行低级编辑时遗漏了其他更优的方法。我们假设有两种工具层面的设计选择导致了这一行为：在单个长期运行的智能体中累积上下文，以及仅暴露单个程序状态进行编辑。我们引入了SwarmResearch，一种编排器-子智能体工具，其中Shepherd Agent利用全局上下文来引导一群Search Agents，每个Search Agent在其各自的git分支中利用局部上下文进行操作。在开放式优化任务中，SwarmResearch在13/15个任务上发现了比最先进的LLM引导进化和多智能体技术更好或可比的解决方案，这是由更高级别的探索驱动的。与串行和并行智能体的固定规模相比，SwarmResearch的编排器引导的扩展通过在不同搜索深度自适应并行性发现了性能更好的解决方案。

## Abstract
Long-running coding agents such as autoresearch can persistently discover optimizations for open-ended problems. However, they tend to converge onto a single high-level approach, then proceed with low-level edits while missing other superior approaches to the problem. We hypothesize two harness-level design choices contribute to this behavior: accumulating context in a single long-running agent and only exposing a single program state to edit. We introduce SwarmResearch, an orchestrator-subagent harness in which a Shepherd Agent uses global context to steer a population of Search Agents, each operating with local context in their respective git branch. On open-ended optimization tasks, SwarmResearch discovers better or comparable solutions to state-of-the-art LLM-guided evolution and multi-agent techniques on 13/15 tasks, driven by higher-level exploration. Compared with fixed scaling of serial and parallel agents, SwarmResearch's orchestrator-guided scaling discovers better-performing solutions by adapting parallelism at different search depths.

## 1. 论文的核心问题与整体含义（研究动机和背景）
- **问题**：长期运行的编码智能体（如 autosearch）尽管能持续优化开放式问题，但往往迅速收敛到单一高级方法，随后只做大量低级编辑（微调参数、局部调整），而错失其他更优的高级方法。
- **假设的瓶颈**：作者认为两个工具层面的设计缺陷导致了这种行为：
  - **上下文积累**：智能体在单一对话历史中不断累积关于当前方法的语境，使得转向完全不同的新方法变得困难。
  - **单程序状态**：所有改进都提交到同一可编辑程序，回归被回退，这种设计本质上是贪心局部搜索，难以并行保留多个竞争方向。
- **意义**：需要新的编排机制来促进多样化的高级探索，从而在更广泛的解空间中寻找更优的方案。

## 2. 论文提出的方法论：核心思想与关键技术细节
### 核心思想
- 采用 **编排器-子智能体（orchestrator-subagent）** 架构。
- **Shepherd Agent**（编排器）利用全局上下文战略性地引导一群 **Search Agents**（子智能体），每个搜索智能体在各自的 git 分支上运行，拥有局部上下文，避免全局信息收敛。

### 关键技术细节
1. **上下文管理**：
   - **局部上下文**（Search Agent）：仅包含其工作目录内容（对于 Explorer）或继承父智能体的对话历史（对于 Optimizer）。
   - **全局上下文**（Shepherd Agent）：所有 Search Agent 的简短摘要（方法、得分），用于战略分配资源。
   - *Explorer*：初始上下文完全干净，适合尝试全新方法；*Optimizer*：继承父对话历史，便于在已有方向上做序列精化。

2. **Git 分支管理**：
   - 每个 Search Agent 被分配一个新 git 分支和工作目录。
   - 分支可从 **main**（基线）开始探索全新方法，或从已完成智能体的提交开始延续某个方向。
   - 通过分支的扇出（从同一父分支起步多个并行智能体）和串行链（在一条直系上反复优化），Shepherd 可以灵活表达多种搜索模式。

3. **Shepherd Agent 的三个引导机制**：
   - **父选择**：决定新智能体从哪个解决方案（分支）出发，控制探索邻域。
   - **搜索智能体类型**：选择 Explorer（新鲜上下文）或 Optimizer（继承上下文）。
   - **提示**（prompts）：向子智能体提供最小但有价值的补充信息（如避免重复的思路、当前最强方案的优劣势），但严格禁止指定具体方法。
   - 编排器默认行为：先批量启动 4-8 个 Explorer，之后针对最有前景的方案启动 Optimizer，再逐步集中资源。

4. **迭代流程**：
   - Shepherd 循环遍历预算，每次选择一个父分支、类型和可选提示 → 以 CLI 命令生成新 Search Agent → 子智能体自行完成“思考-实现-评估-提交”循环 → 返回摘要 → Shepherd 更新全局知识并继续下一轮。

5. **与常规方法的区别**：
   - 传统进化算法（如 EvoX）采用固定启发式选择父代，且 LLM 作为变异算子受限于流水线。
   - 多智能体系统（如 CORAL）所有智能体共享记忆，容易因看到更强方案而集体放弃差异化方向。
   - SwarmResearch 通过编排器和分支隔离避免了这两类收敛问题。

## 3. 实验设计：数据集/场景、benchmark、对比方法
### Benchmark
15 个开放优化任务，分为三个领域：
- **数学优化**（5 个任务）：Circle Packing, Signal Processing, Erdos Min Overlap, MMD-14-3, 3rd-Autocorrelation（来自 prior work）。
- **系统优化**（5 个任务）：EPLB, LLM-SQL, Txn Scheduling, Cloudcast, PRISM（来自 ADRS 基准，基于真实系统研究）。
- **启发式算法**（5 个任务）：Territory (AHC008), Halloween Candy (AHC015), Graphorean (AHC016), Balancing by Balance (AHC025), Stack of Boxes (AHC026)（来自 ALE-Bench-Lite，源自 AtCoder 启发式竞赛）。
- 所有任务包含一个最小可行基线解。

### 对比方法
- **EvoX**（Liu et al., 2026）：最先进的 LLM 引导进化方法，可演化搜索策略本身。
- **CORAL**（Qu et al., 2026）：最先进的多智能体系统，使用共享文件系统记忆，4 个并发智能体。
- **固定并行/串行扩展**（fixed scaling）：使用 Pi 工具，测试 (n, k) 配置（n 并行×k 串行迭代），共享 git 历史，无编排器。
- **具体实现**：SwarmResearch 和 CORAL 使用 Claude Code（Opus 4.6/4.8），EvoX 使用 Opus 4.6。所有方法无互联网访问。

### 额外案例研究
- **近似投机解码（Approximate Speculative Decoding）**：在 30 个推理密集型任务（LiveCodeBench v6, AIME 2026, GPQA Diamond, HLE-MCQ）上对比 SwarmResearch vs 原始 autosearch vs CORAL，测量 token 吞吐量（平均 tok/s）和准确率。

## 4. 资源与算力

## 6. 论文的主要结论与发现
1. **性能超越**：SwarmResearch 在 **13/15** 任务上超过或持平 EvoX 和 CORAL（严格超过 CORAL 在 10 个任务，持平 2 个）。在数学优化任务上差距较小，在系统优化和启发式任务上优势更明显。
2. **更高级别探索**：平均代码变更规模比 CORAL 大 **3.2 倍**，比 EvoX 大 **1.7 倍**，说明 SwarmResearch 的实验更倾向于测试全新的高级方法，而非局部微调。
3. **自适应扩展优于固定扩展**：编排器引导的扩展（动态调整并行度与深度）在 **4/5** 任务上超过了最优固定配置（fixed scaling），且不需要手动选择超参数。最优固定配置通常是“宽而浅”（高并行、少串行）而不是“窄而深”。
4. **投机解码案例**：SwarmResearch 达到平均 **4.58 倍** 加速（vs 朴素解码），优于 autosearch 的 1.80 倍和 CORAL 的 2.26 倍。最佳优化包括批量推理、持久线程池、自适应 MoE 专家数等。
5. **编排器行为**：默认模式接近于贪心：先并行探索，然后集中到 top-2 方案，最终聚焦到最好方案。有时会陷入特定思路，通过对提示的约束可改善。

## 7. 优点：方法或实验设计上的亮点
- **轻量且兼容**：SwarmResearch 实现为三个技能（shepherd, explorer, optimizer），可即插即用到流行编码智能体工具（如 Claude Code），易于使用和推广。
- **无手动超参数**：编排器自适应调整并行/串行规模，无需用户指定并行度或迭代次数。
- **解释性强**：git 分支树可视化清晰展示了搜索轨迹（图 1），便于理解不同路径的探索和合并决策。
- **分离关注点**：全局 vs 局部上下文的设计巧妙解决了多智能体系统容易“自我强化”导致的多样性丧失问题。
- **高吞吐实验**：支持同时运行多个 Search Agent，有效利用预算，并通过 Shepherd 控制避免重复。
- **案例详尽**：投机解码案例不仅展示了性能，还讨论了风险（如产生说服力强但无效的算法），增加了可信度。

## 8. 不足与局限
- **单次运行局限**：所有主实验仅一次运行，受 LLM 采样随机性影响，统计稳定性不足。作者虽在附录说明，但未执行多轮复现以量化方差。
- **模型依赖**：Shepherd Agent 的引导能力高度依赖 LLM 的推理和规划能力，当前模型（Opus 4.6/4.8）并未展现真正的战略规划（如精准识别瓶颈、设计差异化实验），执行上仍偏向贪心。
- **低效探索风险**：高级方法实验有时没有成效，浪费预算，尤其是当探索过早集中于非生产性方向时。
- **误导性结果**：在投机解码案例中，SwarmResearch 产生了“单调采样”等看似合理但实际无效的方案，需人工仔细审核。高吞吐量的低质量解决方案可能误导用户。
- **启发式任务仍落后人类**：在所有启发式竞赛任务上，AI 系统的得分仍远低于人类专家设计的 SOTA 方案（差距约 1000 分以上），说明开放域创造力仍是瓶颈。
- **缺少消融实验**：没有单独消融每个组件（如去掉分支管理、去掉 Explorer 类型等）来验证各自贡献。
- **计算成本**：一次完整 benchmark 需 $1700，对于一般研究者负担较重，可复现性受限。
- **领域覆盖有限**：仅数学、系统、启发式三类，未包含科学发现、算法设计等更多开放式问题。
