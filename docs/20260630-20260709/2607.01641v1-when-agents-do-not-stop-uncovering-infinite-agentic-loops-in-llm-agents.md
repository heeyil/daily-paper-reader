## 摘要
LLM代理越来越依赖迭代执行，通过规划、工具使用、状态更新和代理协作来解决问题。虽然这种设计实现了灵活的自动化，但也创造了一类新的故障：当反馈路径未得到有效约束时，代理可能会重复执行模型调用、工具、工作流转换或代理交接。我们将此问题称为无限代理循环（Infinite Agentic Loops, IALs）。IALs并非普通的编程循环；它们源于代理逻辑、框架语义、运行时观察和终止机制之间的交互。此类故障可将单个请求放大为长期运行的模型和工具执行，导致成本耗尽、模型拒绝服务、上下文增长以及重复的外部副作用。我们提出IAL-Scan，一种用于检测真实LLM代理项目中IAL故障的静态分析工具。IAL-Scan将异构代理代码抽象为框架无关的代理IR，构建代理循环依赖图（Agentic Loop Dependence Graph, ALDG）以恢复显式及框架引入的反馈路径，并检查这些路径能否在无有效边界的情况下重复到达高成本或状态增长的操作。我们在6,549个LLM代理仓库上评估IAL-Scan。它报告了74个潜在发现，其中人工审查确认了47个项目中的68个IAL故障，实现了91.9%的精确率。

## Abstract
LLM agents increasingly rely on iterative execution to solve tasks through planning, tool use, state updates, and agent collaboration. While this design enables flexible automation, it also creates a new class of failures: an agent may repeatedly execute model calls, tools, workflow transitions, or agent handoffs when the feedback path is not effectively bounded. We call this problem Infinite Agentic Loops (IALs). IALs are not ordinary programming loops; they arise from the interaction between agent logic, framework semantics, runtime observations, and termination mechanisms. Such failures can amplify a single request into long running model and tool execution, causing cost exhaustion, model denial of service, context growth, and repeated external side effects. We propose IAL-Scan, a static analysis tool for detecting IAL failures in real-world LLM agent projects. IAL-Scan abstracts heterogeneous agent code into a framework independent Agent IR, builds an Agentic Loop Dependence Graph (ALDG) to recover explicit and framework induced feedback paths, and checks whether these paths can repeatedly reach costly or state growing operations without an effective bound. We evaluate IAL-Scan on 6,549 LLM agent repositories. It reports 74 potential findings, among which manual review confirms 68 IAL failures across 47 projects, achieving 91.9% precision.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：大语言模型（LLM）代理在迭代执行任务（规划、调用工具、状态更新、多代理协作）时，由于反馈路径未被有效约束，可能陷入**无限代理循环（Infinite Agentic Loops, IALs）**——即反复触发模型调用、工具执行、工作流转换或代理交接，且没有有效的终止条件。
- **危害**：单个请求被放大为长时间运行，导致API成本耗尽、模型服务拒绝、上下文窗口膨胀、重复外部副作用等严重后果。
- **现有不足**：尽管主流框架提供了循环控制机制（如`max_iterations`、`recursion_limit`），但开发者可能遗漏、误配或将其置于反馈路径之外；通用静态分析工具（如CodeQL）无法建模代理执行语义；已有代理安全分析研究未聚焦于IAL检测。

## 2. 方法论：核心思想、关键技术细节、算法流程
- **核心思想**：通过静态分析，将异构的代理代码（不同框架、自定义逻辑）统一抽象为框架无关的中间表示（Agent IR），构建**代理循环依赖图（Agentic Loop Dependence Graph, ALDG）**，以此恢复显式和框架隐含的反馈路径，并检查每条路径是否能在无有效边界条件下反复到达高成本或状态增长的操作。
- **关键技术细节**：
  - **Agent IR 构建**：定义事实（ExecutionUnit, Controller, Invocation, StateUpdate, Bound, ExitRecord）和关系（calls, updates, guards, constrains, transitions等），从源代码和框架行为中提取。支持8个主流框架：LangChain, LangGraph, CrewAI, AutoGen, LlamaIndex, OpenAI Agents SDK, Google ADK, Semantic Kernel。
  - **ALDG 构建**：将Agent IR中的控制器、高成本调用、状态增长更新、执行作用域映射为12类节点；根据10类边（控制流、调用、反馈、退出等）连接；每个控制器节点标注循环属性（如body_cost_kinds, state_growth, guard_source等）。
  - **IAL 检测算法**（算法1）：
    1. 从ALDG中提取含有循环边的子图，计算强连通分量（SCC），得到候选反馈区域。
    2. 对每个候选，计算属性集Π和拓扑集Θ，过滤非目标循环（如流消费、分页、生命周期循环等）。
    3. 选择控制控制器cL，分类为确定性/模型控制/工具控制/外部状态控制等。
    4. 检查边界覆盖：分析是否存在有效边界（如`max_turns`, `max_iterations`, `recursion_limit`, `retry_limit`, `timeout`等）约束控制器的反馈路径。边界状态分为`Covered`（验证/框架默认/配置依赖）和`UncoveredOrWeak`（缺失/弱/禁用/无效/绕过）。
    5. 仅当反馈路径无有效边界覆盖且可达高成本/状态增长操作时，报告IAL故障。可选的LLM辅助剪枝用于降低误报。

## 6. 主要结论与发现
- **性能**：IAL-Scan在6,549个LLM代理仓库中，端到端精确率91.9%（68个真实故障/74个潜在发现），覆盖全部8个框架。
- **故障模式**：最常见的模式：重试反馈无边界（25%）、工具调用迭代无边界（23.5%）、多代理对话无轮次边界（20.6%）、工作流循环无有效边界（13.2%）、消息重入无边界（10.3%）、委托/评估员反馈（7.4%）。
- **影响**：95.6%故障导致API成本耗尽和模型DoS，27.9%导致上下文窗口耗尽，7.4%导致外部工具速率限制耗尽。
- **根因**：全部68个故障的根因均为缺少强边界。工具控制重试（41.2%）、模型控制终止（38.2%）、缺失退出（33.8%）、工作流循环无验证边界（30.9%）、状态增长放大（27.9%）、代理工具重入（25.0%）。
- **稳定性**：静态阶段完全可重复；LLM剪枝阶段略有波动（三遍重复：74/73/70个警报，覆盖68/64/60个真实故障），但总体可接受。不同LLM模型剪枝效果差异大，默认gpt-5.5最佳。

## 7. 优点
- **框架无关性**：通过Agent IR统一表示，支持8个主流框架，可扩展。
- **针对IAL的专用分析**：首次系统定义IAL故障，并设计静态分析检测，填补现有工具空白。
- **边界覆盖推理**：不仅检查是否存在循环，还验证边界是否**有效覆盖**反馈路径（考虑控制器类型、模型/工具依赖性）。
- **高效的实用性**：在真实大规模仓库上达到91.9%精确率，平均每项目仅31.2秒、4.2K tokens，远优于LLM基线。
- **消融和稳定性实验全面**：验证了各组件贡献，并量化了LLM剪枝的变异风险。

## 8. 不足与局限
- **静态分析过度近似**：可能报告假阳性（实验中6个FP），因为静态分析无法精确预测运行时行为（如间接边界配置、外部状态）。
- **语言和框架范围有限**：当前仅支持Python和8个框架，对其他语言或不支持的框架可能遗漏（FN中7个，主要源于LangGraph工具循环或自定义框架）。
- **自定义语义支持不足**：对于高度定制化的调度逻辑、基于自然语言输出的语义检查、外部状态驱动的终止条件，静态分析精度有限。
- **LLM辅助剪枝不稳定**：作为可选负过滤，不同LLM模型表现差异大（覆盖从60到68个TP），对用户选择模型敏感；LLM可能误判导致漏报或误报增加。
- **未考虑多实例/分布式场景**：分析局限于单仓库代码，未探讨跨代理部署、微服务架构中的IAL扩展。
